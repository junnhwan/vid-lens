# VidLens 分片上传与断点续传：Redis 状态、MinIO 合并及面试防守

> 对应简历：**采用分片上传与断点续传机制，通过 Redis 记录分片状态并使用 MinIO 服务端合并，避免上传中断后整文件重传。**
> 核心原则：Redis 保存的是带 TTL 的临时上传规格、状态和片号，不保存视频字节；断点续传是应用层跳过已完成分片，不是 TCP 连接从断点继续。

---

## 1. 先记住最重要的结论

### 30 秒回答

> 长视频如果用一个请求上传完整文件，网络在最后阶段中断也要从零重传。VidLens 在客户端计算文件 MD5，并按固定大小切片。后端先校验 file size、chunk size 和 total chunks，每片写入 MinIO 临时对象后，再把完成片号写到 Redis。重新选择同一文件时，客户端先查询 Redis 中的进度并跳过已完成片段。所有片齐全后，后端用分布式锁避免同一文件并发合并，优先使用 MinIO ComposeObject 在对象存储端合并，再校验最终文件大小并创建资产。

### 最重要的区分

```text
MinIO：保存分片字节和最终视频对象
Redis：保存上传规格、会话状态和已完成片号
PostgreSQL：保存合并完成后的 VideoAsset 和业务任务
```

不要说“视频分片存在 Redis 里”。

---

## 2. 为什么普通整文件上传不够

VidLens 配置允许较大的视频文件。整文件上传的问题不只是“速度慢”，而是失败粒度太大：

```text
上传 95% 时网络断开
→ 单请求失败
→ 服务端无法确认一个完整可用文件
→ 用户只能重新传全部内容
```

分片后：

```text
上传 95% 时网络断开
→ 已成功分片仍在 MinIO
→ Redis 仍记录已完成片号
→ 恢复时只补剩余分片
```

这项方案主要减少的是**中断后的重复上传量**，不是声称网络传输本身变快。没有压测时，不要说吞吐提升多少倍。

---

## 3. 1 分钟回答

> 分片上传的目标是把大文件上传的失败粒度从“整个文件”缩小到“一个分片”。客户端先计算 MD5，再带着文件大小、分片大小和总片数查询服务端。后端使用 `upload:chunks:<md5>` 这一组 Redis key 保存上传规格、状态和片号集合，TTL 当前是 24 小时。每个分片先写到 MinIO 的 `chunks/<md5>/<index>`，成功后才把 index 加入 Redis，这样 Redis 中的完成状态不会先于对象落盘。
>
> 恢复时客户端拿到已上传 index，只补缺失片。合并前服务端重新校验 MD5 格式、总大小、分片大小、总片数以及会话规格，并逐片确认 Redis 记录完整。为了避免相同文件被并发合并，还按 MD5 加了 Redis 锁。正常 5MiB 分片走 MinIO 服务端 Compose；如果非末尾分片小于 Compose 的 5MiB 限制，则改为按顺序流式读取分片并 PutObject，不会把完整文件一次载入内存。最后校验合并对象大小、落库资产，再 best-effort 清理临时分片。

---

## 4. 端到端流程

```text
客户端选择文件
  → 计算文件 MD5
  → 根据 chunk_size 计算 total_chunks
  → GET/POST check-upload(md5, file_size, chunk_size, total_chunks)
      → 校验上传规格
      → 查询 Redis 已完成片号
      → 若旧会话规格不兼容，清理旧片和状态
      → 返回 uploaded indexes 或 completed
  → 客户端跳过 uploaded indexes，只上传缺失片
      → 后端校验参数
      → PutObject 到 chunks/<md5>/<index>
      → 对象写成功后 SADD index
  → 客户端请求 merge
      → 检查是否已有相同 MD5 的 VideoAsset
      → 获取 vidlens:merge:<md5> 锁
      → 再次校验上传会话规格
      → 确认 0..total-1 每片都存在完成记录
      → ComposeObject 或流式 PutObject
      → 校验合并后 size
      → 创建/恢复 VideoAsset
      → 创建 VideoTask
      → 清理临时分片和进度集合
```

对应入口：

- handler：`internal/handler/media.go`
- service：`internal/service/media.go`
- storage：`internal/storage/minio.go`
- 前端续传：`web/src/chunkedUpload.js`

## 5. 为什么是“先落盘，后记账”

### 面试官问题

> 上传一个分片时，为什么先写 MinIO，再写 Redis？

### 直接口语回答

> Redis 的片号表示服务端已经有可用的分片对象。如果先把片号写进 Redis，再上传 MinIO，中间一旦 MinIO 写失败或进程退出，Redis 会声称该片完成，但实际对象不存在，恢复时客户端会错误跳过它。所以当前顺序是先把对象写入 MinIO，成功后再 `SADD` 片号。

### 仍然存在的窗口

如果 MinIO 写成功后、Redis `SADD` 前进程崩溃，会留下一个对象但 Redis 没记账。客户端会重新上传这一片，覆盖相同对象名，结果通常仍正确，但产生重复传输。

因此当前设计偏向：

```text
允许少量重复上传
不允许把不存在的片段误认为已完成
```

这不是数据库与对象存储之间的严格事务。

## 6. Redis 中具体保存了什么

基础 key 前缀：

```text
upload:chunks:<fileMD5>
```

当前相关数据包括：

- 集合本身：已完成的 `chunkNumber`；
- `:file_size`：本次上传声明的文件大小；
- `:chunk_size`：分片大小；
- `:total`：总片数；
- `:status`：如 `INIT`、`COMPLETED`；
- TTL：约 24 小时。

### 为什么不仅记录片号，还要记录上传规格

同一个 MD5 理论上应对应同一内容，但客户端可能更换分片大小，或者旧会话状态残留。如果只看到“0、1、2 已完成”，却不知道这些片是按 2MiB 还是 5MiB 切的，新会话会错误复用不兼容片段。

当前 `CheckUploadProgress` 会比较：

```text
file_size + chunk_size + total_chunks
```

如果规格不一致，会清理旧片段和旧 Redis 状态，重新开始会话。这一逻辑对应真实修复提交 `fix(upload): reject incompatible resume sessions`。

## 7. 为什么要有 TTL

### 直接口语回答

> 用户可能上传一部分后永远不再继续。如果状态没有过期机制，Redis 会一直积累废弃会话。当前上传状态设置 24 小时 TTL，在这段时间内允许恢复，超时后状态会过期。这个值也是产品折中，不是固定标准。

### 注意边界

Redis 状态过期不一定自动删除 MinIO 中的临时分片对象。当前成功合并后会 best-effort 清理，但对于永远没有合并的废弃上传，还需要对象生命周期策略或定时清理，不能声称临时对象绝不会残留。

---

## 8. 为什么合并前还要重新校验

客户端说“都传完了”不能作为服务端的唯一依据。`MergeChunks` 会检查：

1. MD5 格式是否合法；
2. `totalChunks`、`expectedFileSize`、`chunkSize` 是否为正；
3. 总片数是否等于按文件大小计算出的片数；
4. Redis 保存的会话规格是否完全一致；
5. 每个 0 到 `totalChunks-1` 的片号是否都存在；
6. 合并后的对象大小是否等于预期文件大小。

### 面试口语回答

> 断点续传会跨多个 HTTP 请求，客户端状态可能过期、被篡改或和旧会话冲突，所以 merge 不是简单执行拼接，而是最后一道完整性检查。如果缺少任何片，或者会话声明的 chunk size 和之前不一致，就拒绝合并。合并后再校验对象总大小，不一致就删除异常目标对象。

### 当前仍未做什么

当前没有在服务端重新计算最终文件 MD5 并与客户端 MD5 对比，因此 MD5 主要是客户端提供的内容标识和去重键。最终大小一致可以发现部分错误，但不能证明字节内容完全正确。

## 9. MinIO 是怎么合并的

### 正常路径：服务端 ComposeObject

客户端默认分片约 5MiB。后端把：

```text
chunks/<md5>/0
chunks/<md5>/1
...
```

按顺序构造成 `CopySrcOptions`，调用 MinIO `ComposeObject` 创建最终 `videos/<uuid>.<ext>` 对象。大分片场景下，字节主要在对象存储内部组合，应用服务不需要把完整文件读进内存再上传。

### 边界：非末尾 source 至少 5MiB

MinIO/S3 Compose 对非最后一个 source 有最小 5MiB 限制。项目实际修复过这个问题：如果客户端使用更小分片，直接 Compose 会失败。

当前 fallback：

```text
逐个 GetObject 打开分片
  → io.Pipe 按顺序流式复制
  → PutObject 写入最终对象
```

这样虽然数据经过应用进程，但仍是流式处理，不把完整视频一次加载到内存。

### 简历表述的精确边界

“使用 MinIO 服务端合并”对默认 5MiB 合法分片成立，但不是所有配置都必然走 Compose。面试中可主动补充：

> 正常分片走服务端 Compose；考虑到 MinIO 对非末尾 part 的 5MiB 限制，小分片会走流式合并 fallback。

这比假装没有组件限制更有说服力。

## 10. 为什么合并要加分布式锁

### 面试官问题

> 用户重复点击合并，或者两个请求同时到达，会怎样？

### 直接口语回答

> 两个请求同时合并同一 MD5，可能重复生成目标对象、重复插入资产，或者一个请求清理分片时另一个还在读取。因此 `MergeChunks` 按 `fileMD5` 获取 Redis 锁。拿不到锁时会再查询是否已经有资产；如果有就直接复用，否则提示合并正在进行。

### 锁不是唯一保障

还需要：

- `VideoAsset` 的内容标识约束和 `CreateOrRestore`；
- 已存在资产时直接创建任务复用；
- 合并失败时不提前清理源分片；
- 合并后大小校验。

不能把正确性全部归功于一把锁，也不要把它单独包装成“高并发亮点”。

---

## 11. MD5 在这里做什么，安全吗

### 直接口语回答

> MD5 在这里用于客户端识别同一文件、恢复同一上传会话和内容级资产复用，不用于密码存储、签名或安全防篡改。我知道 MD5 存在碰撞，不适合安全场景。当前还会比较文件大小，降低误复用，但如果要做更强完整性校验，可以在服务端合并后计算 SHA-256，或者使用分片哈希和最终哈希校验。

### 不要说

- “MD5 保证文件绝对唯一”；
- “MD5 保证文件没有被篡改”；
- “客户端传来的 MD5 可以无条件信任”。

---

## 12. 断点续传是不是 TCP 级续传

### 直接口语回答

> 不是。每个分片仍是一个独立 HTTP 上传请求。如果某一片传到一半连接断开，这一片通常要重新上传；系统能保留的是之前已经完整落到 MinIO 并记账的片段。恢复时客户端查询已完成片号并跳过它们，所以这是应用层的分片续传。

这一区分很重要：

```text
已完成分片可以跳过       —— 已实现
单个 HTTP 分片从字节断点继续 —— 未实现
```

---

## 13. Redis 挂了或数据丢失怎么办

### 直接口语回答

> 分片字节在 MinIO，所以 Redis 丢失不等于视频内容立即消失，但服务端会失去完成片号和原上传规格，客户端无法可靠知道哪些片可复用。当前项目依赖 Redis 的可用性和持久化配置，没有实现从 MinIO 扫描 `chunks/<md5>/` 自动重建会话。生产化可以把上传会话放 MySQL，或在 Redis 丢失时通过对象元数据重建，但这会增加复杂度。

这也是为什么不要说“Redis 保证分片永不丢失”。

---

## 14. 为什么不直接使用 MinIO Multipart Upload

### 直接口语回答

> MinIO/S3 原生 multipart upload 本身就支持分段上传和 complete，生产项目通常应优先评估它。我当前自己维护应用层协议，是因为还需要结合客户端 MD5 做已上传进度查询、内容资产复用和业务任务创建，也方便我完整实践会话规格校验。但从降低自研状态机复杂度看，原生 multipart 和预签名上传是很有竞争力的替代方案。

### 承认权衡

当前方案的学习价值在于能解释每个状态，但也意味着自己承担：

- 会话 TTL；
- 临时对象清理；
- 分片规格兼容；
- 合并幂等；
- Redis 与 MinIO 的一致性窗口。

不能把“自己实现”自动说成比原生 multipart 更高级。

---

## 15. 为什么不用本地磁盘合并

### 直接口语回答

> 单机项目当然可以把分片写本地磁盘再合并，组件更少。但后端如果重启、迁移或以后运行多个实例，本地临时文件不容易共享；而视频最终本来就要进入对象存储。当前让分片也进入 MinIO，默认情况下可以直接在对象存储端 Compose，应用服务不需要准备和视频一样大的本地合并空间。代价是引入对象存储依赖和临时对象管理。

---

## 16. 并发上传与限流

前端可以并发上传多个不同片段，但不能无限并发。面试中可以这样回答：

> 分片使并发上传成为可能，但并发数应受浏览器连接数、带宽、服务端内存和 MinIO 压力约束。当前重点是断点恢复和正确性，不应该在没有压测的情况下声称高并发上传能力。后续可以基于文件大小和网络状况调整并发数，并记录每片耗时、重试次数和失败率。

后端当前 `UploadChunk` 接收 `[]byte`，意味着单片会进入应用内存；它不是整文件内存加载，但也不是完全零拷贝流式接收。分片大小必须有上限。

## 17. 清理策略和失败场景

### 合并成功

- 创建最终对象；
- 校验大小；
- 创建/恢复资产；
- 尝试删除每个临时分片；
- 删除 Redis 片号集合；
- 清理失败只记录日志，不回滚已成功业务结果。

### 合并失败

源分片应继续保留，用户可以重试 merge；异常目标对象在已知大小不一致时会被删除。

### 清理失败

会浪费对象存储空间，但不应让已经创建成功的资产变为失败。生产化需要：

- MinIO lifecycle；
- 定时扫描过期 Redis 上传状态对应的孤儿分片前缀；
- 清理失败指标与告警；
- 对象与临时上传状态的可审计关系。

## 19. 当前限制与未来改进

### 当前限制

1. Redis 与 MinIO 之间没有分布式事务；
2. Redis 丢失后不能自动扫描 MinIO 重建会话；
3. 服务端主要校验最终大小，没有重新计算最终 MD5/SHA-256；
4. 废弃未合并分片仍需要更完整的生命周期清理；
5. 单片通过 `[]byte` 进入应用内存；
6. 没有足够实测证明 2GB、弱网成功率或吞吐指标；
7. 默认 5MiB 与 MinIO Compose 限制接近，配置变化必须走兼容逻辑；
8. 应用层协议比原生 S3 multipart 更需要维护。

### 合理未来改进

1. 预签名 URL，让客户端直接上传 MinIO，减少应用带宽；
2. 评估原生 multipart upload，并保存 upload ID；
3. 服务端合并后计算 SHA-256；
4. 每片校验 hash，发现传输错误；
5. MySQL 持久化上传会话，或支持从对象存储重建；
6. MinIO lifecycle 自动清理过期 `chunks/`；
7. 有界并发、重试和上传指标；
8. 用真实大文件和故障注入验证恢复行为。

这些应描述为未来方向，不是已完成事实。

## 20. 简历表述校准

当前表述基本成立，但“服务端合并”可在面试中补充 fallback：

> 采用分片上传与断点续传机制，通过 Redis 记录上传规格和已完成片号，分片写入 MinIO；合并时优先使用 MinIO ComposeObject，并针对小分片限制提供流式合并 fallback，避免上传中断后整文件重传。

简历若要保持简短，原句可以保留：

> 采用分片上传与断点续传机制，通过 Redis 记录分片状态并使用 MinIO 服务端合并，避免上传中断后整文件重传。

不要写：

> 基于 Redis + MinIO 实现零拷贝、秒传和高并发 GB 级文件上传，弱网成功率提升 99%。

原因：零拷贝不准确，秒传只在已有资产复用条件下成立，高并发和成功率也没有测试证据。

## 21. 一页速记

```text
【解决什么】
整文件上传中断后全部重传 → 把失败粒度缩小到单片。

【数据在哪里】
MinIO：分片字节；Redis：规格/状态/完成片号；MySQL：最终资产和任务。

【为什么先落盘后记账】
避免 Redis 显示完成，但 MinIO 实际没有对象。
允许重复上传，不允许假完成。

【怎么续传】
查询 uploaded indexes → 客户端跳过完成片 → 只补缺片。
是应用层续传，不是单片字节级 TCP 续传。

【怎么合并】
检查规格与每个片号 → MD5 锁 → 默认 MinIO Compose → 校验最终大小。
非末尾片 < 5MiB 时走流式 GetObject + PutObject fallback。

【MD5】
用于内容标识/会话/复用，不是安全哈希，也不保证绝对唯一。

【不能吹】
没有 GB 级与弱网指标；不是严格事务；临时片可能残留；
Redis 丢失会损害恢复；引用 Compose 不代表所有配置都服务端合并。
```
