## 2. Kafka 异步任务和 DB 重试

### 简历点原句

引入 Kafka 将视频下载、ASR 转写、摘要生成和 RAG 索引构建等长耗时任务异步化，HTTP 接口仅负责任务落库和消息投递，避免分钟级视频处理阻塞请求链路。

### 背景痛点

视频处理链路里最耗时的不是上传本身，而是后续处理：

- URL 下载可能受目标站点、网络和文件大小影响。
- FFmpeg 提取音频需要读取完整视频文件。
- ASR 转写会受音频时长和三方 API 稳定性影响。
- 摘要生成和 RAG 索引构建都依赖外部 AI 服务。

如果这些步骤都放在 HTTP 请求里同步执行，接口容易超时，用户页面也会一直等待。更严重的是，一旦中间步骤失败，很难知道任务卡在哪一步，也不好恢复。

### 主流程

可以这样讲：

1. 用户上传视频或提交视频 URL。
2. 后端创建 `video_tasks` 任务和对应 `task_jobs`。
3. 根据任务类型投递 Kafka 消息，比如下载、转写、分析、RAG 索引构建。
4. Consumer 消费消息后更新任务状态，执行对应业务逻辑。
5. 当前阶段成功后，再投递下一阶段消息。
6. 如果失败，先按错误类型判断是否可重试。
7. 可重试错误写入 DB，由 retry scheduler 按退避时间重新投递。
8. 不可重试错误或超过最大次数后，任务进入失败或 dead 状态。

### 具体实现

当前项目不是只用了 Kafka。Kafka 负责异步解耦，MySQL 负责业务状态和恢复。

关键实现点：

- Kafka topic 拆成 `video-download`、`video-transcribe`、`video-analyze`、`video-rag-index`。
- Producer 使用同步写入，`RequiredAcks = All`，`MaxAttempts = 3`。
- Consumer 使用手动 commit。
- 业务处理失败后，不是简单依赖 Kafka 重新投递，而是先把失败状态和错误原因写入 MySQL。
- `TaskRetryPolicy` 默认最大重试 3 次，退避时间是 60 秒、300 秒、900 秒。
- `retry.go` 会区分可重试和不可重试错误。
- 可重试错误包括网络超时、429、5xx、MinIO、Milvus 等外部依赖异常。
- 不可重试错误包括 AI 配置缺失、API Key 解密失败、未授权、文件不存在、embedding 维度不匹配、ASR 空结果、视频不可用等。

### 选型问题

**Q：为什么用 Kafka，不直接同步调用？**

答：

同步调用的问题是请求生命周期太短，而视频处理生命周期很长。上传接口应该只负责接收用户请求、保存元数据、投递任务，不能等待下载、FFmpeg、ASR、LLM、Milvus 全部完成。Kafka 的作用是把长耗时处理从 HTTP 请求里拆出去，让后端可以按阶段消费、失败恢复和削峰。

**Q：为什么用 Kafka，而不是 RabbitMQ 或 RocketMQ？**

答：

这个项目更看重任务流削峰、消费组并行和较高吞吐，Kafka 比较适合这种日志型异步任务链路。RabbitMQ 更偏传统队列和复杂路由，RocketMQ 对延迟消息、事务消息支持更强，但我当前项目的实现重点是 Go 后端、Kafka topic 拆分、MySQL 状态恢复，所以没有为了堆能力引入 RocketMQ。

补一句边界：

如果业务强依赖延迟消息或事务消息，RocketMQ 会更适合；如果只是轻量任务队列，RabbitMQ 也够用。我的选择是结合项目栈和当前需求做的。

**Q：为什么还需要 MySQL 任务表，Kafka 不是已经可靠了吗？**

答：

Kafka 解决的是消息传递，不解决业务状态可解释性。视频处理任务需要展示给用户看，比如处理中、转写中、建库中、失败原因、重试次数。Kafka offset 不能直接作为产品状态，也不适合承载业务失败原因。所以我用 Kafka 做异步流转，用 MySQL 的 `video_tasks` 和 `task_jobs` 记录业务状态。

### 具体实现追问

**Q：消费失败后为什么还 commit offset？**

答：

如果业务失败后不 commit，Kafka 会不断把同一条消息重新投递，容易造成热循环。例如三方 ASR 连续 429 时，消息会被立刻重复消费，反而加重外部服务压力。

我这里的做法是：

1. 消费到消息后执行任务。
2. 如果失败，先把错误类型、错误信息、重试次数和下一次重试时间写入 DB。
3. 然后 commit Kafka offset。
4. 后续由 DB retry scheduler 根据退避策略重新投递。

这样 Kafka 不负责业务重试节奏，DB 负责可观测和可控重试。

**Q：这样 commit 了 offset，消息不会丢吗？**

答：

严格说，Kafka offset 已经前进了，但业务任务没有丢，因为失败状态已经落库了。这里的可靠性边界从“Kafka 重新投递同一条消息”转成了“DB 中有一个待重试任务”。只要 DB 落库成功，retry scheduler 就能恢复。

要注意顺序：

先落业务失败状态，再 commit offset。如果反过来先 commit 后落库，中间进程挂掉才会真的丢业务恢复入口。

**Q：重复消费怎么处理？**

答：

Kafka 本身提供的是至少一次语义，不能假设绝对不重复。所以业务层要做幂等。

项目里的幂等主要靠几类手段：

- `task_jobs` 记录 job 状态，避免同一阶段重复执行。
- 转写分片通过 `task_id + chunk_index` 唯一约束持久化。
- RAG 索引构建前会按 task 和 embedding model 删除旧向量，再写入新 chunks。
- 上传侧用 MD5 内容指纹复用 `video_assets`。
- 分片上传用 Redis Set 记录已上传分片，重复上传同一分片不会破坏整体状态。

### 结合八股

可以主动往这些基础知识上靠：

- Kafka 消费语义：至少一次，需要业务幂等。
- offset：记录消费进度，不等于业务成功。
- consumer group：同一个 topic 可以按分区并行消费。
- ACK：Producer 等待 ISR 副本确认，提高消息写入可靠性。
- 最终一致性：HTTP 请求只创建任务，最终由异步消费者把状态推进到完成。
- 退避重试：避免瞬时故障时打爆外部服务。
- 失败分类：不是所有失败都应该重试。

### tradeoff

这个设计的代价：

- 链路变长，调试时要看 HTTP、Kafka、DB 状态和 Consumer 日志。
- 状态机复杂度提高，需要避免重复推进任务。
- 任务不是立即完成，前端必须轮询或订阅状态。
- DB retry scheduler 需要保证自身不会重复调度。

为什么仍然值得：

- 视频处理天然长耗时。
- 外部 AI 服务不稳定，必须有失败恢复入口。
- 任务状态需要展示给用户。
- 后续扩展并发消费和任务隔离更容易。

### 兜底保障

遇到失败时可以这样说：

- 网络超时、429、5xx 这类可恢复错误进入退避重试。
- 配置缺失、密钥解密失败、文件不存在这类不可恢复错误直接失败，避免无效重试。
- 超过最大重试次数后记录终态，避免无限消耗资源。
- 用户可以看到任务失败原因，后续可以手动重新触发。
- Consumer 侧不把所有异常都吞掉，而是沉淀到任务表里。

### 压力追问

**Q：你这个是不是就是 MQ 套壳？**

答：

不是只把同步代码塞进 MQ。关键点在于我把视频处理拆成了多个业务阶段，每个阶段有自己的状态、错误分类和恢复策略。Kafka 只负责异步解耦，真正保证可恢复的是 `task_jobs`、失败分类和 DB retry scheduler。

**Q：为什么不用工作流引擎？**

答：

当前链路是固定的：下载、转写、分析、RAG 索引构建，阶段数量不多，用工作流引擎会增加部署和学习成本。我先用 Kafka topic 加任务表实现轻量状态机。等后续出现动态 DAG、人工审批、复杂补偿，再考虑 Temporal 这类工作流框架会更合理。

**Q：如果 Consumer 挂了怎么办？**

答：

Consumer 挂了以后，未 commit 的消息会由 Kafka 在消费者恢复后重新分配消费。已经记录到 DB 的失败任务也可以由 retry scheduler 恢复。也就是说，短期挂掉影响处理速度，不应该直接导致业务任务永久丢失。

### 30 秒话术

我这个项目里视频处理不是同步做的，因为下载、FFmpeg、ASR、摘要和 RAG 索引构建都可能很慢。我的设计是上传接口只创建任务和投递 Kafka，后续由 Consumer 按阶段处理。失败时不会只依赖 Kafka offset 重试，而是先把失败原因、重试次数和下一次重试时间写到 MySQL，再由 DB retry scheduler 按退避策略重新投递。这样既能削峰，也能让任务状态对用户可见。

### 2 分钟话术

我把视频处理拆成了下载、转写、摘要分析、RAG 索引构建几个异步阶段，对应不同 Kafka topic。用户上传视频或提交 URL 后，后端先落 `video_tasks` 和 `task_jobs`，然后投递消息。Consumer 消费后执行对应业务逻辑，成功后推进状态并投递下一阶段。

失败处理上我没有简单依赖 Kafka 重放，因为三方 AI 服务如果 429 或超时，立即重放会造成热循环。所以 Consumer 失败后会先判断错误是否可重试，可重试错误比如 timeout、429、5xx、MinIO、Milvus 异常，会写入 DB 并设置下一次重试时间；配置缺失、密钥错误、文件不存在、embedding 维度不匹配这类错误直接终止。之后 Kafka offset 可以 commit，恢复入口变成 DB 中的待重试任务。这个设计的核心是 Kafka 做异步解耦，MySQL 做业务状态和可恢复性。

### 代码证据路径

- `cmd/server/main.go`：初始化 Kafka topics、producer、consumer、retry scheduler。
- `internal/mq/producer.go`：Kafka producer 配置，包含 `RequiredAcks` 和 `MaxAttempts`。
- `internal/mq/consumer.go`：下载、转写、分析、RAG 索引构建的消费逻辑。
- `internal/mq/retry.go`：失败分类、退避策略、终态记录。
- `internal/model/task.go`、`internal/model/task_job.go`：任务和 job 状态模型。

### 不要这么说

- 不要说“我实现了严格 exactly-once”。当前是 Kafka 至少一次加业务幂等。
- 不要说“Kafka 保证业务不会丢”。Kafka 保证消息传递，业务恢复靠 DB 状态。
- 不要说“用了 RocketMQ 延迟消息”。当前项目是 Kafka 加 DB retry scheduler。
- 不要说“完整工作流引擎”。当前是轻量任务状态机。

## 3. Redis 分布式锁和 MD5 内容去重

### 简历点原句

设计 Redis 分布式锁 + WatchDog，结合 MD5 内容指纹和视频文件复用，保护分片合并、分析任务消费等临界区，降低并发重复入库和重复处理风险。

### 背景痛点

视频处理的成本主要在后处理，而不是上传本身。同一个视频如果被同一个用户或多个用户重复上传，后端如果每次都重新 ASR、摘要和 RAG 索引构建，会浪费大量外部 AI 服务调用成本。

并发场景下还有一个问题：两个相同视频几乎同时上传时，都还没查到已有资产，就可能同时创建任务并进入后处理。

所以需要两层能力：

- 内容级去重：用文件 MD5 判断内容是否相同。
- 并发互斥：同一 MD5 的合并或入库过程只能有一个执行者。

### 主流程

可以这样讲：

1. 上传前端或后端计算文件 MD5。
2. 后端以 MD5 作为内容指纹。
3. 查 `video_assets` 是否已有相同 MD5 的视频资产。
4. 如果已有资产，直接复用对象存储里的文件，创建新的 `video_tasks` 引用它。
5. 如果没有资产，需要上传到 MinIO 并创建资产记录。
6. 对关键的入库或合并流程加 Redis 分布式锁，避免同一 MD5 并发创建多份资产。

### 具体实现

当前项目使用的是自实现 Redis 锁，不是 Redisson。

锁的关键点：

- 使用 `SETNX` 加 TTL 获取锁。
- 锁 value 使用 owner 标识，避免误删别人锁。
- watchdog 每隔 `ttl / 3` 自动续期。
- 解锁时用 Lua 脚本判断 owner 一致才删除。
- 续期时也需要校验 owner，避免锁过期后被别人拿到，旧执行者又错误续期。

内容去重的关键点：

- `video_assets` 存储真实视频资产。
- `video_tasks` 存储用户维度的视频任务。
- 多个任务可以引用同一个 asset。
- 删除任务时不是直接删除对象存储文件，而是判断是否还有其他任务引用。

### 选型问题

**Q：为什么用 Redis 锁，不用本地锁？**

答：

本地锁只在单进程内有效。如果后端水平扩容到多个实例，同一个 MD5 的上传请求可能落到不同机器，本地锁无法互斥。Redis 是独立服务，所有实例都能访问，所以适合做跨进程互斥。

**Q：为什么不用 MySQL 唯一索引就行？**

答：

MySQL 唯一索引可以兜底防止最终重复数据，但不能很好地保护前面的长耗时过程。例如 MinIO 合并、对象写入、AI 任务投递都可能发生在最终插入之前。Redis 锁可以把临界区前移，减少重复工作；MySQL 唯一索引更适合作为最终一致性的兜底。

**Q：为什么要 watchdog？**

答：

视频上传和合并可能超过固定锁 TTL。如果只设置 30 秒 TTL，业务还没执行完锁就过期了，另一个请求可能拿到锁并重复处理。watchdog 的作用是在持锁者还活着时续期，避免长任务中途锁失效。

### 具体实现追问

**Q：释放锁为什么要校验 owner？**

答：

因为锁可能过期。如果 A 拿到锁后执行太久，锁过期，B 又拿到了同一把锁。这时 A 执行完如果直接 `DEL key`，就会删掉 B 的锁。owner 校验可以保证只有锁的当前持有者才能释放。

**Q：watchdog 会不会导致死锁？**

答：

watchdog 只在持锁上下文存活时续期，并且续期要校验 owner。如果进程挂了，watchdog 也会停止，锁最终会因为 TTL 过期释放。所以它降低了长任务误过期的风险，但不会让锁永久存在。

**Q：Redis 挂了怎么办？**

答：

当前锁依赖 Redis，如果 Redis 不可用，严格并发去重能力会下降。兜底仍然要靠数据库唯一约束、任务幂等和对象存储命名策略。面试里我会承认这一点：当前实现适合单 Redis 或开发部署，生产上可以考虑 Redis 高可用、锁降级策略和更严格的数据库约束。

### 结合八股

可以关联这些点：

- `SETNX key value EX ttl` 是 Redis 分布式锁基础。
- 只加锁不加 TTL 可能死锁。
- 只加 TTL 不续期，长任务可能锁过期。
- 解锁必须比较 value，避免误删。
- Lua 脚本保证比较和删除原子性。
- 分布式锁不等于数据一致性，还需要 DB 约束兜底。

### tradeoff

这个设计的代价：

- 锁实现复杂度比直接插库高。
- Redis 可用性会影响并发控制。
- 长时间持锁会降低同一 MD5 请求的并发吞吐。
- 需要谨慎设置 TTL 和续期间隔。

为什么仍然值得：

- 视频处理成本高，重复执行代价明显。
- 文件合并和资产创建属于典型临界区。
- 锁加 MD5 去重可以把重复请求尽早挡住。

### 兜底保障

可以这样答：

- Redis 锁用于减少并发重复处理。
- MySQL 资产表和任务表用于最终状态表达。
- 对象存储删除时判断引用关系，避免误删共享资产。
- 重复请求即使穿透锁，也应尽量通过 MD5 和任务幂等避免重复后处理。

### 压力追问

**Q：你这个锁是不是不如 Redisson？**

答：

Redisson 是更成熟的工业级实现，包含可重入锁、watchdog、Lua 原子释放等能力。我这个项目没有引入 Java 生态的 Redisson，而是在 Go 里实现了项目需要的最小能力：`SETNX + TTL + owner + Lua 释放 + watchdog 续期`。如果是生产高并发系统，我会优先考虑成熟库或者更强的数据库约束组合。

**Q：MD5 会不会碰撞？**

答：

理论上 MD5 有碰撞可能，但在视频文件去重这个场景里，用它做工程上的内容指纹是常见折中。如果要进一步提高严谨性，可以组合文件大小、SHA-256、用户维度和对象元数据一起判断。

### 30 秒话术

视频处理成本比较高，所以我不希望相同视频重复触发 ASR、摘要和 RAG 索引构建。我用 MD5 作为内容指纹，真实文件存在 `video_assets`，用户任务存在 `video_tasks`，多个任务可以引用同一个资产。并发上传时，用 Redis 分布式锁保护同一 MD5 的合并和入库流程，锁里做了 TTL、watchdog 续期和 owner 校验释放，避免长任务锁过期或误删别人锁。

### 2 分钟话术

这个点的核心不是单纯加锁，而是把视频内容和用户任务拆开。`video_assets` 表示真实视频资产，`video_tasks` 表示某个用户的一次处理任务。相同 MD5 的视频可以复用同一个 asset，这样不会重复存储和重复处理。

并发问题上，如果两个相同视频同时上传，都发现 asset 还不存在，就可能同时写 MinIO、同时建任务、同时触发 AI 处理。所以我在关键流程上加 Redis 锁，用 MD5 构造 lock key。锁获取用 `SETNX` 加 TTL，value 是 owner；执行时间可能比较长，所以 watchdog 会按 `ttl / 3` 续期；释放时用 Lua 比较 owner 后删除，避免锁过期后误删其他请求持有的新锁。这个实现解决的是并发重复处理，最终一致性还要靠 MySQL 状态和唯一约束兜底。

### 代码证据路径

- `internal/pkg/lock/redis_lock.go`：Redis 锁、watchdog、Lua 解锁。
- `internal/service/media.go`：上传、分片、合并、asset 复用、删除引用处理。
- `internal/model/task.go`：`VideoAsset` 和 `VideoTask` 相关模型。

### 不要这么说

- 不要说“用了 Redisson”。当前是 Go 项目自实现 Redis 锁。
- 不要说“MD5 绝对不会冲突”。应该说工程上够用，生产可加 SHA-256 和文件大小。
- 不要说“分布式锁保证强一致”。锁是并发控制，最终一致还要靠数据库和幂等。

## 4. 分片上传、断点续传和 MinIO

### 简历点原句

采取分片上传与断点续传机制，通过 Redis Set 记录分片状态，使用 MinIO 存储分片并通过 ComposeObject 合并，提升弱网环境下大文件上传可靠性。

### 背景痛点

视频文件可能很大，单次上传有几个问题：

- 请求时间长，容易超时。
- 弱网中断后需要重新上传整个文件。
- 服务端一次性接收大文件会占用较多内存和连接资源。
- 用户体验差，上传失败成本高。

所以需要把大文件切成多个小分片，每个分片独立上传，失败时只补传缺失分片。

### 主流程

1. 前端按固定大小把文件切片。
2. 每个分片携带 `file_md5`、`chunk_number`、`total_chunks` 等信息上传。
3. 后端把分片写入 MinIO 的临时对象路径。
4. 后端用 Redis Set 记录这个文件 MD5 下已经上传的分片编号。
5. 前端可以查询已上传分片，跳过已经成功的部分。
6. 所有分片上传完成后，请求后端合并。
7. 后端校验分片完整性，使用 MinIO ComposeObject 合并成最终视频对象。
8. 合并后创建或复用 `video_assets`，再创建 `video_tasks`。

### 具体实现

当前实现的关键点：

- 分片对象路径类似 `chunks/{fileMD5}/{chunkNumber}`。
- Redis key 类似 `upload:chunks:{fileMD5}`。
- Redis Set 存储已上传的分片编号。
- 合并前校验 `total_chunks` 和已上传分片数量。
- 合并用 MinIO ComposeObject，不需要把所有分片读回后端内存再拼接。
- 合并和入库流程配合 Redis 锁，避免多个合并请求并发执行。
- 最终文件进入 `video_assets`，用户任务进入 `video_tasks`。

### 选型问题

**Q：为什么用 Redis Set 记录分片状态？**

答：

Set 天然适合存“已完成分片编号”这种去重集合。重复上传同一个分片，只是重复 `SADD`，不会产生多条状态。查询时可以快速知道哪些分片已完成，断点续传时前端只需要补传缺失分片。

**Q：为什么不用 MySQL 记录每个分片？**

答：

分片状态更新频繁，而且是上传过程中的临时状态。用 MySQL 也能做，但会带来大量临时行和清理成本。Redis 更适合这种短期、高频、可过期的状态。最终资产和任务状态仍然落 MySQL。

**Q：为什么用 MinIO，不直接存本地磁盘？**

答：

本地磁盘会绑定到某台机器，后端多实例部署时文件访问会有问题。MinIO 提供对象存储语义，适合存大文件，也方便后续替换成 S3 这类云对象存储。

### 具体实现追问

**Q：分片上传到 99% 断了怎么办？**

答：

前端重新进入上传时，可以用文件 MD5 查询已经上传的分片集合，只补传缺失分片。服务端 Redis Set 保存了已上传编号，所以不用重传整个文件。

**Q：合并时怎么确认分片都到了？**

答：

合并前会根据 `total_chunks` 检查 Redis Set 中的分片数量和编号。如果缺分片，就不执行合并，返回缺失信息或错误，让前端继续补传。

**Q：合并时为什么需要锁？**

答：

因为前端可能重复点击合并，或者多个请求同时触发同一个 MD5 的合并。如果不加锁，可能重复 ComposeObject、重复创建 asset、重复投递后处理任务。锁保护的是同一文件 MD5 的合并临界区。

**Q：分片临时文件怎么清理？**

答：

当前可以通过合并成功后删除临时对象，或者设置定时任务清理过期 `chunks/{fileMD5}`。如果面试官追问生产化，我会说临时对象和 Redis key 都应该设置生命周期策略，避免长期占用对象存储。

### 结合八股

可以关联：

- Redis Set 去重和集合查询。
- 对象存储适合大文件，不适合频繁小事务。
- 大文件上传的断点续传和幂等。
- 分布式锁保护合并临界区。
- 最终资产落 MySQL，临时上传状态放 Redis。

### tradeoff

代价：

- 上传协议更复杂，前端也要配合切片和补传。
- 需要处理临时分片清理。
- 合并失败时要设计恢复策略。
- 小文件直接分片反而增加请求次数。

收益：

- 大文件弱网体验明显更好。
- 服务端内存压力更低。
- 上传失败后只补传缺失分片。
- 后续可以支持秒传和内容复用。

### 兜底保障

- 分片编号用 Set 去重。
- 合并前校验分片完整性。
- 合并阶段加 Redis 锁。
- asset 层做 MD5 去重。
- 删除视频时判断引用，避免共享 asset 被误删。

### 压力追问

**Q：你这个断点续传是不是只做了一半？**

答：

断点续传最核心的是服务端能识别已经上传的分片，前端能跳过这些分片继续上传。当前项目用 Redis Set 保存分片编号，具备这个能力。更完整的生产版本还可以加临时分片 TTL、上传会话表、秒传接口和后台清理任务。

**Q：为什么不用 MinIO 原生 multipart upload？**

答：

MinIO 原生 multipart upload 很适合生产化对象上传。当前实现选择服务端管理分片对象和 ComposeObject，主要是为了让业务层更容易控制分片状态、MD5 去重和合并后的任务创建。后续如果要提高大文件上传性能，可以迁移到预签名 URL 加 MinIO multipart，让前端直传对象存储。

### 30 秒话术

大视频直接上传容易超时，弱网失败后重传成本很高。所以我做了分片上传，服务端把每个分片存到 MinIO，并用 Redis Set 记录已上传的分片编号。用户断网后重新上传，可以查询已完成分片，只补传缺失部分。全部分片到齐后，服务端用 MinIO ComposeObject 合并，合并和入库过程用 Redis 锁保护，最终通过 MD5 复用 `video_assets`。

### 2 分钟话术

这个链路是为大视频和弱网设计的。前端把文件切成多个 chunk，每个 chunk 上传时带上文件 MD5、分片编号和总分片数。后端收到后把分片写入 MinIO 临时路径，同时把分片编号写到 Redis Set。Set 的好处是天然去重，重复上传某个分片不会产生重复状态。

合并时，后端先根据 Redis Set 校验分片是否完整，不完整就拒绝合并。完整后用 MinIO ComposeObject 在对象存储侧合并，避免后端把所有分片读到内存。合并过程对同一 MD5 加 Redis 锁，防止重复合并和重复建资产。最终 `video_assets` 记录真实对象，`video_tasks` 记录用户任务，这样相同视频可以复用资产。

### 代码证据路径

- `internal/service/media.go`：`UploadChunk`、分片状态、合并、asset 复用。
- `internal/pkg/storage/minio.go`：MinIO 上传、合并、删除能力。
- `internal/pkg/lock/redis_lock.go`：合并临界区锁。
- `internal/model/task.go`：asset 和 task 模型。

### 不要这么说

- 不要说“完全生产级 multipart upload”。当前是业务层分片加 MinIO ComposeObject。
- 不要说“断点续传不依赖前端”。它需要前端查询状态并补传缺失分片。
- 不要说“Redis Set 是最终状态”。最终资产和任务状态在 MySQL。

## 5. Redis Lua 令牌桶限流

### 简历点原句

基于 Redis Lua 令牌桶限流，按用户和接口维度限制高成本 AI 请求，缓解恶意请求和重复点击带来的外部 AI 服务调用成本。

### 背景痛点

这个项目接入了外部 AI 服务，成本和稳定性都受请求量影响：

- 恶意用户可以频繁触发视频处理或问答。
- 用户重复点击可能导致短时间大量请求。
- AI 服务有速率限制，超过后会返回 429。
- 高并发请求会拖慢后端和外部依赖。

所以需要在进入核心业务前做限流。

### 主流程

1. 请求进入 Gin middleware。
2. 根据 route 和 userID 生成限流 key。
3. 如果没有登录，则按 IP 维度限流。
4. Redis Lua 脚本读取当前 token 数和上次刷新时间。
5. 按时间差补充 token。
6. token 足够则扣减并放行。
7. token 不足则返回限流错误。

### 具体实现

当前实现：

- 使用 Redis Hash 保存令牌桶状态。
- Lua 脚本保证读取、补充、扣减是原子的。
- key 粒度是 route 加 user 或 IP。
- Redis 异常时采用 fail-open，也就是放行请求。

fail-open 的含义是：限流系统故障时，不让整个业务因为 Redis 抖动直接不可用。

### 选型问题

**Q：为什么用令牌桶？**

答：

令牌桶既能限制平均速率，又允许一定突发流量。比如用户连续点几次按钮，只要桶里还有 token，可以允许短暂突发；但长期请求频率会被限制住。相比固定窗口，令牌桶更平滑，不容易在窗口边界出现双倍流量。

**Q：为什么用 Lua？**

答：

限流要做读取 token、计算补充、判断、扣减、写回。如果这些操作分多条 Redis 命令执行，并发下可能出现竞态。Lua 脚本在 Redis 内部原子执行，可以保证同一 key 的限流状态一致。

**Q：为什么 Redis 出错时放行？**

答：

这是可用性和保护能力的权衡。限流是保护系统的中间层，如果 Redis 短暂异常就拒绝所有请求，会让限流器本身变成故障点。当前项目选择 fail-open，优先保证核心功能可用。生产上可以对高成本接口单独配置 fail-close 或降级策略。

### 结合八股

可以关联：

- Redis Lua 原子性。
- 令牌桶和漏桶区别。
- 固定窗口、滑动窗口、令牌桶的优缺点。
- 用户维度和 IP 维度限流。
- fail-open 和 fail-close 的权衡。
- 限流和 MQ 削峰的区别。

### tradeoff

代价：

- Redis 变成限流依赖。
- route 加 user 维度比较粗，不是精细配额系统。
- fail-open 时 Redis 故障期间保护能力下降。
- 没有实现复杂的套餐、余额、按 token 计费。

收益：

- 能快速拦截重复点击和恶意高频请求。
- Lua 保证并发扣减准确。
- 对 AI 成本和外部 API 429 有保护作用。

### 兜底保障

- 对高成本接口配置较低速率。
- 对普通查询接口配置更宽松速率。
- Redis 失败时记录日志并放行，避免主业务完全不可用。
- 后续可结合 `user_usage_daily` 做日级额度。

### 压力追问

**Q：限流和 Kafka 削峰有什么区别？**

答：

限流发生在请求入口，是为了挡掉过高频率或恶意流量；Kafka 削峰发生在任务进入系统之后，是为了让后端按消费能力慢慢处理。限流解决“要不要进来”，Kafka 解决“进来后怎么异步处理”。

**Q：只按 route 加 user 限流够吗？**

答：

当前项目够用，但不是完整商业配额系统。更精细可以按接口成本、模型、视频时长、用户套餐、日额度一起控制。当前实现的重点是先保护高成本入口，避免无限制触发 AI 调用。

### 30 秒话术

因为项目里 ASR、LLM、Embedding 都会产生外部调用成本，所以我在入口做了 Redis 令牌桶限流。限流 key 按 route 加 userID，没有登录时按 IP。Lua 脚本在 Redis 里原子完成 token 补充和扣减，避免并发下计数不准。Redis 异常时当前选择 fail-open，优先保证主流程可用。

### 2 分钟话术

这个限流主要保护高成本 AI 请求。请求进入后，middleware 会根据接口和用户生成限流 key，然后执行 Redis Lua 脚本。脚本里保存两个核心状态：当前 token 数和上次刷新时间。每次请求根据时间差补充 token，再判断是否够扣减。够就扣一个 token 放行，不够就返回限流错误。

我选择 Lua 是因为限流需要多步读写，如果拆成多条 Redis 命令，并发情况下可能出现超发。令牌桶相比固定窗口更平滑，也允许合理突发。当前实现的 tradeoff 是它还不是完整计费系统，只是入口保护；Redis 异常时 fail-open，保证业务可用，但保护能力会短暂下降。

### 代码证据路径

- `internal/middleware/ratelimit.go`：令牌桶限流 middleware 和 Lua 脚本。
- `cmd/server/main.go`：middleware 注册。
- `internal/model/ai_call_log.go`：可结合日级用量统计扩展配额。

### 不要这么说

- 不要说“实现了完整计费系统”。当前是限流和用量统计，不是账单系统。
- 不要说“Redis 挂了也能严格限流”。当前 Redis 异常时 fail-open。
- 不要说“令牌桶解决所有成本问题”。它是入口保护，成本还要结合任务去重、重试限制和用量审计。

## 6. FFmpeg 长视频音频切片和 ASR

### 简历点原句

基于 FFmpeg 对长视频提取 16k 单声道低码率音频并按时间切片，结合 ASR 分片转写、分片结果持久化和转写复用，提升长视频转写成功率和失败恢复能力。

### 背景痛点

长视频直接送 ASR 有几个问题：

- 视频文件体积大，上传到 ASR 服务成本高。
- ASR 服务通常对文件大小或时长有限制。
- 长音频一次转写失败后需要整体重试。
- 摘要和 RAG 都依赖转写结果，转写失败会阻塞后续链路。

所以要先用 FFmpeg 抽取更适合 ASR 的音频，再按时间切片。

### 主流程

1. Consumer 拿到转写任务。
2. 从 MinIO 获取视频文件或本地临时路径。
3. FFmpeg 提取音频，去掉视频流。
4. 音频转成 16k、单声道、低码率 mp3。
5. 按默认 300 秒切片。
6. 对每个音频切片调用 ASR。
7. 每个切片转写结果落库。
8. 合并切片文本得到完整转写。
9. 后续摘要和 RAG 索引构建复用完整转写。

### 具体实现

当前 FFmpeg 参数重点：

- `-vn`：不处理视频流。
- `-ac 1`：单声道。
- `-ar 16000`：16k 采样率。
- `-acodec libmp3lame`：mp3 编码。
- `-b:a 32k`：低码率，减少音频体积。

切片策略：

- 默认每 300 秒一个音频片段。
- 转写分片结果持久化到 `video_transcription_chunks`。
- 完整转写写入 `video_transcriptions`。
- 摘要阶段优先复用已有转写，不重复调用 ASR。

### 选型问题

**Q：为什么先抽音频？**

答：

ASR 只需要音频，不需要视频画面。直接上传视频会浪费网络和处理成本。抽取音频后文件更小，处理更快，也更符合 ASR 服务输入。

**Q：为什么转 16k 单声道？**

答：

语音识别不需要高清双声道音频。16k 单声道通常已经能满足语音识别要求，同时显著降低文件大小和上传耗时。

**Q：为什么 300 秒切片？**

答：

这是成功率和调用次数之间的折中。切得太长，单片失败成本高，也可能触发 ASR 限制；切得太短，请求次数变多，上下文断裂更明显。300 秒是一个工程上的默认值，后续可以根据 ASR provider 的限制动态调整。

### 具体实现追问

**Q：切片会不会把一句话切断？**

答：

会有这个风险。当前按时间切片，可能在句子中间断开。工程上先保证长视频能稳定处理，后续可以优化成带 overlap 的音频切片，或者根据静音检测切分，减少语义断裂。

**Q：为什么分片结果要持久化？**

答：

长视频转写中某个切片失败时，如果前面切片已经成功，不应该全部重做。分片持久化后，可以复用已完成结果，只重试失败部分。同时后续排查 ASR 问题时，也能定位是哪个切片异常。

**Q：摘要为什么复用转写结果？**

答：

摘要的输入是文本，不需要重新处理视频或音频。如果每次摘要都重新 ASR，会浪费成本，也会引入不一致。复用已有转写可以减少调用成本，并保证摘要和 RAG 使用同一份文本基础。

### 结合八股

可以关联：

- 长任务拆分和失败恢复。
- 幂等：`task_id + chunk_index` 唯一标识一个转写分片。
- 存储优化：视频转音频降低数据量。
- 外部接口限流和重试。
- 批处理和局部重试。

### tradeoff

代价：

- 切片会带来上下文断裂。
- 分片数量增加后 ASR 请求数变多。
- FFmpeg 依赖外部二进制，部署时需要环境准备。
- 音频临时文件需要清理。

收益：

- 长视频成功率更高。
- 单片失败可以局部恢复。
- 音频体积更小，ASR 成本更低。
- 转写结果可复用给摘要和 RAG。

### 兜底保障

- FFmpeg 失败会进入任务失败和重试链路。
- ASR 网络失败、429、5xx 可以退避重试。
- ASR 空结果属于不可重试或需要人工检查的异常。
- 已完成的切片转写可以复用，减少重复处理。

### 压力追问

**Q：你这个 ASR 是不是只是调接口？**

答：

ASR 调接口只是其中一步。项目里真正要处理的是长视频怎么变成稳定可处理的音频、失败怎么恢复、分片结果怎么持久化、摘要和 RAG 怎么复用转写结果。如果只是简单调接口，长视频超时、大小限制和中途失败都会很难处理。

**Q：如果一个切片转写失败，整体任务怎么办？**

答：

当前会把错误交给任务失败分类和重试策略处理。可重试错误进入退避重试，不可重试错误记录失败原因。因为切片结果已落库，后续可以优化为只重试失败切片，而不是重做全部。

### 30 秒话术

长视频不能直接丢给 ASR，因为文件大、耗时长、失败后重试成本高。我先用 FFmpeg 去掉视频流，把音频转成 16k 单声道低码率 mp3，再按默认 300 秒切片。每个切片单独 ASR，结果持久化到转写分片表，最后合并成完整转写。摘要和 RAG 都复用这份转写，避免重复调用 ASR。

### 2 分钟话术

这个链路的核心是提高长视频处理稳定性。用户上传的是视频，但 ASR 只需要音频，所以我先用 FFmpeg 提取音频，并降低采样率、声道数和码率，减少文件大小。然后按时间把音频切片，每个切片单独调用 ASR。这样即使视频很长，也不会把一个超大音频一次性提交给三方服务。

分片转写结果会持久化，一方面方便失败恢复，另一方面方便排查是哪个音频片段有问题。完整转写合并后写入 `video_transcriptions`，后续摘要和 RAG 索引构建都复用它。这个设计的不足是按时间切片可能切断句子，后续可以用静音检测或 overlap 优化。

### 代码证据路径

- `internal/pkg/ffmpeg/ffmpeg.go`：音频提取和切片参数。
- `internal/mq/consumer.go`：转写任务消费、切片 ASR、结果合并。
- `internal/model/transcription.go`、`internal/model/transcription_chunk.go`：完整转写和分片转写模型。
- `internal/service/rag_index.go`：基于转写内容建 RAG 索引。

### 不要这么说

- 不要说“我实现了 ASR 模型”。当前是调用外部 ASR provider。
- 不要说“切片完全不影响语义”。当前按时间切片，有语义断裂风险。
- 不要说“已经实现失败切片精确续跑到生产级”。当前有分片持久化和任务重试基础，可以作为后续优化方向。

## 7. RAG 混合检索、RRF 和 citations

### 简历点原句

基于 RAG 检索增强生成实现视频智能问答，将转写文本切分为 chunk 并生成向量写入 Milvus，结合 BM25 关键词召回与 RRF 融合排序，返回引用片段提升答案可解释性。

### 背景痛点

视频问答不能只靠大模型自己回答，因为：

- 大模型不知道用户刚上传的视频内容。
- 直接把整段转写塞进 prompt，长视频容易超过上下文窗口。
- 只做摘要会丢细节，用户问具体片段时答不上来。
- 没有引用来源时，用户不知道答案依据是什么。

RAG 的目标是先从视频转写中找相关片段，再让模型基于片段回答。

### 主流程

1. ASR 完成后拿到完整转写文本。
2. 按 chunk size 和 overlap 切成文本块。
3. 对每个 chunk 调用 Embedding。
4. chunk 元数据落 MySQL。
5. 向量写入 Milvus。
6. 用户创建 chat session 并提问。
7. 对问题做 embedding。
8. Milvus 做向量召回。
9. MySQL 做关键词召回。
10. 用 RRF 融合两路结果。
11. 把引用片段和最近对话拼成 prompt。
12. LLM 生成答案。
13. 返回答案和 citations，并保存问答记录和检索快照。

### 具体实现

当前实现要点：

- 默认 chunk size 是 800，overlap 是 120。
- embedding 维度默认 1536，但实际以用户 AI profile 配置为准。
- Milvus collection 里保存 `user_id`、`task_id`、`chunk_id`、`chunk_index`、`embedding_model`、`content`、`embedding`。
- 搜索时 filter 带 `user_id`、`task_id`、`embedding_model`，避免跨用户、跨视频、跨模型污染。
- MySQL keyword retrieval 通过 `SearchByBM25` 做关键词召回。
- `FuseRetrievedChunks` 用 RRF 融合 vector 和 keyword 结果。
- `citations` 返回 chunk 信息、来源、rank 和得分。
- 问答消息保存到 `chat_messages`，assistant 消息保存 retrieval snapshot。

### 选型问题

**Q：为什么不用摘要做问答？**

答：

摘要适合回答“这个视频讲了什么”，但不适合回答细节问题。比如用户问某个时间点、某句话、某个步骤，摘要可能已经把这些信息压缩掉了。RAG 基于转写原文切片检索，可以保留更多细节。

**Q：为什么不用整段转写直接塞给 LLM？**

答：

长视频转写可能非常长，直接塞进 prompt 会超过上下文限制，也会增加 token 成本。即使没超过，模型也可能在长上下文里忽略关键细节。RAG 先召回相关片段，可以控制 prompt 大小，也更容易解释答案来源。

**Q：为什么 Milvus 还要配 BM25 关键词召回？**

答：

向量检索擅长语义相似，但对专有名词、数字、代码、短词有时不稳定。关键词召回能补足这类精确匹配。两路召回后用 RRF 融合，既保留语义召回，又提高关键词命中率。

### 具体实现追问

**Q：RRF 是什么？**

答：

RRF 是 Reciprocal Rank Fusion，用排名倒数来融合多个检索结果。它不要求不同检索器的分数在同一尺度上，只看排名。公式可以简化说成：一个 chunk 在某一路排得越靠前，贡献越大；如果它同时被向量和关键词召回，分数会叠加，排序会更靠前。

**Q：为什么不用直接加权分数？**

答：

向量检索分数和关键词检索分数不是同一种尺度，直接加权需要大量调参。RRF 基于 rank，更稳健，适合项目初期做混合召回。

**Q：为什么检索要带 user_id 和 task_id？**

答：

这是权限隔离和检索准确性的要求。用户只能问自己的视频，问题也只能在当前视频范围内检索。如果不带这些 filter，可能召回其他用户或其他视频的内容，既有安全问题，也会污染答案。

**Q：embedding_dim 不一致怎么办？**

答：

Milvus collection 的向量维度是固定的。如果用户换了 embedding 模型，维度不一致会导致写入或搜索失败。当前项目会把 embedding model 和 dim 作为配置，并把维度不匹配归为不可重试错误，避免盲目重试。

### 结合八股

可以关联：

- 向量检索解决语义相似，关键词检索解决精确匹配。
- RAG 本质是检索加生成，不是训练模型。
- prompt 控制和上下文窗口。
- 多租户权限隔离。
- MySQL 和向量数据库的数据一致性。
- 离线评估指标：Recall@K、MRR、source mix。

### tradeoff

代价：

- RAG 索引构建多了 embedding 调用成本。
- Milvus 增加部署和维护复杂度。
- chunk 切分不合理会影响召回。
- 当前没有 rerank，融合排序仍有提升空间。
- 向量库和 MySQL 之间有一致性问题，需要失败恢复。

收益：

- 长视频问答不受完整转写长度限制。
- 可以回答细节问题。
- citations 提升可解释性。
- 混合召回比单向量召回更稳。

### 兜底保障

- 建库前先删除旧向量，再替换新 chunks，避免旧索引污染。
- Milvus 初始化失败时服务可降级，核心上传和转写不必完全不可用。
- 检索不到上下文时拒绝编造答案。
- 返回 citations，让用户知道答案依据。
- 问答和检索快照落库，便于排查错误回答。

### 压力追问

**Q：你怎么证明 RAG 效果变好了？**

答：

我不会只说主观感觉。项目里准备了 RAG eval 思路，可以用一组带期望关键词的问题集评估 Recall@K 和 MRR，并统计结果来自 vector、keyword 还是 hybrid。这样能对比纯向量召回和混合召回在命中率上的差异。

**Q：为什么现在没有 rerank？**

答：

rerank 能提升排序质量，但会增加一次模型调用，带来延迟和成本。当前项目先用 vector 加 keyword 的混合召回和 RRF 融合，复杂度较低，也能解决一部分召回问题。后续如果评估发现 topK 里有相关片段但排序靠后，再加 rerank 更合理。

**Q：Milvus 挂了怎么办？**

答：

Milvus 挂了会影响 RAG 索引构建和问答检索，但不应该影响用户上传和基础转写。项目启动时对 Milvus 有超时和降级思路，任务失败也会进入重试链路。生产上还要加健康检查、告警和向量索引重建能力。

### 30 秒话术

RAG 这块我不是用摘要问答，而是基于 ASR 转写原文建索引。转写文本会按 800 字左右切 chunk，带 120 overlap，然后生成 embedding 写入 Milvus，同时 chunk 元数据落 MySQL。用户提问时先做向量召回，再用 MySQL 做关键词召回，最后用 RRF 融合排序，把相关片段放进 prompt，并把 citations 返回给前端，避免模型脱离视频内容乱答。

### 2 分钟话术

这个项目的视频问答基于 RAG，因为大模型本身不知道用户上传的视频内容，直接塞完整转写又容易超上下文和增加成本。我的链路是：ASR 完成后拿到完整转写，按固定 chunk size 和 overlap 切片，然后对每个 chunk 做 embedding。chunk 内容、模型、维度、hash、vector_id 等元数据写 MySQL，向量写 Milvus。

问答时，先对问题做 embedding，在 Milvus 里按 `user_id + task_id + embedding_model` 做向量检索，保证只查当前用户当前视频。然后再用 MySQL 做关键词召回，补足专有名词、数字和短词问题。两路结果用 RRF 融合，因为向量分数和关键词分数尺度不同，直接加权不稳定。最终把融合后的 citations 和最近对话拼 prompt，让 LLM 基于这些片段回答。答案和检索快照会落库，便于追溯。

### 代码证据路径

- `internal/service/rag_index.go`：转写切 chunk、embedding、写 MySQL 和 Milvus。
- `internal/service/chunk_splitter.go`：chunk size 和 overlap 切分。
- `internal/vector/milvus.go`：Milvus collection、字段、filter、搜索。
- `internal/service/chat.go`：RAG 问答主链路、prompt、citations、消息保存。
- `internal/service/retrieval_fusion.go`：RRF 融合和 source 标记。
- `internal/service/rag_eval.go`：Recall@K、MRR 等评估。

### 不要这么说

- 不要说“训练了自己的大模型”。这是 RAG 应用，不是模型训练。
- 不要说“已经用了 rerank”。当前没有 rerank。
- 不要说“已经接入 Elasticsearch”。当前关键词召回是 MySQL 侧实现。
- 不要说“RAG 完全解决幻觉”。它降低幻觉，但仍要靠 prompt 约束和 citations。

## 8. SSE 流式问答

### 简历点原句

基于 SSE 实现视频问答流式响应，服务端先推送检索 citations，再增量推送模型 answer 和 done 事件，降低长回答场景下的用户等待感。

### 背景痛点

RAG 问答比普通接口慢，因为它包含：

- 问题 embedding。
- 向量检索。
- 关键词召回。
- RRF 融合。
- LLM 生成答案。

如果等完整答案生成后一次性返回，用户会感觉页面卡住。SSE 可以让服务端边生成边推送。

### 主流程

1. 前端请求流式问答接口。
2. 后端设置 SSE 响应头。
3. 服务端先完成 RAG 检索。
4. 推送 `citations` 事件，让前端先拿到引用片段。
5. 如果 provider 支持 streaming，调用 `StreamChat`。
6. 每个模型 delta 通过 `answer` 事件推给前端。
7. 完成后推送 `done` 事件。
8. 服务端保存完整答案和检索快照。

### 具体实现

当前实现：

- Handler 里使用 Gin 的 `c.SSEvent(event.Type, event.Data)`。
- 每次事件后 `Flush()`。
- Service 层 `AskStream` 先 emit citations。
- 如果 chat client 实现 `StreamingChatClient`，使用 provider 的真流式。
- 如果不支持 streaming，则生成完整答案后按小段拆分推送，保证接口形态一致。
- 最终仍然保存完整 assistant message。

### 选型问题

**Q：为什么用 SSE，不用 WebSocket？**

答：

这个场景是服务端向客户端单向推送答案，客户端不需要在同一个连接上高频双向通信。SSE 基于 HTTP，浏览器原生支持，断线重连和服务端实现都更简单。WebSocket 更适合聊天室、协同编辑这类双向实时场景。

**Q：SSE 有什么限制？**

答：

SSE 主要是单向推送，不适合复杂双向交互；部分代理需要注意 buffering；浏览器对同域连接数也有限制。对当前问答流式输出场景，这些限制可以接受。

**Q：为什么先推 citations？**

答：

因为 citations 是检索阶段的结果，通常比完整 LLM 答案更早得到。先返回 citations 可以让前端先展示“模型依据哪些视频片段回答”，也方便用户知道不是凭空生成。

### 结合八股

可以关联：

- HTTP 长连接。
- SSE 和 WebSocket 区别。
- 流式响应和缓冲刷新。
- 后端 context cancel：用户断开连接后应停止生成。
- 幂等和消息保存：流式输出完成后保存完整消息。

### tradeoff

代价：

- 流式接口比普通 JSON 接口更难处理错误。
- 中途断开时可能只有部分答案。
- 代理层如果缓冲，前端可能看不到实时效果。
- fallback split 不等于 provider 原生 streaming。

收益：

- 长答案体验更好。
- citations 可以更早展示。
- 接口复杂度低于 WebSocket。

### 兜底保障

- provider 支持 streaming 时走真流式。
- provider 不支持时，用完整答案拆分成 answer chunks。
- 中途异常发送 error 事件。
- 完成后发送 done 事件。
- 保存完整答案，避免刷新后丢失历史记录。

### 压力追问

**Q：你这个是真流式还是假流式？**

答：

当前有两种路径。如果底层 chat client 支持 `StreamChat`，就读取 provider 的 SSE delta 并逐段推给前端，这是真流式。如果 provider 不支持 streaming，则服务端先拿完整答案，再拆分成小段推送，这是兼容 fallback。面试里我会明确区分这两种情况。

**Q：SSE 中途失败怎么办？**

答：

中途失败时，后端会推送 error 事件并 flush。已经生成的部分前端可以展示为临时内容，但最终是否落库要看服务端有没有拿到完整答案。生产上可以增加 partial message 状态，当前项目主要保存完整问答。

### 30 秒话术

RAG 问答生成完整答案可能比较慢，所以我做了 SSE 流式接口。服务端先完成检索并推送 citations，让用户先看到引用片段；然后如果模型 provider 支持 streaming，就把模型 delta 作为 answer 事件持续推给前端，最后发送 done。相比 WebSocket，这个场景主要是服务端单向推送，SSE 更简单。

### 2 分钟话术

流式问答的链路是：前端调用 stream 接口，后端进入 `AskStream`。服务端先做问题 embedding、向量检索、关键词召回和 RRF 融合，这一步完成后立刻 emit citations。然后判断 chat client 是否支持 streaming，如果支持，就调用 provider 的 `StreamChat`，每收到一个 delta 就通过 Gin 的 `SSEvent` 推送 `answer` 事件并 flush；如果不支持，就走普通 Ask 得到完整答案，再拆分成多个 answer chunk，保持前端接口一致。

最后服务端发送 done 事件，并保存完整答案、citations 和检索快照。这个设计的 tradeoff 是中途失败处理比普通 JSON 复杂，但换来的是长回答体验更好。

### 代码证据路径

- `internal/handler/chat.go`：SSE handler、`SSEvent`、`Flush`。
- `internal/service/chat.go`：`AskStream`、citations、answer、done 事件。
- `internal/ai/chat.go`：OpenAI-compatible `StreamChat`。
- `internal/ai/observed.go`：streaming 调用审计包装。
- `internal/service/chat_test.go`：流式事件顺序测试。

### 不要这么说

- 不要说“所有模型都是真流式”。只有支持 `StreamChat` 的 provider 是真流式。
- 不要说“SSE 能替代所有 WebSocket 场景”。SSE 适合单向推送。
- 不要说“中途断开完全无影响”。断开时需要处理 partial answer。

## 9. BYOK、API Key 加密和 AI 调用审计

### 简历点原句

设计 BYOK 用户 AI 配置体系，支持用户分别配置 ASR、LLM 和 Embedding provider，并使用 AES-GCM 加密存储 API Key，结合 AI 调用日志和日级用量统计增强成本可观测性。

### 背景痛点

如果公开部署 AI 应用，直接使用平台自己的 API Key 有几个问题：

- 用户请求会消耗平台额度。
- 不同用户希望使用不同模型和 endpoint。
- API Key 泄露风险集中在平台侧。
- 不方便统计每个用户的 AI 调用情况。

BYOK 的思路是让用户配置自己的 AI provider 和 key，平台只负责加密保存和调用。

### 主流程

1. 用户在前端配置 LLM、ASR、Embedding 的 provider、endpoint、model 和 API Key。
2. 后端校验配置。
3. API Key 使用 AES-GCM 加密后保存。
4. 返回给前端时只返回 masked key。
5. 调用 AI 前，后端读取用户默认 profile 并解密。
6. 根据 profile 创建对应 client。
7. AI 调用经过 observer 包装，记录调用日志和日级用量。

### 具体实现

当前实现要点：

- `UserAIProfile` 存储用户 AI 配置。
- LLM、ASR、Embedding 三套配置分开。
- API Key 密文保存，不明文返回。
- 更新配置时，如果用户不传新 key，可以保留原密文。
- `GetDefaultAIProfile` 获取用户默认配置。
- 如果用户没有配置，返回“请先配置 AI 服务”，而不是偷偷 fallback 到服务端 key。
- `AICallLog` 记录 kind、provider、model、status、duration、输入输出长度、错误等。
- `UserUsageDaily` 做用户日级调用统计。

### 选型问题

**Q：为什么 ASR、LLM、Embedding 要分开配置？**

答：

因为这三类能力的模型、价格、endpoint 和调用方式都不同。用户可能想用一个 provider 做 ASR，用另一个 provider 做 LLM，再用 OpenAI-compatible embedding 模型做向量。分开配置更灵活，也能分别统计成本和错误。

**Q：为什么 API Key 用 AES-GCM？**

答：

AES-GCM 是对称加密里的认证加密模式，既能加密，也能校验密文是否被篡改。API Key 需要在调用时解密，所以不能只做哈希；同时不能明文入库，所以用可解密的对称加密。

**Q：为什么不使用 bcrypt 存 API Key？**

答：

bcrypt 是单向哈希，适合密码校验，不适合 API Key。平台调用 AI provider 时需要拿到原始 key，所以必须能解密。用户密码用 bcrypt，API Key 用 AES-GCM，这两个场景不一样。

### 具体实现追问

**Q：密钥泄露怎么防？**

答：

当前做了几层：

- DB 中保存的是密文。
- 返回前端只返回 masked key。
- 日志里不打印明文 key。
- 调用时按用户读取自己的 profile，避免跨用户访问。

边界也要承认：当前主密钥仍然在服务端配置里，生产上可以接 KMS 或密钥轮换机制。

**Q：为什么不直接做平台统一 key？**

答：

统一 key 对用户最简单，但平台承担所有成本和风控压力。BYOK 更适合个人项目或公开部署，用户用自己的额度，平台只做调度和加密保存。当然，商业化产品也可以支持平台 key 加套餐计费，这是另一个方向。

**Q：AI 调用审计有什么用？**

答：

它可以回答三个问题：

- 谁调用了什么模型。
- 调用成功还是失败，耗时多少。
- 每天大概用了多少请求和输入输出长度。

这些信息对排查三方 API 问题、统计成本和限制滥用都有用。

### 结合八股

可以关联：

- 密码哈希和可逆加密区别。
- AES-GCM 的认证加密。
- 多租户数据隔离。
- 敏感信息脱敏。
- 审计日志和成本统计。
- 配置中心和密钥轮换。

### tradeoff

代价：

- 用户上手成本增加，需要自己配置 key。
- 后端要处理不同 provider 的 endpoint 和 model。
- 密钥管理变复杂。
- 配置错误会导致任务失败。

收益：

- 降低平台统一 key 的成本风险。
- 用户模型选择更灵活。
- 调用审计更清晰。
- 适合公开部署和个人项目演示。

### 兜底保障

- profile 缺失时直接提示配置，不继续调用。
- key 解密失败归类为不可重试错误。
- 前端只展示 masked key。
- AI 调用日志记录状态和错误，便于排查。
- 可结合 rate limit 和 daily usage 限制高成本调用。

### 压力追问

**Q：用户 API Key 存你这里，安全吗？**

答：

我不会说绝对安全。当前项目做的是基本工程防护：密文入库、AES-GCM、返回脱敏、日志不打印明文、按用户隔离读取。生产上还需要 KMS、密钥轮换、更细的权限控制和操作审计。面试里我会把当前实现和生产增强边界讲清楚。

**Q：如果用户配置恶意 endpoint 呢？**

答：

这是 BYOK 带来的风险。当前 profile 主要解决 provider 配置和密钥加密，生产上应该对 endpoint 做白名单、协议限制、内网地址拦截和请求超时控制。这个点可以和 URL 下载的 SSRF 防护思路结合。

### 30 秒话术

公开部署 AI 应用时，如果所有请求都用平台自己的 key，成本和风控压力很大。所以我做了 BYOK，用户可以分别配置 ASR、LLM、Embedding 的 provider、endpoint、model 和 API Key。API Key 用 AES-GCM 加密入库，返回前端只展示脱敏值。调用时按用户解密自己的 profile，同时用 AI observer 记录调用日志和日级用量。

### 2 分钟话术

BYOK 这块主要解决公开部署时的成本和模型灵活性问题。用户创建 AI profile 时，需要分别填写 LLM、ASR、Embedding 的 provider、endpoint、model 和 key。后端校验后，不会明文保存 key，而是用 AES-GCM 加密保存。返回给前端时只返回 masked key。更新配置时，如果用户不填新 key，可以保留旧密文，避免每次修改都重新输入。

在实际调用时，服务会读取用户默认 profile 并解密，然后创建对应的 AI client。这个 profile 是用户维度的，不会跨用户共享。如果没有配置，就直接返回“请先配置 AI 服务”，不会偷偷 fallback 到服务端 key。调用过程会经过 observer，记录 provider、model、kind、状态、耗时、输入输出长度和错误，并累加到日级用量统计里。当前边界是还没有接 KMS 和完整计费系统，生产上可以继续增强。

### 代码证据路径

- `internal/service/ai_profile.go`：AI profile 创建、更新、解密、脱敏。
- `internal/model/ai_profile.go`：用户 AI 配置模型。
- `internal/pkg/secret/crypto.go`：AES-GCM 加解密。
- `internal/service/ai_observer.go`：AI 调用日志和用量统计。
- `internal/model/ai_call_log.go`：`AICallLog` 和 `UserUsageDaily`。
- `internal/ai/factory.go`：按 profile 创建 AI client。

### 不要这么说

- 不要说“API Key 绝对安全”。只能说做了加密、脱敏和隔离。
- 不要说“用了 KMS”。当前是服务端 secret 加 AES-GCM。
- 不要说“已经实现完整 token 计费”。当前是调用审计和日级用量统计。
- 不要说“没配置用户 key 时自动用平台 key”。当前 BYOK resolver 下要求用户配置。

## 10. URL 下载和 SSRF 防护

### 简历点原句

针对 URL 视频下载场景实现 SSRF 防护，限制 http/https 协议和 B 站、YouTube 等白名单域名，并在 DNS 解析后拦截 localhost、内网、回环、链路本地和组播地址。

### 背景痛点

URL 下载看起来只是给后端一个视频链接，但安全风险很高。攻击者可能提交：

- `http://localhost:3306`
- `http://127.0.0.1`
- `http://169.254.169.254`
- 内网 IP 或内网域名
- 非 http/https 协议

如果后端直接用下载工具访问，就可能变成攻击内网服务的跳板，这就是 SSRF 风险。

### 主流程

1. 用户提交视频 URL。
2. 后端解析 URL。
3. 校验协议只能是 http 或 https。
4. 校验域名在支持列表中，比如 Bilibili、b23、YouTube、youtu.be。
5. DNS 解析域名。
6. 检查解析结果是否为内网、回环、本地链路、组播等地址。
7. 校验通过后创建下载任务。
8. 下载任务进入 Kafka，由 Consumer 异步执行。

### 具体实现

当前实现：

- URL 下载不是同步执行，而是投递 `video-download` job。
- URL validator 会限制协议。
- 支持域名白名单，不是任意 URL。
- DNS 解析后判断 IP 类型。
- 对 URL 做 sanitize。
- 下载失败会进入任务失败和重试链路。
- 对 B 站 412、视频不可用等错误做失败分类。

### 选型问题

**Q：为什么要限制域名白名单？**

答：

因为这个产品本身只需要支持公开视频平台，不需要让后端下载任意 URL。白名单能显著缩小攻击面，比黑名单更可靠。黑名单很容易漏掉各种内网地址写法和 DNS 变化。

**Q：为什么 DNS 解析后还要检查 IP？**

答：

只看域名字符串不够。攻击者可以用一个看起来正常的域名解析到内网 IP，也可能利用 DNS rebinding。解析后检查 IP 类型，可以拦截解析到本地、内网、回环、链路本地等地址的请求。

**Q：为什么 URL 下载也进 Kafka？**

答：

远程下载受网络、目标站点限速、视频大小影响，不适合同步阻塞请求链路。进入 Kafka 后，用户请求可以快速返回任务 ID，后端异步下载并更新状态，失败也能重试或展示原因。

### 结合八股

可以关联：

- SSRF 原理。
- URL 解析和协议限制。
- DNS 解析和私网 IP 判断。
- 白名单优于黑名单。
- 异步下载和任务状态。
- 网络超时和资源限制。

### tradeoff

代价：

- 支持的视频站点更少。
- 用户不能提交任意视频 URL。
- 白名单需要维护。
- DNS rebinding 在生产上还需要更严格的下载时校验和网络隔离。

收益：

- 大幅降低 SSRF 风险。
- 下载任务更可控。
- 失败原因更清晰。
- 更符合产品定位。

### 兜底保障

- 只允许 http/https。
- 只允许白名单域名。
- DNS 解析后检查 IP。
- 下载任务异步执行，避免 HTTP 长时间占用。
- 下载错误进入失败分类。
- 生产上可加容器网络隔离、下载服务独立部署和 egress 策略。

### 压力追问

**Q：只做这些就完全防住 SSRF 了吗？**

答：

不能说完全防住。当前实现已经做了协议、白名单、DNS 解析和私网 IP 拦截，是应用层的基础防护。生产上还应该加网络层隔离，比如下载服务不能访问内网数据库和 metadata 服务，并且下载时也要控制重定向、超时和最大文件大小。

**Q：为什么不支持任意 URL？**

答：

因为产品目标是公开视频解析，不是通用下载器。任意 URL 会显著扩大安全风险和内容合规风险。先支持明确的视频平台更可控。

### 30 秒话术

URL 下载有 SSRF 风险，所以我没有让后端下载任意地址。用户提交 URL 后，后端先校验协议只能是 http/https，再校验域名是否在 B 站、YouTube 等白名单中，并且 DNS 解析后检查是不是 localhost、内网、回环或链路本地地址。校验通过后才创建下载任务并投递 Kafka 异步处理。

### 2 分钟话术

URL 下载是一个安全敏感点，因为后端如果直接访问用户传入的地址，攻击者可能让服务访问内网数据库、本机端口或云厂商 metadata 地址。所以我做了多层校验：第一层限制协议，只允许 http/https；第二层限制域名白名单，只支持产品需要的视频平台；第三层对域名做 DNS 解析，解析后检查 IP 是否属于内网、回环、本地链路、组播等范围。

校验通过后，也不是同步下载，而是创建任务并投递到 Kafka。这样远程下载慢或失败不会阻塞接口，失败原因也能进入任务状态。这个防护不是绝对完美，生产上还要加网络隔离、重定向校验、最大文件大小和下载超时控制。

### 代码证据路径

- `internal/service/remote_video_url.go`：URL 协议、白名单、DNS 和私网 IP 校验。
- `internal/service/media.go`：URL upload 创建任务。
- `internal/mq/consumer.go`：下载任务消费。
- `internal/pkg/ytdlp`：远程视频下载封装。
- `internal/mq/retry.go`：视频不可用、Bilibili 412 等错误分类。

### 不要这么说

- 不要说“完全防住 SSRF”。应该说做了应用层基础防护，生产还要网络隔离。
- 不要说“支持任意 URL”。当前是白名单公开视频平台。
- 不要说“URL 下载同步完成”。当前是 Kafka 异步任务。

## 11. MySQL 表设计、幂等和最终一致性

### 简历点原句

围绕视频资产、用户任务、异步 Job、转写、摘要、RAG chunk、问答消息和 AI 调用日志拆分 MySQL 表结构，支撑任务状态追踪、内容复用、幂等处理和最终一致性。

### 背景痛点

如果只用一张 `video_tasks` 大表保存所有东西，会有几个问题：

- 真实视频资产和用户任务混在一起，无法复用相同视频。
- 转写文本、摘要、RAG chunk、聊天记录都塞进任务表，会导致表臃肿。
- 异步任务每个阶段的失败原因和重试次数不好表达。
- 问答需要保存多轮消息和检索快照，不适合放在视频任务表。

所以需要按业务边界拆表。

### 主流程

可以这样讲表之间关系：

- `users`：用户。
- `video_assets`：真实视频文件资产，按 MD5 复用。
- `video_tasks`：用户发起的一次视频处理任务。
- `task_jobs`：任务内部的异步阶段和重试状态。
- `video_transcriptions`：完整转写。
- `video_transcription_chunks`：音频分片转写结果。
- `ai_summaries`：AI 摘要。
- `video_chunks`：RAG 文本 chunk 元数据。
- `video_rag_indexes`：某个视频某个 embedding model 的建库状态。
- `chat_sessions`：针对某个视频的问答会话。
- `chat_messages`：用户和 assistant 消息，包含 retrieval snapshot。
- `user_ai_profiles`：用户 BYOK 配置。
- `ai_call_logs`、`user_usage_daily`：AI 调用审计和用量统计。

### 具体实现

核心拆分逻辑：

- asset 和 task 拆开：支持同一视频多用户或多任务复用。
- task 和 job 拆开：任务是用户视角，job 是内部处理阶段。
- transcription 和 transcription chunks 拆开：完整文本用于摘要和 RAG，分片文本用于长视频恢复。
- video chunks 和 Milvus vector 拆开：MySQL 保存元数据和关键词召回，Milvus 保存向量检索。
- chat session 和 message 拆开：支持同一视频多轮、多会话问答。
- AI profile 和 call log 拆开：配置和审计分离。

### 选型问题

**Q：为什么 `video_assets` 和 `video_tasks` 要拆开？**

答：

asset 是文件内容，task 是用户的一次处理请求。相同视频可以被不同用户或同一用户多次使用，如果不拆开，就无法复用对象存储文件和处理结果，也不好做安全删除。拆开后，一个 asset 可以被多个 task 引用。

**Q：为什么有 `task_jobs`，任务表里一个 status 不够吗？**

答：

一个 status 只能表达用户视角的整体状态，比如 processing、completed、failed。但内部有下载、转写、分析、RAG 索引构建多个阶段，每个阶段都有自己的错误、重试次数和下一次重试时间。`task_jobs` 可以把阶段状态拆开，方便恢复和排查。

**Q：为什么 RAG chunk 既存 MySQL 又存 Milvus？**

答：

Milvus 负责向量检索，MySQL 负责业务元数据、关键词召回和状态管理。只存在 Milvus 里，不方便做普通业务查询、权限管理和关键词召回；只存在 MySQL 里，又没有高效向量检索。

### 结合八股

可以关联：

- 数据建模：一对多、多对一、复用和引用。
- 索引和唯一约束支撑幂等。
- 大字段拆表。
- 业务状态机。
- 最终一致性：任务状态由异步 job 推进。
- 数据删除：共享 asset 的引用计数或引用检查。

### tradeoff

代价：

- 表数量增加，查询链路更复杂。
- 需要维护跨表一致性。
- 删除任务时要处理转写、摘要、chunk、向量和对象存储。
- 开发初期不如单表简单。

收益：

- 业务边界清晰。
- 支持内容复用。
- 支持异步阶段追踪。
- 支持 RAG 和问答审计。
- 后续扩展更容易。

### 兜底保障

- 用唯一索引保证分片转写、RAG chunk 等幂等写入。
- 删除视频时清理关联转写、摘要、chunks、RAG index 和向量。
- 共享 asset 删除前检查引用。
- job 失败状态和 retry count 落库。
- 问答保存 retrieval snapshot，便于追溯。

### 压力追问

**Q：表拆这么多是不是过度设计？**

答：

如果只是一个 demo，单表确实更快。但这个项目的业务对象本来就不同：文件资产、用户任务、异步阶段、转写文本、RAG chunk、聊天消息、AI 配置和审计日志的生命周期都不一样。拆表不是为了复杂，而是为了复用、幂等、失败恢复和后续查询。

**Q：怎么保证 MySQL 和 Milvus 一致？**

答：

当前建库流程会先处理旧向量，再替换 MySQL chunks 和写入 Milvus。失败时会记录 RAG index 状态，任务可重试。严格分布式事务没有做，因为成本太高；这里采用最终一致性和可重试补偿。

### 30 秒话术

我没有把视频所有数据都塞在一张表里，而是按生命周期拆分。`video_assets` 表示真实文件，`video_tasks` 表示用户任务，`task_jobs` 表示内部异步阶段；转写、摘要、RAG chunks、问答消息、AI 配置和调用日志分别拆表。这样能支持相同视频复用、异步状态追踪、失败重试、RAG 检索和问答审计。

### 2 分钟话术

表设计是围绕视频处理链路拆的。首先 asset 和 task 要分开，因为同一个视频文件可能被多个任务引用，真实对象不应该重复存储。其次 task 和 job 要分开，因为用户看到的是整体任务状态，但内部有下载、转写、分析、RAG 索引构建多个阶段，每个阶段都有独立失败和重试策略。

然后转写结果也拆成完整转写和分片转写，完整转写给摘要和 RAG 用，分片转写用于长视频恢复。RAG 这块，MySQL 保存 chunk 元数据和关键词召回，Milvus 保存向量。问答用 session 和 message 表保存多轮上下文，assistant message 里还保存 retrieval snapshot，方便追溯答案依据。这个设计的代价是表多、链路复杂，但换来的是复用、幂等和可恢复。

### 代码证据路径

- `internal/model/model.go`：统一 AutoMigrate 模型列表。
- `internal/model/task.go`：`VideoAsset`、`VideoTask`。
- `internal/model/task_job.go`：异步 job 状态。
- `internal/model/transcription.go`、`internal/model/transcription_chunk.go`：转写表。
- `internal/model/video_chunk.go`、`internal/model/rag_index.go`：RAG chunk 和索引状态。
- `internal/model/chat.go`：chat session 和 message。
- `internal/model/ai_profile.go`、`internal/model/ai_call_log.go`：BYOK 和 AI 审计。

### 不要这么说

- 不要说“强事务保证 MySQL 和 Milvus 完全一致”。当前是最终一致性和重试补偿。
- 不要说“删除 asset 只要删任务就行”。共享 asset 要检查引用。
- 不要说“task_jobs 是完整工作流引擎”。它是项目内轻量 job 状态表。

## 12. 用户鉴权和权限隔离

### 简历点原句

基于 JWT 和用户维度数据隔离实现视频任务、AI 配置、RAG 检索和问答会话的访问控制，避免跨用户读取视频内容、API Key 和检索结果。

### 背景痛点

这个项目里用户数据比较敏感：

- 视频内容可能包含个人信息。
- ASR 转写是视频完整文本。
- RAG chunk 可以还原视频内容。
- 用户 API Key 是敏感凭证。
- 聊天记录包含用户问题和模型回答。

所以权限隔离不能只做页面隐藏，后端查询必须带用户维度。

### 主流程

1. 用户注册时密码 bcrypt 哈希。
2. 登录成功后生成 JWT。
3. 前端请求携带 Bearer token。
4. Gin middleware 解析 token，取出 userID。
5. 后续 handler 从 context 获取 userID。
6. 查询任务、AI profile、chat session、RAG 检索都带 userID。
7. 不属于当前用户的数据返回未授权或不存在。

### 具体实现

当前实现：

- JWT 使用 HS256。
- claims 包含 user_id、username、role、issuer。
- 密码使用 bcrypt。
- AI profile 查询按 userID。
- RAG 检索 filter 带 user_id 和 task_id。
- chat session 属于用户和任务。
- MinIO 对象不是直接公开给所有用户，而是由后端控制访问。

### 选型问题

**Q：为什么用 JWT？**

答：

JWT 适合前后端分离应用，后端可以无状态解析 token，拿到用户身份。当前项目是 Vue 加 Go API，JWT 接入简单，不需要服务端维护 session。代价是 token 签发后在过期前不容易主动失效，生产上可以加短 token、refresh token 或黑名单。

**Q：为什么密码用 bcrypt？**

答：

bcrypt 是专门用于密码存储的慢哈希算法，带 salt，可以降低数据库泄露后被离线撞库的风险。密码不需要还原，所以用单向哈希；API Key 调用时需要还原，所以用 AES-GCM。

**Q：RAG 为什么也要做用户隔离？**

答：

RAG chunk 本质上是视频转写内容。如果检索时不带 user_id 和 task_id，就可能把其他用户的视频片段召回到当前回答里，既是严重隐私问题，也会导致答案错误。

### 结合八股

可以关联：

- JWT 结构和签名校验。
- 无状态认证和 session 的区别。
- bcrypt 和 AES-GCM 的区别。
- 多租户数据隔离。
- 后端权限校验不能只靠前端隐藏。
- 对象存储访问控制。

### tradeoff

代价：

- JWT 主动注销较麻烦。
- HS256 需要保护好服务端 secret。
- 每个 repository 查询都要注意 userID 条件。
- 对象存储如果要做临时访问，还需要签名 URL 或代理下载。

收益：

- 前后端分离接入简单。
- 用户数据隔离清晰。
- RAG 和 BYOK 安全边界更明确。

### 兜底保障

- 后端 middleware 统一解析 token。
- 查询用户资源时带 userID。
- AI profile 只能按当前用户读取。
- RAG 检索 filter 带 user_id。
- API Key 不明文返回。

### 压力追问

**Q：只用 JWT 安全吗？**

答：

JWT 是认证方案的一部分，不等于全部安全。还要配合 HTTPS、合理过期时间、secret 管理、后端权限校验和敏感数据脱敏。当前项目实现了基础登录鉴权和用户维度隔离，生产上可以继续加 refresh token、黑名单和操作审计。

### 30 秒话术

用户登录后后端发 JWT，前端请求带 Bearer token。后端 middleware 解析 userID，后续任务、AI profile、chat session 和 RAG 检索都按 userID 查询。尤其 RAG 检索会带 `user_id + task_id + embedding_model` filter，避免召回其他用户的视频片段。密码用 bcrypt，API Key 用 AES-GCM，这是两个不同场景。

### 2 分钟话术

这个项目的数据隔离比较重要，因为视频、转写、RAG chunk、问答记录和用户 API Key 都是敏感数据。认证上我用 JWT，用户登录成功后生成带 user_id 的 token，前端每次请求放到 Authorization header。后端 middleware 解析后把 userID 放到 context，handler 和 service 再用这个 userID 查询资源。

权限隔离不是只在页面做，而是在后端查询里做。例如 AI profile 必须按当前用户查；chat session 也属于某个用户和任务；RAG 检索时 Milvus filter 带 user_id 和 task_id，避免跨用户召回。密码存储用 bcrypt，因为密码不需要还原；API Key 要调用三方服务，需要可逆加密，所以用 AES-GCM。

### 代码证据路径

- `internal/service/user.go`：用户注册、登录、bcrypt。
- `internal/pkg/jwt/jwt.go`：JWT 生成和解析。
- `internal/middleware/auth.go`：Bearer token 鉴权。
- `internal/service/ai_profile.go`：用户维度 AI profile。
- `internal/service/chat.go`：chat session 和 RAG 检索用户隔离。
- `internal/vector/milvus.go`：Milvus filter。

### 不要这么说

- 不要说“JWT 能主动踢下线”。当前没有完整 token 黑名单机制。
- 不要说“对象存储文件公开访问”。应说由后端控制访问和删除。
- 不要说“只靠前端做权限控制”。真正隔离在后端查询条件。

## 13. 前端和 AI Coding 怎么讲

### 简历点原句

使用 Vue 3 和 Vite 实现视频上传、任务状态、AI 配置、摘要展示和 RAG 问答页面，并借助 AI Coding 提升页面原型和接口联调效率。

### 背景痛点

这个项目主要竞争力在后端和 AI 应用链路，前端不要喧宾夺主。前端可以作为“我能独立完成全链路”的证明，但不要把它包装成复杂前端工程。

### 主流程

可以这样讲：

1. 前端提供登录、视频上传、URL 提交、任务列表、任务详情。
2. 用户配置 AI profile。
3. 上传成功后展示任务状态。
4. 前端轮询任务状态，直到转写、摘要和 RAG 状态完成。
5. 用户进入问答页，发送问题。
6. 普通问答返回 JSON，流式问答走 SSE。

前端重点：

项目重点是后端视频处理和 AI 链路，前端需要快速实现一个可用管理界面。Vue 3 和 Vite 开发效率高，适合快速做上传、任务状态和问答页面。

**Q：前端怎么知道异步任务完成？**

当前主要通过任务状态接口轮询。因为视频处理是异步的，上传接口只返回任务 ID，后续前端定时查询任务状态。等状态变成 completed 或 failed，再展示摘要、转写或错误原因。后续可以换成 SSE 或 WebSocket 推任务状态。

**Q：AI Coding 在项目中怎么体现？**

我主要把 AI Coding 用在前端页面初稿、接口封装和测试用例整理上，但不会直接照收。比如接口字段、错误处理、鉴权逻辑都需要根据后端实际 handler 和响应格式核对。面试里我会强调 AI 提效，但最终代码正确性由我负责。

### 结合八股

可以关联：

- 前后端分离。
- JWT token 存储和鉴权 header。

- API 统一响应封装。
- 前端测试覆盖接口 envelope 和任务状态策略。

### tradeoff

代价：

- 轮询会产生额外请求。
- 前端不是复杂设计系统，视觉和交互还有优化空间。

收益：

- 能完整演示上传到问答的闭环。

## 14. 横向高频追问速查

### Q1：这个项目最大的难点是什么？

推荐答：

最大的难点不是调某一个 AI API，而是把视频处理做成稳定的异步链路。因为它同时涉及大文件上传、远程下载、FFmpeg、ASR、LLM、Embedding、Milvus 和对象存储，任何一个外部依赖失败都会影响任务。所以我重点做了 Kafka 异步拆分、MySQL 任务状态、失败分类、退避重试、Redis 去重和 RAG 索引构建恢复。

### Q2：你这个项目是不是就是调 API？

推荐答：

如果只是调 API，长视频、重复上传、失败恢复、RAG 检索、BYOK 和成本控制都解决不了。我项目里的 API 调用只是能力入口，工程重点在输入如何处理、任务如何异步化、失败如何恢复、结果如何落库、问答如何基于视频内容而不是凭空生成。

### Q3：你项目里最能体现后端能力的是哪几个点？

推荐答：

我会讲三个点：

1. Kafka 加 DB retry 的异步任务链路。
2. Redis 锁、MD5 去重、分片上传和 MinIO 的大文件处理。
3. RAG 混合检索、RRF 和 citations 的问答链路。

这三个点分别对应可靠性、大文件工程和 AI 应用落地。

### Q4：如果让你优化下一版，你先做什么？

推荐答：

我会按收益排序：

1. 加更完整的任务进度推送，比如 SSE 推任务状态，减少轮询。
2. 给 RAG 加 rerank，并用评估集对比 Recall@K 和 MRR。
3. URL 下载服务独立部署并做网络隔离，加强 SSRF 防护。
4. BYOK 接 KMS 和密钥轮换。
5. 用 Prometheus/Grafana 做生产级指标监控。

### Q5：怎么证明这是你写的？

推荐答：

我可以从启动入口 `cmd/server/main.go` 讲初始化链路，再从一次上传请求讲到 `MediaService`、Kafka producer、Consumer、FFmpeg、ASR、RAGIndexService、ChatService。也可以具体看 `internal/mq/retry.go` 的失败分类、`internal/pkg/lock/redis_lock.go` 的锁释放 Lua、`internal/service/rag_index.go` 的 chunk 建库和 `internal/service/chat.go` 的混合检索问答。

### Q6：项目目前最明显的不足是什么？

推荐答：

我会讲真实不足：

- RAG 目前还没有 rerank。
- 任务状态主要靠轮询，实时进度还可以用 SSE 推送优化。
- BYOK 密钥管理还没接 KMS。
- URL 下载 SSRF 主要是应用层防护，生产上还要网络隔离。
- 监控主要是 AI 调用日志和任务状态，还不是完整 Prometheus 告警体系。

### Q7：你如何处理外部 AI 服务不稳定？

推荐答：

我把外部 AI 错误分成可重试和不可重试。网络超时、429、5xx 这类进入退避重试；配置缺失、密钥解密失败、embedding 维度不匹配、ASR 空结果这类直接失败。调用日志会记录 provider、model、状态、耗时和错误，方便后续排查。

### Q8：如何控制 AI 调用成本？

推荐答：

主要有几层：

- BYOK 让用户使用自己的 key。
- Redis 限流控制高频请求。
- MD5 去重避免相同视频重复处理。
- 转写结果复用，摘要和 RAG 不重复 ASR。
- 失败重试有最大次数，避免无限重试。
- AI 调用日志和日级用量统计提供可观测性。

### Q9：你怎么讲最终一致性？

推荐答：

用户请求完成不代表视频处理完成，请求只创建任务和投递消息。后续由 Kafka Consumer 异步推进状态。中间如果失败，失败原因和重试信息落 MySQL，由 scheduler 重新投递。最终任务可能成功、失败或 dead，但每个状态都有记录，这就是最终一致性而不是同步强一致。

### Q10：如果用户删除视频，怎么处理 RAG 数据？

推荐答：

删除任务时不只是删 `video_tasks`，还要清理转写、摘要、RAG chunks、RAG index、chat 相关数据，并删除 Milvus 中对应 task 和 embedding model 的向量。对象存储里的真实视频 asset 还要看是否被其他任务引用，不能直接删共享文件。

## 15. 最后一页背诵模板

### 项目总介绍

VidLens 是一个 AI 视频理解平台，用户可以上传本地视频或提交公开视频 URL，系统会异步完成视频下载、音频提取、ASR 转写、摘要生成和 RAG 索引构建，最后支持基于视频内容的智能问答。后端用 Go、Gin、GORM、MySQL、Redis、Kafka、MinIO、Milvus、FFmpeg 和 OpenAI-compatible AI provider，前端用 Vue 3 和 Vite。

### 技术主线

我会从三条线讲：

第一条是视频处理可靠性：Kafka 异步任务、MySQL job 状态、失败分类、退避重试。

第二条是大文件、并发和成本控制：分片上传、MinIO、Redis 分布式锁、MD5 内容指纹、视频文件复用和限流。

第三条是 AI 问答效果：ASR 转写、chunk 索引构建、Milvus 向量检索、BM25 关键词召回、RRF 融合和 citations。

### 最推荐现场展开的 5 个故事

1. 为什么不能同步处理视频，所以做 Kafka 加 DB retry。
2. 为什么相同视频不能重复处理，所以做 MD5 内容指纹、视频文件复用和 Redis 锁。
3. 为什么大视频弱网体验差，所以做 Redis Set + MinIO ComposeObject 分片续传。
4. 为什么不能只用摘要问答，所以做基于 ASR 原文的 RAG。
5. 为什么外部依赖异常不能无脑重试，所以做错误分类和阶梯退避重投递。

### 面试边界

可以说：

当前项目已经实现了异步任务、上传、转写、摘要、RAG、限流、失败治理、SSE、BYOK、SSRF 基础防护和 AI 调用审计。最终简历主线先讲前六个工程点，SSE、BYOK 和 SSRF 放追问补充。

不要说：

- 已经实现完整工作流引擎。
- 已经实现严格 exactly-once。
- 已经接入 Elasticsearch。
- 已经实现 rerank。
- 已经接入 KMS。
- 已经有生产级计费系统。
- 已经有完整 Prometheus 告警体系。

### 结尾话术

这个项目我会重点讲工程落地，而不是只讲 AI API。因为真正难的是视频处理链路长、外部依赖多、成本高、失败点多。我主要用 Kafka 和 MySQL 状态解决异步可靠性，用 Redis、MinIO 和 MD5 解决大文件和重复处理，用 RAG 混合检索和 citations 解决问答准确性和可解释性。
