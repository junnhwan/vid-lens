# VidLens 简历项目面试防守总览

> 用途：把简历中的四条技术描述串成一条真实业务链路，并作为各专题复习文档的导航。  
> 核心原则：先讲视频处理遇到了什么具体问题，再讲技术如何解决；不把个人项目包装成高并发生产系统，也不因为用了中间件就默认方案合理。

---

## 1. 当前简历原文

### AI 长视频内容理解与智能问答平台

**项目简介：**一个 AI 驱动的长视频内容理解与智能问答平台，支持大文件分片上传、长视频分段转写、AI 摘要和带引用的视频问答，并通过异步任务调度与失败重试处理耗时且易受外部服务影响的视频处理链路。

- 引入 Kafka 异步调度 ASR、摘要生成和 RAG 索引等长耗时任务，结合任务状态与阶梯退避重试，实现阶段追踪和异常恢复；
- 设计长视频分段 ASR 与结果持久化机制，任务异常后仅重跑失败片段并复用已完成结果，避免整段视频重复处理；
- 采用分片上传与断点续传机制，通过 Redis 记录分片状态并使用 MinIO 服务端合并，避免上传中断后整文件重传；
- 基于 Milvus + BM25 + RRF 实现语义与关键词混合检索，并返回引用片段，提升视频问答结果的可追溯性。

这四条不是四个互不相关的“技术名词”，而是围绕同一条视频链路解决四类问题。

```text
大文件一次上传容易因网络中断而全部重传
  → 分片上传，Redis 记录会话和进度，分片对象暂存在 MinIO

视频处理要经过媒体处理和多次外部 AI 调用，无法在 HTTP 请求中等待
  → 后台异步任务、持久化状态和失败重试

长音频整段调用 ASR 会撞输入限制，也可能返回明显不完整的结果
  → 按时长切片、片段结果落库、失败后复用已完成片段

摘要只能回答整体内容，无法支持细粒度追问和证据定位
  → 对 ASR 原文分块，向量与关键词混合检索，返回引用片段
```

## 2. 30 秒项目介绍

> VidLens 是我做的一个长视频内容理解后端。用户上传视频后，系统会提取音频、分段调用 ASR、保存转写并生成摘要，之后把转写原文建立 RAG 索引，支持带引用的视频问答。这个项目的重点不是简单调用一次大模型，而是处理长视频带来的大文件上传、长耗时任务、外部 AI 失败和转写结果复用问题。我用分片上传解决上传中断后的重传问题，用异步任务和持久化状态处理长链路，再用分段 ASR 和混合检索让任务可以恢复、回答可以定位到原始片段。

这个版本适合自我介绍阶段。不要一上来把 Kafka、Redis、MinIO、Milvus 全部报一遍，否则容易给人“技术栈清单”的印象。

## 3. 2 分钟项目介绍

> 我做这个项目的起点是，课程和技术视频比较长，只看一段摘要不够，我还希望能针对视频里的细节提问，并看到答案依据。实现以后我发现，真正困难的不是调用模型，而是把长视频处理成一条可恢复的后台任务链路。  
>  
> 上传方面，大文件如果整文件上传，中途失败就要从头再来。所以客户端按片上传，服务端把每片写入 MinIO，Redis 只记录上传规格和已经完成的片号。恢复时客户端查询进度并跳过已完成片段，所有片齐全后再合并，并校验最终大小。  
>  
> 处理方面，视频需要经过音频提取、分段 ASR、摘要和 RAG 索引，单次可能持续几分钟，而且依赖外部 AI 服务。我没有让 HTTP 请求一直等待，而是使用 Kafka 调度后台任务，同时在 MySQL 中保存业务阶段、错误和重试状态。这里 Kafka 不是唯一答案，当前规模用 MySQL 队列也可以；我更看重的是可靠异步、持久化状态和失败恢复，而不是为了声称高并发。  
>  
> ASR 方面，我实际遇到过长音频超过接口输入限制，以及任务显示成功但转写明显过短的问题。因此现在统一按时长切片，每个片段的状态和文本单独落库。任务失败后重新执行时，会跳过已经完成且有内容的片段，只重新调用未完成片段，最后按序合并。  
>  
> 问答方面，RAG 的数据源是 ASR 原文，不是 AI 摘要。我把原文分块后做 embedding，语义召回走 Milvus，同时在 MySQL 中对当前视频的 chunks 做 BM25 关键词召回，再用 RRF 按两路排名融合。最终片段既进入提示词，也作为 citations 返回。当前数据规模很小，Milvus确实偏重，所以我会把它解释为可替换的向量检索实现，而不会说当前规模必须使用 Milvus。

## 4. 面试官问：这是不是一个 AI 套壳项目

### 直接口语回答

> 如果项目只是上传一段文字，然后调用一次 LLM 返回结果，那确实比较接近套壳。VidLens 里面调用 ASR、LLM 和 Embedding 本身并不是我最想强调的部分，我主要处理的是调用这些外部能力之后的工程问题。比如长音频不能整段送给 ASR，某一片失败后不能把前面已经付费完成的片段全部重跑；视频处理不能占着 HTTP 请求几分钟；上传中断后不能要求用户重新传完整文件；RAG 也不能把整个转写原文无脑塞进上下文。  
>  
> 所以我会把项目定位成 AI 视频理解后端，而不是自研模型项目。模型能力来自外部服务，我负责的是媒体预处理、任务状态、失败恢复、数据持久化、检索和引用链路。

### 防守边界

不能说自己做了：

- ASR 模型训练或推理优化；
- 大模型微调；
- 向量索引算法研发；
- 生产级 AI 平台或海量视频处理。

可以说自己做了：

- AI 服务的工程接入和输入限制处理；
- 长任务拆分、状态设计和失败恢复；
- ASR 片段持久化与复用；
- RAG 索引、混合召回、融合和引用返回；
- 针对真实故障的定位与修复。

## 5. 面试官问：是不是为了简历堆中间件

### 直接口语回答

> 这个质疑是合理的，因为以当前个人项目的数据量，Kafka 和 Milvus都不是唯一选择，甚至不是部署成本最低的选择。我不会用“高并发、海量数据”来反驳。我的判断是，异步任务机制、对象存储和检索能力分别有真实业务需求，但具体产品可以替换。比如 Kafka 可以替换成 MySQL 任务队列，Milvus 可以替换成 MySQL 存向量后由 Go 做精确余弦检索。  
>  
> 我保留这些组件的原因，是它们确实接入了完整业务链路，也让我实践了重复消费、重试状态、向量元数据过滤、索引重建和组件故障等问题。但如果目标变成给个人长期低成本使用，我会优先减少组件，而不是为了技术名词继续维持复杂部署。

### 一句话原则

```text
业务能力是否需要，与当前实现是否必须用某个中间件，是两个问题。
```

- 需要可靠异步任务，不等于必须 Kafka；
- 需要对象存储，不等于只能 MinIO；
- 需要向量召回，不等于当前数据量必须 Milvus；
- 需要断点续传，不等于 Redis 是唯一状态存储。

## 6. 四条简历分别在解决什么问题

| 简历点 | 真实问题 | 当前方案 | 主动承认的边界 |
|---|---|---|---|
| Kafka 异步任务 | HTTP 不适合等待数分钟；外部 AI 可能失败 | Kafka 调度，MySQL 保存业务状态与重试 | 当前低流量可用 MySQL 队列；单 broker 不是生产 HA |
| 分段 ASR | 长音频超限或转写不完整；重试成本高 | 默认约 300 秒切片，片段状态与内容落库 | 当前顺序调用；完成状态不代表转写质量绝对正确 |
| 分片上传 | 大文件中断后整文件重传 | MinIO 存分片，Redis 记会话与完成片号 | 是应用层续传；没有实测 GB 级与弱网成功率 |
| Milvus + BM25 + RRF | 只做语义召回会漏专有词，只做关键词不理解语义 | 两路召回后按名次融合，并返回片段 | 当前数据量小，Milvus偏重；引用不能保证模型零幻觉 |

## 7. 统一答题公式

面对任何“为什么引入 X”的问题，都按下面顺序回答：

```text
用户场景
→ 遇到的具体故障或限制
→ 不处理会造成什么后果
→ 当前实现怎么解决
→ 为什么在备选方案里这样选
→ 当前方案有什么限制
```

以 Milvus 为例，错误回答是：

> Milvus 性能高、可扩展，所以我用了 Milvus。

更好的回答是：

> 视频转写可能很长，不能每次问答都把全文塞给模型，所以需要先检索相关片段。当前向量召回由 Milvus 实现，并按 user、task 和 embedding model 过滤。不过项目现在只有很少的 chunks，Milvus 并不是规模上的刚需；若以轻量部署为第一目标，我会先用 MySQL 保存向量并在 Go 中精确计算，保留检索接口以后再替换。

## 8. 高频总览追问

### 8.1 为什么 RAG 用转写原文，而不是摘要

> 摘要已经经过一次信息压缩，细节、人名、数字和过程可能被省略。如果再基于摘要检索，用户问具体细节时根本没有原始证据。VidLens 的 RAG 索引来源是 ASR 转写，摘要主要用于快速了解整体内容。

### 8.2 为什么不把整段转写直接塞给 LLM

> 长视频转写可能超过上下文限制，即使没有超过，也会带来更高 token 成本和更多无关信息。RAG 先找与问题相关的少量片段，再把这些片段交给模型，并把相同片段作为引用返回。

### 8.3 为什么任务失败不直接全量重跑

> ASR 是多次外部调用，已经成功的片段有时间和调用成本。每片结果落库后，任务重试可以跳过 completed 且有内容的片段，只处理未完成部分。但这不是通用工作流引擎，而是针对 ASR 阶段实现的细粒度恢复。

### 8.4 Redis 挂了，视频分片是不是也丢了

> 分片内容不在 Redis，而在 MinIO。Redis 丢失会导致服务端不知道哪些片已经完成和原上传规格，当前会话恢复能力会受影响，但不是分片字节直接从 Redis 消失。生产场景还需要 Redis 持久化、状态重建或把会话元数据放到数据库。

### 8.5 引用能不能保证没有幻觉

> 不能。引用让用户能检查模型使用了哪些片段，也能限制提示词中的上下文范围，但模型仍可能误读或把结论与证据对应错。更严格的做法还需要回答后做引用一致性校验和离线评测。

### 8.6 如果要求你把项目轻量化，你先删什么

> 我会先根据部署目标替换两个偏重组件：Kafka 改成 MySQL 持久化任务队列，Milvus改成 MySQL 存 chunk 和 embedding、Go 侧精确余弦检索。MinIO 是否保留取决于是否仍需要独立对象存储；个人单机也可以先用本地文件系统抽象。业务层仍保留异步任务接口、RAGRetriever 接口和上传会话协议，避免以后扩容时重写全部业务。

## 10. 面试前一页速记

```text
【项目定位】
AI 长视频理解后端，不是自研模型，也不只是 MQ/Redis demo。

【主线】
大文件上传 → 后台长任务 → 分段 ASR → 转写原文 RAG → 引用问答。

【最重要的诚实边界】
需要异步任务，但不一定必须 Kafka。
需要向量检索，但当前规模不一定必须 Milvus。
引用提升可追溯性，但不保证零幻觉。
分片复用减少重复调用，但当前不是工作流引擎。

【回答方法】
场景 → 故障 → 后果 → 方案 → 选型 → 边界。

【不能吹】
高并发、海量数据、生产级高可用、Exactly-once、零丢失、
GB 级弱网指标、RAG 保证无幻觉、Milvus 是当前规模刚需。
```
