# VidLens 简历项目经历最终草稿

这份文档只放可以直接改进简历的项目写法。前两份文档偏“面试准备”和“问答话术”，这份只解决简历版面怎么写。

## 推荐最终版本

### 项目名称

镜知 VidLens - AI 视频内容理解平台

### 技术栈

Go、Gin、GORM、MySQL、Redis、Kafka、MinIO、Milvus、FFmpeg、JWT、Vue3、Vite

### 项目简介

镜知 VidLens 是一个面向视频内容理解场景的 AI 应用项目。用户上传本地视频或提交公开视频链接后，系统会异步完成视频下载、音频提取、ASR 转写、LLM 摘要和 RAG 索引构建，并支持基于视频转写内容的智能问答。项目重点解决视频处理中的长耗时阻塞、大文件弱网传输、重复 AI 处理、外部 API 不稳定和 RAG 问答可解释性问题。

### 项目经历

- 引入 Kafka 将视频下载、ASR 转写、摘要生成和 RAG 索引构建等长耗时任务异步化，HTTP 接口仅负责任务落库和消息投递，避免分钟级视频处理阻塞请求链路
- 设计 Redis 分布式锁 + WatchDog，结合 MD5 内容指纹和视频文件复用，保护分片合并、分析任务消费等临界区，降低并发重复入库和重复处理风险
- 采取分片上传与断点续传机制，通过 Redis Set 记录分片状态，使用 MinIO 存储分片并通过 ComposeObject 合并，提升弱网环境下大文件上传可靠性
- 基于 Redis Lua 令牌桶限流，按用户和接口维度限制高成本 AI 请求，缓解恶意请求和重复点击带来的外部 AI 服务调用成本
- 基于 RAG 检索增强生成实现视频智能问答，将转写文本切分为 chunk 并生成向量写入 Milvus，结合 BM25 关键词召回与 RRF 融合排序，返回引用片段提升答案可解释性
- 设计任务失败治理机制，基于错误分类和阶梯退避重试处理外部依赖异常，对可恢复错误延迟重投递，对不可恢复错误快速失败，避免重试风暴和 AI 成本浪费

## 五个点的取舍理由

### 1. Kafka 引入

这个点应该放第一条。它是全项目主线，能引出异步、削峰、任务状态、失败恢复、幂等、最终一致性。

不要照参考简历写 RocketMQ 阶梯延迟、死信队列。你的项目是 Kafka，业务重试主要靠 MySQL 任务状态和 DB retry scheduler。

### 2. Redis 分布式锁和限流

这两个点可以合并成一条，因为都属于 Redis 工程治理：

- 分布式锁解决并发重复处理。
- 令牌桶限流解决高成本请求保护。

如果拆成两条，简历会显得 Redis 点太多，挤压 RAG 和重试治理的位置。

### 3. 分片上传与断点续传

这个点保留。它和参考文档一致，但要写成你的实现：Redis Set 记录分片，MinIO 存分片，ComposeObject 合并，Redis 锁保护合并。

不要写“GB 级”或“弱网成功率提升多少”，除非你自己测过。

### 4. RAG

这个点必须放。相比参考文档，你的项目 RAG 不是“后续规划”，而是已经实现了 Milvus、chunk、向量检索、关键词召回、RRF 和 citations。

这里可以体现 AI 应用深度，但不要写成“训练模型”或“完整知识库平台”。当前更准确的说法是“单视频 RAG 问答”。

### 5. 工程性点

推荐选“错误分类 + DB 退避重试 + 最终一致性”。

原因：

- 它和参考简历的“指数退避重试”对应。
- 它是真实实现，不是空泛优化。
- 它能补足 Kafka 的可靠性问题：Kafka 只负责消息，业务失败要靠 DB 状态恢复。
- 它能引出可重试 / 不可重试错误分类、最大重试次数、dead 状态、任务可观测。

不建议把第五条优先写 BYOK、SSRF 或前端 AI Coding。它们可以放到面试里补充，但作为主简历 bullet，不如重试治理更贴近后端工程岗位。

## 如果要加量化指标

只有自己测过才建议写数字。

可以写：

- “接口由同步等待改为即时返回任务 ID”
- “避免分钟级视频处理阻塞请求链路”
- “按 60s / 300s / 900s 退避重投递”
- “默认 300s 音频切片”

谨慎写：

- “响应时间压缩至 50ms”
- “任务成功率提升 xx%”
- “节省 xx% Token”
- “支持 GB 级文件”

这些数字如果没有 JMeter、日志或实际测试记录，面试官追问时会比较危险。

### 需要注意的差异

参考截图里的 RocketMQ、Redisson、Function Calling 不要照搬：

- 你的 MQ 是 Kafka，不是 RocketMQ。
- 你的锁是 Go 里自实现 Redis 锁，参考了 WatchDog 思路，但不是 Redisson。
- 你的 AI 问答核心是 RAG，不是 Function Calling。
- 你的重试不是 RocketMQ 死信队列，而是 Kafka + MySQL retry scheduler。

## 最后一个工程点的替换建议

你不喜欢把 `60s / 300s / 900s` 直接写在简历里是合理的。这个数字属于实现细节，适合面试追问时讲，不一定适合放在 bullet 正文里。简历正文更建议写成“错误分类、延迟重投递、最终状态兜底”这类工程表达。

### 最推荐版本

- 设计 **任务失败治理机制**，基于错误分类和阶梯退避重试处理外部依赖异常，对可恢复错误延迟重投递，对不可恢复错误快速失败，避免重试风暴和 AI 成本浪费。

这版最适合放在简历上。它短，工程感更强，也不会把注意力拉到具体时间配置上。

### 更完整版本

- 设计 **任务失败治理机制**，结合错误分类、阶梯退避重试和最终状态兜底处理外部依赖异常；对网络抖动、接口限流、三方服务异常等可恢复错误延迟重投递，对配置缺失、密钥错误、文件不存在等不可恢复错误快速失败，避免无效重试和 AI 成本浪费。

这版适合你的简历空间比较充足时使用，表达比第一版更完整，但仍然不写具体退避时间。

### 更贴近后端可靠性的版本

- 设计 **异步任务失败恢复机制**，将外部依赖异常按可重试和不可重试分类处理，并通过 MySQL 记录失败原因、重试次数和最终状态，支撑 Kafka 消费失败后的延迟重投递和问题排查。

这版和 Kafka 第一条衔接最强。它强调 MySQL 状态表，不只是“写了一个 retry”。

### 更贴近 AI 成本控制的版本

- 设计 **AI 调用失败兜底机制**，对 ASR、LLM、Embedding、Milvus 等外部依赖异常进行分类处理，可恢复错误延迟重试，不可恢复错误快速失败，避免重复调用造成 Token 和算力浪费。

这版适合你想突出 AI 应用工程化和成本意识，但后端味道略弱于“任务失败治理机制”。

### 不建议放在简历正文的写法

- 对超时、429、5xx 等可恢复错误按 60s / 300s / 900s 重新投递。

这句话不是错，但更适合作为面试追问时的补充。简历正文如果写具体时间，面试官很容易追问“为什么是这三个时间”“是否压测过最优值”“不同错误是否应该不同退避策略”，会把讨论带到配置细节上。
