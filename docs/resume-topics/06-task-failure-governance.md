# 专题 6：任务失败治理、错误分类和阶梯退避重试

## 1. 简历原句

设计任务失败治理机制，基于错误分类和阶梯退避重试处理外部依赖异常，对可恢复错误延迟重投递，对不可恢复错误快速失败，避免重试风暴和 AI 成本浪费

## 2. 这个点想证明什么

这条证明你不是只把任务丢进 Kafka，而是考虑了外部依赖失败后的恢复策略。它能体现错误分类、退避重试、最终一致性、任务状态可观测、成本控制和可靠性设计。

## 3. 背景痛点

视频处理依赖很多外部系统：

- 远程视频站点。
- MinIO 对象存储。
- ASR 服务。
- LLM 服务。
- Embedding 服务。
- Milvus 向量库。

失败是常态。如果所有错误都立即重试，可能形成重试风暴；如果所有错误都直接失败，又会放弃很多可恢复任务。比如网络超时、429、5xx 可能过一会就恢复；但 API Key 错误、配置缺失、文件不存在，重试再多次也没用。

所以要按错误类型治理。

## 4. 主流程怎么讲

1. Consumer 执行任务阶段。
2. 如果成功，更新任务状态并推进下一阶段。
3. 如果失败，先进入错误分类。
4. 可恢复错误写入失败状态、重试次数和下次重试时间。
5. Kafka offset 可以推进，因为任务已经转入 DB 状态机。
6. Retry scheduler 扫描到期任务。
7. 到期后重新投递 Kafka。
8. 不可恢复错误直接标记失败，不再重试。
9. 超过最大重试次数后进入最终失败状态。

## 5. 具体实现怎么讲

项目里有 `TaskRetryPolicy`，默认最大重试次数是 3 次，退避时间是 60s、300s、900s。简历里不写具体时间，因为这是配置细节，面试追问时再补。

可重试错误包括：

- timeout。
- 网络异常。
- 429。
- 5xx。
- MinIO 临时异常。
- Milvus 临时异常。

不可重试错误包括：

- AI 配置缺失。
- API Key 解密失败。
- 未授权。
- 文件不存在。
- Embedding 维度不匹配。
- ASR 空结果。
- 缺少转写文本。
- 视频不可用。

失败状态写入 `video_tasks` 和 `task_jobs`，retry scheduler 到期后重新投递 Kafka。

## 6. 代码证据路径

- `internal/mq/retry.go`：错误分类、退避策略、retry scheduler。
- `internal/mq/consumer.go`：消费失败处理。
- `internal/repository/task.go`：任务失败状态。
- `internal/repository/task_job.go`：job 失败和重试状态。
- `cmd/server/main.go`：retry scheduler 启动。

## 7. 选型理由

### Q：为什么要错误分类？

答：

不是所有错误都应该重试。网络超时、429、5xx 可能恢复；配置缺失、密钥错误、文件不存在重试再多次也不会成功。错误分类能避免无效重试和 AI 成本浪费。

### Q：为什么要延迟重试？

答：

如果外部服务 429 或 5xx，立即重试可能继续失败，还会加剧外部服务压力。延迟重试给依赖恢复时间，也避免短时间内大量任务反复执行。

### Q：为什么不用 Kafka 自己一直重试？

答：

Kafka offset 重放不适合表达业务重试时间、失败原因和最大次数。用户也看不到任务卡在哪里。把失败写入 MySQL 后，任务状态可见，重试时间可控，也方便排查。

### Q：为什么不写具体 60s / 300s / 900s 到简历？

答：

具体时间是实现参数，不是核心能力。写在简历里容易被问为什么这几个值是否最优。简历写“阶梯退避重试”更稳，追问时再说当前默认配置。

## 8. 高频问题与回答话术

### Q1：哪些错误可重试？

答：

网络超时、429、5xx、MinIO 或 Milvus 临时异常通常可重试。这类问题可能是瞬时抖动，延迟后有成功机会。

### Q2：哪些错误不可重试？

答：

AI 配置缺失、API Key 解密失败、未授权、文件不存在、Embedding 维度不匹配、ASR 空结果这类错误不可重试。它们不是时间问题，而是配置、数据或请求本身不满足条件。

### Q3：失败后 commit offset 会不会丢任务？

答：

不会简单丢掉，因为失败已经写入 MySQL 的任务状态和重试计划。commit offset 只是表示 Kafka 这条消息处理告一段落，后续恢复由 DB retry scheduler 负责。

### Q4：为什么不用不 commit offset，让 Kafka 重放？

答：

不 commit 会导致同一消息快速重复消费。如果错误是 429 或外部服务故障，立即重放只会继续失败，还可能形成重试风暴。DB retry 能控制重试间隔和次数。

### Q5：什么是重试风暴？

答：

外部服务异常时，大量失败任务如果立即重试，会在短时间内再次打到同一个依赖，导致依赖更难恢复，也浪费 AI 调用成本。退避重试就是为了解决这个问题。

### Q6：超过最大重试次数怎么办？

答：

超过最大次数后进入最终失败状态，记录错误摘要。用户能看到任务失败原因，开发者也能根据任务状态排查。不能无限重试。

### Q7：如果 LLM 已经生成结果，但写 MySQL 失败怎么办？

答：

这是典型的外部调用和本地落库不一致问题。要靠任务状态和幂等兜底。重试时可能再次调用 LLM，所以高成本步骤最好在调用后尽快落库，并用任务状态判断是否已有结果，减少重复调用。

### Q8：retry scheduler 自己挂了怎么办？

答：

任务失败状态已经在 MySQL，scheduler 挂了不会丢任务，只是重试延迟。服务恢复后继续扫描到期任务即可。生产上可以做单实例调度锁或多实例抢占，避免重复调度。

### Q9：失败治理和限流有什么关系？

答：

限流是在入口减少高频请求，失败治理是在任务执行过程中处理外部依赖异常。两者都和成本控制有关：限流减少请求进入，失败治理避免无效重试和重复 AI 调用。

### Q10：怎么保证任务不丢？

答：

任务先落 MySQL，Kafka 只是推进通道。失败时也写 MySQL 状态，retry scheduler 基于 DB 恢复。这样即使 Kafka 消息处理失败，也能从业务状态中找到任务。

## 9. 结合八股怎么答

### 最终一致性

Kafka 消息、MySQL 状态、外部服务调用不可能放在一个本地事务里，所以系统追求最终一致。失败后通过状态表和补偿调度恢复。

### 幂等

重试一定可能导致重复执行，所以每个阶段要通过任务状态、唯一约束和数据替换保证幂等。

### 退避重试

退避重试不是盲目 sleep，而是按错误类型和次数控制下一次执行时间，避免重试风暴。

### 快速失败

不可恢复错误快速失败能节省成本，也能更快暴露配置问题。

## 10. tradeoff

任务失败治理增加了系统复杂度：

- 需要维护 `task_jobs`。
- 需要错误分类策略。
- 需要 retry scheduler。
- 状态从同步成功/失败变成异步最终一致。

但收益是任务可观测、失败可恢复、成本可控，不会让外部依赖异常直接拖垮请求链路。

## 11. 兜底保障

- 可恢复错误：延迟重投递。
- 不可恢复错误：快速失败。
- 超过次数：进入最终失败。
- scheduler 挂了：DB 状态保留，恢复后继续扫描。
- 重复投递：业务幂等和任务状态兜底。
- 外部 429：退避，避免继续打爆 provider。

## 12. 压力追问怎么顶

### 面试官：你为什么不直接让 Kafka 重试？

答：

Kafka 重放只能控制消息是否再次消费，不能很好表达业务错误类型、下次重试时间、最大次数和用户可见状态。我把失败放到 MySQL 状态机里，是为了让任务可观测、可控地恢复。

### 面试官：60s / 300s / 900s 是怎么来的？

答：

这是当前工程默认值，不是最优参数。它体现的是阶梯退避思路：第一次失败稍后重试，多次失败逐步拉长间隔。生产上应该根据 provider 限流策略、任务时长和监控数据调整。

### 面试官：可重试错误分类靠字符串判断靠谱吗？

答：

当前项目是工程实现，主要根据错误类型、状态码和错误信息分类。更生产化可以定义结构化错误码，让外部 provider adapter 返回统一错误类型，减少字符串匹配的不确定性。

## 13. 30 秒话术

视频处理依赖 ASR、LLM、Embedding、MinIO、Milvus 等外部服务，失败不能简单一刀切。我把错误分成可恢复和不可恢复两类。timeout、429、5xx 这类可恢复错误会写入 MySQL 重试状态，由 retry scheduler 按阶梯退避延迟重投递；配置缺失、密钥错误、文件不存在、维度不匹配这类不可恢复错误快速失败。这样能避免重试风暴和无效 AI 成本浪费。

## 14. 2 分钟话术

Kafka 解决了异步任务流转，但不等于解决了业务失败。视频处理依赖很多外部系统，比如远程下载、MinIO、ASR、LLM、Embedding 和 Milvus。外部服务失败时，如果所有错误都立即重试，可能形成重试风暴；如果所有错误都直接失败，又会放弃很多可恢复任务。

所以我做了任务失败治理。Consumer 执行任务失败后，会先做错误分类。网络超时、429、5xx、MinIO 或 Milvus 临时异常这类可恢复错误，会记录到 `task_jobs`，包括重试次数、失败原因和下一次重试时间。Retry scheduler 扫描到期 job，再重新投递 Kafka。配置缺失、API Key 解密失败、文件不存在、Embedding 维度不匹配这类不可恢复错误，则快速失败，不做无效重试。

这个设计的重点是把失败状态放到 MySQL，而不是只依赖 Kafka offset。用户和开发者都能看到任务失败在哪一步，系统也能控制重试节奏。

## 15. 5 分钟深讲版本

我会先讲 Kafka 异步化的不足。Kafka 只能解决消息流转，不能自动判断业务失败是否应该重试。视频任务依赖很多外部系统，失败场景非常多，如果不做治理，要么失败就终止，要么无限重试。

我的设计是把失败纳入任务状态机。Consumer 处理任务时，如果成功就推进下一阶段；如果失败，就进入错误分类。错误分类把异常分为可恢复和不可恢复。可恢复错误包括 timeout、429、5xx、MinIO 和 Milvus 临时异常，这些可能过一段时间恢复；不可恢复错误包括配置缺失、密钥错误、文件不存在、embedding 维度不匹配等，这些重试没有意义。

对可恢复错误，不立即重试，而是写入 `task_jobs`，记录失败原因、重试次数和下次重试时间。Retry scheduler 后台扫描到期任务，再重新投递 Kafka。这样 Kafka offset 可以 commit，因为任务已经从消息系统转入 DB 状态机，不会丢失。

对不可恢复错误，快速失败，避免浪费 AI 成本。比如 API Key 解密失败或 embedding 维度不匹配，重试多少次都不会成功，应该尽快暴露给用户或开发者。

这套机制和限流、幂等一起构成成本控制：限流减少高频请求进入，失败治理避免无效重试，幂等降低重复执行影响。

## 16. 不要这么说

- 不要说用了 RocketMQ 死信队列。
- 不要说 Kafka 原生支持延迟重试。
- 不要说所有失败都会重试。
- 不要说 60s / 300s / 900s 是最优参数。
- 不要说失败后 commit offset 就等于任务成功。
