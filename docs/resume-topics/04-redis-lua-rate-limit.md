# 专题 4：Redis Lua 令牌桶限流

## 1. 简历原句

基于 Redis Lua 令牌桶限流，按用户和接口维度限制高成本 AI 请求，缓解恶意请求和重复点击带来的外部 AI 服务调用成本

## 2. 这个点想证明什么

这条证明你有 AI 应用的成本意识和服务保护意识。AI 请求和普通 CRUD 不一样，ASR、LLM、Embedding 都可能产生外部调用成本，也可能触发 provider 的 429。限流是入口保护，不是为了堆 Redis 技术点。

## 3. 背景痛点

AI 应用里不同接口成本差异很大：

- 普通查询接口成本低。
- ASR 会消耗语音识别服务。
- 摘要和问答会消耗 LLM 调用。
- RAG 索引构建会消耗 Embedding 和 Milvus 写入。

如果用户恶意刷新、重复点击或者脚本调用，高成本接口会产生真实费用，也可能打爆外部 provider 的限额。

所以需要在入口按用户和接口维度做限流。

## 4. 主流程怎么讲

1. 请求进入 Gin middleware。
2. 先识别用户，如果已登录用 userID，未登录或识别失败用 IP。
3. 结合 route 生成限流 key。
4. Redis Hash 保存当前 token 数和上次刷新时间。
5. Lua 脚本在 Redis 内部计算补充 token、判断是否足够、扣减 token、写回状态。
6. 如果 token 足够，请求放行。
7. 如果 token 不足，返回限流错误。
8. Redis 异常时当前策略是 fail-open，优先保证主业务可用。

## 5. 具体实现怎么讲

项目使用 Redis Lua 实现令牌桶，原因是令牌补充和扣减涉及多步读写。

如果不用 Lua，而是拆成多条命令：

1. 读取 token。
2. 计算补充。
3. 判断 token 是否足够。
4. 扣减 token。
5. 写回 Redis。

并发请求下可能同时读取到相同 token，导致超发。Lua 脚本在 Redis 单线程执行模型下具有原子性，能保证同一个 key 的限流状态一致。

## 6. 代码证据路径

- `internal/middleware/ratelimit.go`：Redis Lua 令牌桶实现。
- `internal/config/config.go`：限流配置。
- `cmd/server/main.go`：middleware 注册。

## 7. 选型理由

### Q：为什么用令牌桶？

答：

令牌桶能限制平均速率，同时允许一定突发。用户偶尔连续点击几次可以接受，但长期高频请求要被限制。固定窗口有边界突刺问题，漏桶对突发不友好，所以这里用令牌桶更适合。

### Q：为什么用 Redis？

答：

限流要跨实例共享状态，本地内存限流只能限制单个进程。Redis 是项目已有基础设施，适合存这种高频、短生命周期的计数状态。

### Q：为什么用 Lua？

答：

令牌桶需要读、算、判断、扣减、写回，这些步骤必须原子执行。Lua 可以把多步逻辑放在 Redis 内部一次执行，避免并发超发。

### Q：为什么按用户和接口维度？

答：

只按 IP 会误伤同一网络下多个用户，只按用户又无法覆盖未登录请求。按 route + userID 可以限制某个用户对某类接口的频率，不同接口可以有不同成本策略。

## 8. 高频问题与回答话术

### Q1：令牌桶核心字段是什么？

答：

核心是当前 token 数、上次刷新时间、桶容量、补充速率和本次请求消耗 token 数。项目里用 Redis Hash 保存 token 和 last time。

### Q2：固定窗口有什么问题？

答：

固定窗口在边界处会突刺。比如每分钟 100 次，用户可以在上一分钟最后一秒打 100 次，下一分钟第一秒再打 100 次，短时间内实际达到 200 次。令牌桶能更平滑。

### Q3：滑动窗口为什么不用？

答：

滑动窗口更精确，但需要记录更多请求时间戳或分桶计数，实现和存储成本更高。这里主要是限制高成本 AI 请求的频率，令牌桶已经能满足。

### Q4：限流和 Kafka 削峰有什么区别？

答：

限流发生在入口，决定请求能不能进入系统；Kafka 削峰发生在请求进入系统之后，让后台按消费能力慢慢处理。限流解决“要不要进来”，Kafka 解决“进来后怎么处理”。

### Q5：Redis 挂了怎么办？

答：

当前策略是 fail-open，也就是 Redis 异常时放行请求。原因是限流是保护层，不能因为 Redis 短暂异常导致核心业务全部不可用。生产上可以对极高成本接口改成 fail-close 或降级策略。

### Q6：fail-open 会不会被攻击打穿？

答：

有这个风险，所以这是可用性和保护性的权衡。当前项目优先保证主流程可用。生产上可以按接口成本分级：普通接口 fail-open，高成本 AI 接口在 Redis 不可用时 fail-close 或进入降级队列。

### Q7：限流能不能替代鉴权？

答：

不能。鉴权解决“你是谁、能不能访问”，限流解决“访问频率是否过高”。两者是不同层次。

### Q8：限流能不能完全控制 AI 成本？

答：

不能。限流只是入口频率保护。完整成本控制还需要用户额度、每日用量统计、模型调用审计、任务失败快速终止等机制。项目里限流和失败治理一起降低成本风险。

### Q9：重复点击怎么处理？

答：

重复点击会命中同一用户和接口维度的限流 key。如果短时间超过令牌桶容量，就会被限流，避免反复触发高成本 AI 请求。

### Q10：Lua 会不会阻塞 Redis？

答：

Lua 脚本会在 Redis 内原子执行，所以脚本必须短小。当前令牌桶脚本只做简单数值计算和 Hash 读写，不做长循环，风险可控。

## 9. 结合八股怎么答

### 固定窗口、滑动窗口、漏桶、令牌桶

固定窗口简单但有边界突刺；滑动窗口精确但成本更高；漏桶输出稳定但不适合突发；令牌桶限制平均速率并允许突发。

### Redis Lua 原子性

Redis 执行 Lua 脚本时不会被其他命令打断，适合把多步读写封装为原子操作。

### 限流维度

按 IP 简单但可能误伤；按 userID 更准确；按 route 区分接口成本；组合维度更适合 AI 高成本接口。

## 10. tradeoff

限流也有代价：

- 参数不好会误伤正常用户。
- Redis 异常时要权衡 fail-open 和 fail-close。
- 只做入口频率限制，不能替代完整计费。
- 分布式限流依赖 Redis 性能和可用性。

## 11. 兜底保障

- 并发超发：Lua 原子扣减。
- 重复点击：同用户同接口维度限流。
- 未登录用户：按 IP 限流。
- Redis 异常：当前 fail-open，生产可对高成本接口 fail-close。
- 外部 provider 429：失败治理里可重试分类和退避重投递。

## 12. 压力追问怎么顶

### 面试官：你这不就是简单计数器？

答：

不是。简单计数器通常是固定窗口，会有边界突刺。令牌桶有容量和补充速率，能限制平均速率，同时允许合理突发。项目里还用 Lua 保证补充和扣减原子执行。

### 面试官：Redis 挂了你还放行，那限流有什么用？

答：

这是可用性和保护性的权衡。当前项目选择 fail-open，是避免 Redis 短暂异常导致主业务全挂。生产环境可以按接口成本做差异化，高成本 AI 接口在 Redis 不可用时 fail-close 或降级。

### 面试官：限流为什么写到简历？

答：

因为 AI 应用和普通 CRUD 不一样，请求会产生外部服务调用成本。限流体现的是成本控制和服务保护，不只是 Redis 技术点。

## 13. 30 秒话术

AI 请求会产生外部服务调用成本，所以我在入口做了 Redis Lua 令牌桶限流。限流 key 按 route + userID 生成，未登录时按 IP。Redis Hash 保存 token 和刷新时间，Lua 脚本原子完成 token 补充、判断和扣减。这样可以限制恶意请求和重复点击，同时保留合理突发能力。当前 Redis 异常时 fail-open，优先保证主流程可用。

## 14. 2 分钟话术

这个点主要是 AI 成本控制。ASR、摘要、Embedding 和问答都可能调用外部 AI 服务，如果用户重复点击或脚本刷接口，会产生真实成本，也可能触发外部 provider 的 429。

我用 Redis 实现分布式令牌桶，限流维度是 route + userID，未登录时退化为 IP。令牌桶的好处是既限制平均速率，也允许一定突发，比固定窗口更平滑。

实现上用 Redis Hash 保存当前 token 和上次刷新时间，用 Lua 脚本一次性完成 token 补充、判断是否足够、扣减和写回。因为这些步骤如果拆成多条 Redis 命令，并发下可能超发。Lua 保证同一个 key 的限流状态原子更新。

这不是完整计费系统，只是入口保护。完整成本控制还要结合调用审计、用户额度和任务失败治理。

## 15. 5 分钟深讲版本

我会从 AI 请求成本讲起。普通接口即使重复点几次，成本主要是数据库和服务器资源。但 ASR、LLM、Embedding 请求会调用外部服务，重复点击可能造成真实费用，也可能被 provider 限流。

所以我在入口加了限流 middleware。每个请求进入时，根据路由和用户 ID 生成限流 key。如果没有用户 ID，就按 IP 限流。不同路由可以配置不同限流参数，因为上传、问答、普通查询的成本不一样。

算法上选择令牌桶。固定窗口实现简单但有边界突刺；漏桶输出太平滑，不适合用户正常突发；令牌桶可以限制平均速率，同时允许短时间合理突发。

实现上，令牌桶需要根据当前时间补充 token，再判断本次请求是否有 token 可用，如果有就扣减并放行。这是多步读写，如果用多条 Redis 命令，在并发下会超发。所以我用 Lua 脚本在 Redis 内部原子执行。Redis Hash 保存 token 和 last time。

最后补边界：当前 Redis 异常时 fail-open，是为了避免保护层故障导致主业务不可用。生产上对极高成本接口可以改成 fail-close 或降级。限流不能完全控制成本，还需要用户额度、调用审计和失败快速终止。

## 16. 不要这么说

- 不要说实现了完整计费系统。
- 不要说 Redis 挂了还能严格限流。
- 不要说限流能完全控制 AI 成本。
- 不要把限流和 Kafka 削峰混为一谈。
- 不要说 Lua 适合执行复杂长逻辑。
