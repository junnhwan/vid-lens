# 专题 3：分片上传与断点续传

## 1. 简历原句

采取分片上传与断点续传机制，通过 Redis Set 记录分片状态，使用 MinIO 存储分片并通过 ComposeObject 合并，提升弱网环境下大文件上传可靠性

## 2. 这个点想证明什么

这条证明你处理过大文件上传场景，不只是普通 multipart 表单上传。它能体现文件工程、Redis 数据结构、对象存储、断点续传、服务端合并和并发合并控制。

## 3. 背景痛点

视频文件通常比较大，直接整文件上传会有几个问题：

- 上传时间长，弱网容易中断。
- 中断后如果重传整个文件，用户体验差。
- 大请求长时间占用连接和服务端资源。
- 重复点击合并可能造成重复对象和重复入库。

所以要把大文件拆成多个分片，每片独立上传，失败后只补传缺失分片。

## 4. 主流程怎么讲

1. 前端按固定大小切分文件。
2. 每个分片上传时带 `file_md5`、分片编号和总分片数。
3. 后端把分片写入 MinIO 临时对象路径。
4. 写入成功后，把分片编号加入 Redis Set。
5. 前端恢复上传时查询 Redis Set，拿到已上传分片列表。
6. 前端只补传缺失分片。
7. 所有分片到齐后，后端校验完整性。
8. 后端通过 MinIO `ComposeObject` 合并成最终视频对象。
9. 合并后创建或复用 `video_assets`，再创建 `video_tasks`。

## 5. 具体实现怎么讲

Redis Set 记录的是某个文件 MD5 下已经上传成功的分片编号。Set 的特点是天然去重，同一个分片重复上传不会产生重复状态。

MinIO 存储两类对象：

- 临时分片对象：例如 `chunks/<file_md5>/<chunk_number>`。
- 最终视频对象：例如 `videos/<uuid>.<ext>`。

合并时不把所有分片读回 Go 应用内存，而是调用 MinIO 的 `ComposeObject` 在对象存储侧合并。这样能减少应用内存压力。

合并阶段是临界区，所以同一个 MD5 会用 Redis 锁保护，避免重复合并和重复创建 asset。

## 6. 代码证据路径

- `internal/handler/media.go`：分片上传、查询分片、合并入口。
- `internal/service/media.go`：分片写入、Redis Set 状态、合并和 asset 创建。
- `internal/storage/minio.go`：MinIO 上传、删除和 `ComposeObject`。
- `internal/pkg/lock/redis_lock.go`：合并阶段 Redis 锁。

## 7. 选型理由

### Q：为什么用 Redis Set 记录分片状态？

答：

分片状态本质是一个“已完成编号集合”。Set 天然去重，适合判断某个分片是否已经上传，也方便返回全部已上传编号。分片状态是临时高频数据，放 Redis 比放 MySQL 更轻。

### Q：为什么用 MinIO？

答：

视频和分片都属于对象数据，不适合放本地磁盘。本地磁盘和单机绑定，不利于扩容和容灾。MinIO 提供 S3 兼容对象存储能力，适合存视频文件，也方便后续切换云对象存储。

### Q：为什么用 ComposeObject？

答：

ComposeObject 在对象存储侧合并分片，后端不需要把所有分片下载到应用内存再拼接。对大文件来说，这能降低内存占用和 IO 压力。

### Q：为什么不直接用数据库存分片？

答：

数据库适合存元数据，不适合存视频二进制分片。分片对象放 MinIO，状态放 Redis，任务信息放 MySQL，职责更清楚。

## 8. 高频问题与回答话术

### Q1：断点续传怎么知道哪些分片不用传？

答：

后端用 Redis Set 记录某个 fileMD5 下已上传成功的分片编号。前端恢复上传时先查这个集合，然后只上传缺失分片。

### Q2：重复上传同一个分片怎么办？

答：

Redis Set 天然去重，同一个编号重复加入不会造成重复状态。MinIO 临时对象也可以被覆盖。最终以分片编号集合和合并前完整性校验为准。

### Q3：合并前怎么校验完整性？

答：

根据 `total_chunks` 从 0 到 n-1 检查 Redis Set 中是否都存在对应编号。只要缺一个分片，就不执行 ComposeObject，而是返回缺失分片信息或合并失败。

### Q4：分片上传到 99% 断了怎么办？

答：

用户重新上传时，前端根据 fileMD5 查询已上传分片集合，只补传最后缺失的分片，不需要重传前面 99%。这就是断点续传的核心。

### Q5：ComposeObject 失败怎么办？

答：

合并失败不会创建最终 asset，前端可以稍后重试合并。因为分片还在 MinIO 临时路径，Redis Set 也记录了已上传状态，只要分片没有清理，就不需要重新上传全部。

### Q6：合并被重复点击怎么办？

答：

合并阶段用 Redis 锁保护同一 fileMD5。第一个请求拿到锁后执行合并，其他请求拿不到锁时返回“合并进行中”或稍后重试，避免重复 ComposeObject 和重复创建 asset。

### Q7：Redis 状态丢了怎么办？

答：

当前实现依赖 Redis Set 做断点状态。如果 Redis 状态丢失，最直接影响是前端不知道哪些分片已传，可能需要重新上传或补偿扫描 MinIO 临时对象。生产上可以给分片状态加过期和重建策略，或者把关键上传会话落 MySQL。

### Q8：临时分片怎么清理？

答：

这属于生产化补充。可以按 fileMD5 和上传时间设置清理任务，清掉长期未合并的临时分片，同时删除 Redis Set。面试时要说当前主线实现了上传、状态记录和合并，垃圾清理可作为后续优化。

### Q9：为什么不让前端直传 MinIO？

答：

前端直传可以减少后端带宽压力，但需要预签名 URL、权限控制、回调校验和对象名安全设计。当前项目先由后端接收分片，逻辑更集中，适合项目阶段。后续大规模部署可以考虑预签名直传。

### Q10：分片大小怎么定？

答：

分片太小会导致请求数太多，Redis 和 MinIO 元数据压力变大；分片太大又降低弱网恢复粒度。一般要结合网络环境、文件大小和服务端限制调整。面试里不要说一个固定值最优。

## 9. 结合八股怎么答

### Redis Set

Set 适合去重集合，判断分片是否已上传可以用 `SISMEMBER`，返回已上传列表可以用 `SMEMBERS`。

### 对象存储

MinIO 存的是对象，不是关系数据。对象存储适合大文件，MySQL 存元数据。

### 幂等

重复上传同一分片、重复合并请求都要按幂等考虑。Set 去重和合并锁是两个层面的幂等设计。

### 大文件上传

核心不是“能上传大文件”，而是失败恢复、状态记录、完整性校验和服务端资源控制。

## 10. tradeoff

分片上传比普通上传复杂：

- 前端需要切片、记录进度、查询已上传分片。
- 后端需要维护临时分片和 Redis 状态。
- 需要处理临时对象清理。
- 需要处理合并并发。

但换来的收益是弱网失败后不用重传整个视频，大文件上传体验更可靠。

## 11. 兜底保障

- 分片重复上传：Redis Set 去重。
- 分片缺失：合并前完整性校验。
- 合并并发：Redis 锁保护。
- ComposeObject 失败：不创建 asset，可重试合并。
- 已有相同 MD5 asset：直接复用视频文件。

## 12. 压力追问怎么顶

### 面试官：你这个是不是只是普通上传？

答：

不是。普通上传是一次请求传完整文件，我这里是分片独立上传，Redis Set 记录已完成分片，断点续传时只补缺失分片，最后由 MinIO ComposeObject 合并。核心是断点状态和服务端合并。

### 面试官：Redis 丢了不就断点续传失效？

答：

是的，Redis 状态丢失会影响续传体验，所以它不是唯一可靠存储。当前项目里分片状态是临时数据，丢失后可以重新上传或后续通过 MinIO 临时对象扫描补偿。生产上可以把上传会话元数据落 MySQL。

### 面试官：有没有测过 GB 级？

答：

如果没有实际压测，我不会说支持 GB 级。可以说这个设计面向大文件上传场景，使用分片和对象存储合并降低弱网重传成本，但具体容量要结合服务配置和压测结果确认。

## 13. 30 秒话术

视频文件大，弱网中断后如果重传整个文件体验很差，所以我做了分片上传和断点续传。每个分片先写入 MinIO 临时路径，成功后把分片编号写入 Redis Set。恢复上传时前端查询已上传分片，只补传缺失部分。全部到齐后，后端校验完整性，并用 MinIO ComposeObject 在对象存储侧合并，合并阶段用 Redis 锁避免重复合并。

## 14. 2 分钟话术

这个点解决的是大视频上传可靠性。直接整文件上传在弱网环境下很容易中断，一旦失败就要重传全部文件。所以我让前端按分片上传，每片都带 fileMD5 和分片编号。

后端收到分片后先写 MinIO 临时对象，然后把分片编号加入 Redis Set。Redis Set 记录的是已上传编号集合，天然去重，前端恢复上传时可以查询这个集合，只补传缺失分片。等所有分片到齐后，后端先校验 Redis Set 中是否包含 0 到 totalChunks-1 的所有编号，再调用 MinIO ComposeObject 合并成最终视频对象。

合并是临界区，所以同一个 fileMD5 会加 Redis 锁，避免重复合并和重复创建 asset。合并后再结合 MD5 查询或创建 `video_assets`，让相同视频可以复用真实文件。

## 15. 5 分钟深讲版本

我会先讲普通上传的问题。视频文件大，上传时间长，弱网中断概率高。如果一个 500MB 视频上传到 99% 失败，用户还要从头传，这个体验很差。所以我做了服务端分片续传。

流程上，前端把文件切片，每片上传到后端。后端把分片写到 MinIO 临时路径，写成功后用 Redis Set 记录分片编号。这里用 Set 是因为分片编号天然是集合，重复上传同一片也只保留一份状态。

恢复上传时，前端先根据 fileMD5 查询已上传分片，只上传缺失编号。这样网络中断后可以继续，而不是从头开始。

合并时，后端不会盲目合并，而是先根据 totalChunks 检查所有分片是否都存在。确认完整后调用 MinIO ComposeObject，让对象存储完成合并。这样 Go 服务不需要把所有分片读回内存再拼接，避免大文件占用内存。

并发上，同一个文件可能重复点击合并，所以我用 Redis 锁保护合并流程。合并完成后创建最终视频对象和 asset。如果发现同 MD5 的 asset 已存在，就直接复用。这个点和第二条的 MD5 复用是衔接的。
