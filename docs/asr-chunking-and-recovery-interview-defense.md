# VidLens 长视频分段 ASR：片段持久化、结果复用与面试防守

> 对应简历：**设计长视频分段 ASR 与结果持久化机制，任务异常后仅重跑失败片段并复用已完成结果，避免整段视频重复处理。**  
> 核心原则：这项设计来自真实的长音频输入限制和转写不完整问题；不要把它包装成大规模并行 ASR 或通用分布式工作流。

---

## 1. 先记住最重要的结论

不要只说：

> 长视频太大，所以我做了分片。

更准确的表达是：

> 长音频整段调用 ASR 既可能超过接口体积或时长限制，也可能出现接口返回成功但内容明显不完整。VidLens 因此先把音频按固定时长切片，再逐片调用 ASR，并把每片状态和文本单独持久化。任务重试时，会复用 completed 且有内容的片段，只重新处理未完成片段，最后按片号合并全文。

这里解决了两个不同问题：

```text
按时长切片          → 控制单次 ASR 输入，降低超限和不完整风险
片段结果持久化与复用 → 任务重试时减少重复调用和重复等待
```

只切片但不落库，进程重启后仍要从第一片开始；只落一个最终全文，也无法知道哪片已经完成。

---

## 2. 真实故障背景

项目中实际出现过一段约 15 分钟的视频：

1. 第一次调用失败，错误明确指出 MiMo ASR 的音频 base64 超过 10MB；
2. 初步修复只根据文件大小判断是否切片；
3. 再次执行后任务状态变成完成，但转写结果只有 711 个字符，和视频时长明显不匹配；
4. 最终统一改为按音频时长切片，而不是只在文件大小超过阈值时切片。

这个故障说明：

```text
HTTP 200 / 任务 completed ≠ 转写质量一定正确
文件体积未超限         ≠ 音频时长适合整段识别
```

不同编码、码率和静音比例会导致相同时长的文件大小差异很大，所以“只看文件大小”不能稳定代表模型实际处理压力。

证据记录：`docs/troubleshooting-and-interview-notes.md` 的记录 001、002 和 012。

## 3. 30 秒回答

> VidLens 的长视频不能直接把整段音频送给 ASR。我实际遇到过 base64 超过接口 10MB 限制，也遇到过接口返回成功但 15 分钟视频只转出 711 个字符。所以现在音频会统一按时长切成约 300 秒一段，每段调用 ASR 前后都在 MySQL 记录 running、completed 或 failed，并保存片段文本。任务异常后重新执行时，会按 task_id 和 chunk_index 查询，completed 且内容非空的片段直接复用，只调用没有完成的片段，最后按顺序合并全文。

---

## 4. 1 分钟回答

> 这项设计不是为了单纯展示音频切片，而是解决长音频调用外部 ASR 的输入限制、完整性和恢复成本。最开始整段调用时，15 分钟视频因为 base64 超过 10MB 直接失败；后来只按文件大小判断是否切片，任务虽然完成，但只有 711 个字符。我发现压缩后的文件大小并不能可靠代表音频时长，所以改成统一按时长切片，当前默认每段约 300 秒。  
>  
> 另外，只切片还不够。如果第五片失败，整个任务重试后又从第一片调用，前四片的时间和 API 成本都会浪费。因此我建立了 `video_transcription_chunks` 表，用 `task_id + chunk_index` 做唯一约束，保存每片状态、内容、字符数、错误和重试次数。consumer 每处理一片先检查是否已有 completed 且非空的结果，有就复用；失败就记录 failed，任务重试后从未完成位置继续。当前是顺序处理，不是并行转写，也还需要更完善的质量校验。

## 5. 实际处理流程

```text
Kafka consumer 获取 transcribe 任务和 processing lease
  → 从 MinIO 下载视频到临时文件
  → 提取音频
  → 按 DefaultAudioSegmentSeconds 切成多个临时音频片段
  → 按 chunk_index 顺序处理
      → 查询 task_id + chunk_index
      → 若 completed 且 content 非空：直接复用
      → 否则先 UpsertRunning
      → 调用外部 ASR
      → 成功：UpsertCompleted，保存 content/chars
      → 失败：UpsertFailed，保存 error_msg 并增加 retry_count
  → 按片段顺序使用空行拼接全文
  → 保存 video_transcriptions
  → 触发后续标题/RAG 索引等流程
```

当前循环在 `internal/mq/consumer.go` 的 `transcribeAudio` 中顺序执行。任何一片失败会结束本次任务，由上层任务失败恢复机制决定何时重试；下次执行时，前面已经持久化成功的片段会被跳过。

## 6. 为什么要按时长，而不是按文件大小切片

### 面试官问题

> ASR 有 10MB 限制，控制文件大小不就可以了吗？为什么还要按时长？

### 直接口语回答

> 文件大小是输入限制的一部分，但不能完整代表 ASR 风险。同样 5MB 的音频，可能因为编码和码率不同，对应完全不同的时长；一个低码率长音频虽然没超过体积限制，仍可能让模型处理不完整。项目里就出现过文件大小判断通过，但 15 分钟视频只得到 711 个字符的情况。所以我统一按时长切片，先限制每次请求的语音跨度；同时音频预处理会统一格式，避免输入差异过大。

### 为什么不是把 10MB 阈值直接调大

> 10MB 是外部接口或当前调用方式的限制，不是本地配置想改就能改。即使服务端允许更大请求，base64 还会放大体积，长音频也可能有模型时长限制或识别完整性问题。把阈值调大只是延后失败，不解决恢复成本。

---

## 7. 为什么默认约 300 秒一片

### 直接口语回答

> 300 秒是当前项目的保守工程参数，不是我验证出的通用最优值。片段太长，仍可能撞输入限制或出现识别不完整；片段太短，会增加 API 请求次数、固定开销和片段边界问题。现在用 5 分钟是为了先让单次输入可控，后续应该根据具体 provider 的体积、时长限制、音频码率和评测数据动态配置。

### 不要说

- “300 秒是行业最佳实践”；
- “5 分钟一定准确率最高”；
- “经过大规模压测得到 300 秒最优”。

当前代码中的默认值证据：`internal/pkg/ffmpeg/ffmpeg.go` 的 `DefaultAudioSegmentSeconds`，调用位置在 `internal/mq/consumer.go`。

## 8. 片段表为什么这样设计

`video_transcription_chunks` 当前主要字段：

| 字段 | 用途 |
|---|---|
| `task_id` | 归属哪个视频任务 |
| `chunk_index` | 片段顺序，也是恢复和合并依据 |
| `status` | pending/running/completed/failed |
| `content` | ASR 片段文本 |
| `chars` | 文本字符数，便于观察异常短结果 |
| `error_msg` | 当前失败原因 |
| `retry_count` | 此片段累计失败次数 |
| `audio_object` | 当前代码记录临时音频路径/标识 |
| `start_second/end_second` | 为时间范围保留的字段，当前写入使用仍有限 |

`task_id + chunk_index` 有唯一索引，作用不是加快所有查询，而是防止同一任务同一片号出现多条互相矛盾的记录，并给 upsert 提供物理约束。

### 面试官追问：为什么不把片段 JSON 放到任务表

> 任务表主要用于列表、状态和调度查询，不适合不断写入大量片段文本和 JSON。单独建表后，可以按 task 和 chunk 定位、更新单片状态，也能让最终转写表只保存合并结果。除此之外，关系表的唯一索引比在一个 JSON 数组里做并发更新和幂等控制更直接。

## 9. “仅重跑失败片段”到底是怎么成立的

### 直接口语回答

> consumer 每次重新切出相同顺序的音频片段，然后按 `task_id + chunk_index` 查数据库。如果记录状态是 completed，并且 content 去空格后非空，就把数据库里的文本加入结果，不再调用 ASR；否则把该片标成 running 后调用 ASR。调用失败会写 failed 和错误信息，成功则保存文本。因为前面成功片段已经落库，整个任务下一次重试时会自动跳过它们。

这里“失败片段”的精确含义应是：

- 状态不是 completed；或
- completed 但没有可复用文本；或
- 尚不存在片段记录。

### 当前限制

当前复用依赖“重新切片后 chunk_index 的语义保持一致”。如果以后更换切片时长、音频预处理方式或 provider，却仍复用旧片段，可能出现边界错位。因此更严格的设计应把切片策略版本、起止时间或内容指纹纳入复用条件。

## 10. 如果进程在不同位置崩溃，会发生什么

### 场景一：ASR 调用前已经标记 running，调用过程中进程崩溃

数据库会留下 running。上层任务租约和重试机制恢复任务后，这片不是 completed，因此会再次调用。可能发生外部 ASR 已执行、但结果未保存的重复调用，这是当前 at-least-once 语义的一部分。

### 场景二：ASR 成功，片段结果已经落库，最终全文还没保存时崩溃

下次任务执行会复用已落库片段，重新合并并保存最终全文，不需要重新调用这些片段。

### 场景三：片段文本已全部完成，但后续 RAG 索引失败

ASR 转写不应该被视为丢失。当前 RAG 已拆成独立 job 和状态，索引失败可以单独恢复，不应强迫 ASR 全量重跑。

### 场景四：同一片消息重复消费

任务层有 processing lease，片段层有 `task_id + chunk_index` 唯一约束和 completed 复用。它们降低重复副作用，但不能宣称严格 exactly-once；在外部 ASR 成功而本地落库前崩溃的窗口，仍可能重复调用。

## 11. 分片会不会丢上下文

### 直接口语回答

> 会有这个风险。如果一句话正好跨越切片边界，ASR 可能在边界处丢字或断句。当前版本优先解决长音频无法稳定完成和失败后全量重跑的问题，还没有实现音频重叠窗口与文本去重，所以不能声称切片完全不影响识别质量。后续可以给相邻片保留几秒重叠，再通过时间戳或文本相似度消除重复。

### 可行改进

1. 每段保留 5～10 秒重叠；
2. 保存真实的 `start_second/end_second`；
3. 如果 provider 返回词级时间戳，按时间对齐去重；
4. 没有时间戳时，对相邻片首尾做最长公共片段或相似度去重；
5. 对边界错词建立专门评测样例。

这些目前不能描述为已经完成。

---

## 12. 为什么当前没有并行调用所有片段

### 面试官问题

> 都切片了，为什么不 goroutine 并发转写，速度不是更快吗？

### 直接口语回答

> 并行确实可能缩短单视频耗时，但外部 ASR 通常有 QPS、并发数和计费约束。个人项目当前任务量很小，直接把一个长视频的所有片段并发打出去，容易触发 429，也会让失败重试和限流更复杂。所以目前是顺序处理，先保证结果可恢复。后续如果有明确限额，可以做有界并发，例如 worker pool 或 semaphore，而不是为每片无限开 goroutine。

### 不要声称

- 多分片并行 ASR；
- 自动根据负载动态扩缩；
- 大规模 GPU 推理调度。

这些都不是当前实现。

## 13. “完成”如何判断，质量校验够不够

当前片段成功的主要判定是：

- ASR 调用没有返回错误；
- 文本去空格后非空；
- 成功后保存字符数并记录日志/指标。

这比只看 HTTP 状态好，但仍不足以证明质量正确。真实故障已经表明，外部服务可能“成功但明显过短”。

### 面试中的诚实回答

> 当前已经记录每片字符数和最终字符数，可以帮助定位空结果或明显过短，但还没有建立基于音频时长、语速区间、语言和静音比例的自动质量阈值。因此 completed 主要表示技术调用完成，不代表语义质量已经经过严格验收。

### 后续改进

- 每分钟字符数异常检测，但要考虑静音、音乐和不同语言；
- 随机抽样与人工标注；
- WER/CER 评测，需要有参考转写；
- provider 结果长度、耗时和失败类型指标；
- 片段级重新转写入口，而不是只能重跑整个 task。

---

## 14. ASR 片段和 RAG chunk 是不是一回事

### 直接口语回答

> 不是。ASR 片段是为了控制音频调用大小和实现转写恢复，边界按时间；RAG chunk 是在转写文本上为了检索和模型上下文重新分块，边界按文本长度和句子结构。一个 300 秒 ASR 片段可能会再拆成多个 RAG chunk。把两者混为一谈会让检索粒度受音频调用限制，不合理。

```text
音频切片：面向 ASR provider 限制与失败恢复
文本分块：面向检索粒度、语义完整性与上下文预算
```

## 16. 当前限制与未来改进

### 当前限制

1. 片段顺序调用，没有有界并发；
2. 300 秒是经验参数，不是评测得出的全局最优；
3. 分片边界没有音频重叠和文本去重；
4. `start_second/end_second` 字段尚未形成完整时间引用能力；
5. completed 主要表示调用成功，质量检测仍有限；
6. 切片策略变化时，旧片段复用缺少版本校验；
7. 外部 ASR 成功、本地落库前崩溃仍可能导致重复调用。

### 合理未来改进

1. 按 provider 限制和编码信息动态计算片长；
2. 有界并发并结合 provider 级限流；
3. 音频重叠窗口与边界文本去重；
4. 记录切片策略版本、起止时间和音频指纹；
5. 建立片段级质量规则和 CER/WER 评估集；
6. 支持指定失败片段单独重试；
7. 将引用片段进一步映射回视频时间轴。

## 17. 简历表述校准

当前写法基本成立：

> 设计长视频分段 ASR 与结果持久化机制，任务异常后仅重跑失败片段并复用已完成结果，避免整段视频重复处理。

更严谨版本：

> 将长音频按时长切片并持久化片段状态与转写文本；任务重试时跳过已完成片段，仅处理未完成部分，减少重复 ASR 调用。

不推荐：

> 自研分布式 ASR 工作流，通过并行计算大幅提升识别准确率和吞吐量。

当前既没有自研 ASR，也不是并行工作流，更没有足够数据证明准确率和吞吐提升。

## 18. 一页速记

```text
【为什么切片】
整段音频可能体积超限、时长过长或成功但转写不完整。
真实故障：约 15 分钟视频，先遇到 base64 10MB 限制，后只转出 711 字符。

【为什么按时长】
文件大小受编码/码率影响，不能可靠代表音频跨度。

【为什么落片段表】
只保存最终全文，失败后不知道哪些片已完成。
task_id + chunk_index 唯一；保存 status/content/chars/error/retry_count。

【怎么复用】
completed 且 content 非空 → 跳过 ASR，直接加入 parts；
其他状态 → 重新调用；最后按 index 合并。

【当前实现】
默认约 300 秒；顺序调用；失败片落库后结束本次任务。

【不能吹】
不是并行 ASR，不是通用工作流，不保证 exactly-once，
completed 不等于质量绝对正确，切片边界仍可能丢上下文。
```
