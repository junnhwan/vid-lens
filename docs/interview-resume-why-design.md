## 1. 简历原文（以你当前投递版为准）

**项目名称建议：** 镜知 VidLens — AI 视频内容理解与智能问答平台  

**技术栈（按需写）：** Go、Gin、GORM、MySQL、Redis、Kafka、MinIO、Milvus、FFmpeg、yt-dlp、Vue3  

**项目简介：**

> 一个 AI 驱动的长视频内容理解与智能问答平台，支持大文件分片上传、长视频分段转写、AI 摘要和带引用的视频问答，并通过异步任务调度与失败重试处理耗时且易受外部服务影响的视频处理链路。

**项目经历（4 条）：**

1. 引入 Kafka 异步调度 ASR、摘要生成和 RAG 索引等长耗时任务，结合任务状态与阶梯退避重试，实现阶段追踪和异常恢复  
2. 设计长视频分段 ASR 与结果持久化机制，任务异常后仅重跑失败片段并复用已完成结果，避免整段视频重复处理  
3. 采用分片上传与断点续传机制，通过 Redis 记录分片状态并使用 MinIO 服务端合并，避免上传中断后整文件重传  
4. 基于 Milvus + BM25 + RRF 实现语义与关键词混合检索，并返回引用片段，提升视频问答结果的可追溯性  

### 1.1 四条之间的故事线（先背这个）

```text
用户要传大视频
  → 弱网不能整包传死          【条3 分片续传】
  → 入库后转写/摘要/建索引太久  【条1 Kafka + 状态 + 重试】
  → 长视频 ASR 易失败、整段重跑贵【条2 切片持久化，只重失败片】
  → 用户要问细节、要依据        【条4 混合检索 + 引用】
```

**设计动机一句话：**  
整条链路在优化三件事——**别堵请求、别白烧外部调用、答案可解释可恢复**。

## 2. 统一答题公式（四条共用）

回答时尽量按这个顺序，**不要一上来就说「我用了 Kafka/Redis/Milvus」**：

```text
1. 场景：用户/系统在干什么
2. 痛点：出了什么问题
3. 后果：不解决会怎样（超时 / 烧钱 / 体验差 / 无法排障）
4. 根因：为什么会这样
5. 方案：我做了什么（1～2 个关键机制即可）
6. 取舍：为什么不是更简单的方案
7. 边界：当前做到哪，没吹什么
```

### 2.1 开口句式模板

```text
这个点主要是解决……场景下的……问题。
如果不处理，会出现……（后果）。
根因是……。
所以我做了……。
当时也考虑过……，但在我们场景下……，所以选了现在的方案。
需要说明的是，当前做到了……，还不是……。
```

### 2.2 被问住时的安全句式

```text
当前实现是……；边界是……；如果线上再演进，我会优先……，因为……。
```

比瞎编强十倍。

## 3. 项目开场与真实性

### 3.1 项目介绍（30 秒）

VidLens 是一个 Go 后端的 AI 视频理解项目。用户上传本地视频或提交公开链接后，系统异步完成下载/入库、音频提取、ASR 转写、LLM 摘要，并基于转写全文做带引用的单视频问答。  
难点不在「调一下大模型」，而在于：**链路长、耗时长、外部 AI 贵且不稳、大文件弱网、失败后必须知道卡在哪**。所以重点做了异步任务、分片上传、分段 ASR 恢复，以及混合检索 RAG。

### 3.2 项目介绍（2 分钟）

我可以按链路讲：

1. **接入层**：普通上传 / 分片断点续传 / URL 下载（yt-dlp），对象进 MinIO，任务进 MySQL。  
2. **调度层**：HTTP 只建任务、投 Kafka，立刻返回 task_id；下载、转写、摘要、RAG 索引分 topic 消费。  
3. **处理层**：FFmpeg 提音频并按约 300s 切片 ASR；分片结果落 `video_transcription_chunks`；摘要优先复用已有转写。  
4. **恢复层**：可恢复错误写重试次数与下次重试时间，scheduler 到期再投 Kafka；不可恢复快速失败。  
5. **问答层**：转写全文切 chunk，Embedding 入 Milvus，向量 + BM25 风格关键词 + RRF，返回答案与 citations。  
6. **前端**：Vue 主要是验证与展示，面试重点在后端。

技术栈：Gin、GORM、MySQL、Redis、Kafka、MinIO、Milvus、FFmpeg、yt-dlp。  
参考过 Java 视频 AI 项目的**业务问题与工程拆分**，但用 Go 重做：Kafka 而不是 RocketMQ，自研 Redis 锁而不是 Redisson。

### 3.3 AI 写代码 + 参考项目（必练）

**可以这样说：**

> 编码大量用了 AI 辅助，我不会包装成全程手写。业务场景和「异步、分片、重试、去重」这类问题拆分，参考了 Java 的 DoVideoAI 类项目，但技术栈和实现是 Go 侧自己落地的。  
> 我认为项目质量不来自「AI 生成了多少行」，而来自：有没有识别真实痛点、做取舍、联调排障。  
> 例如长视频 ASR 曾出现「任务显示完成但文本只有几百字」——状态成功不等于质量正确，后来才做成切片 ASR 和分片持久化。再如 RAG 状态接口曾被 SPA fallback 成 HTML，HTTP 200 但 Content-Type 不对。  
> AI 加速实现，**问题识别、方案判断、边界控制**是我这边要讲清楚的。

**不要说：**

- 从零手写所有中间件封装、生产级高可用  
- 把 Java 简历的 RocketMQ / Redisson / Function Calling / 60s→50ms 原样说成自己项目  

### 3.4 全链路一句话

```text
HTTP 接单 → MySQL 记任务 → Kafka 异步跑重活 → 分片 ASR/摘要/RAG 索引
→ 用户轮询状态 → 基于转写全文做混合检索问答（带引用）
```

**Kafka 管消息流转，MySQL 管业务真相（阶段、错误、重试）。**

## 4. 简历条 1：Kafka 异步 + 任务状态 + 阶梯退避重试

### 4.1 简历原句

引入 Kafka 异步调度 ASR、摘要生成和 RAG 索引等长耗时任务，结合任务状态与阶梯退避重试，实现阶段追踪和异常恢复。

### 4.2 七步公式（主稿）

| 步骤 | 内容 |
|------|------|
| **1. 场景** | 用户提交视频后，系统要做 ASR、摘要、RAG 索引构建；任一步都可能几十秒到几分钟，且依赖外部 AI / 存储 / 向量库。 |
| **2. 痛点** | 若把整条链路放在 HTTP 同步执行：请求挂很久；中间失败时前端只看到超时；也不知道卡在下载、转写还是建索引。 |
| **3. 后果** | 接口超时、体验崩溃；机器 worker 被请求线程占满；外部抖动时用户只能「再点一次」，容易重复提交、难排障。 |
| **4. 根因** | 视频 AI 处理是**长耗时、多阶段、易失败**的流水线，不是普通 CRUD；同步请求模型装不下。 |
| **5. 方案** | ① HTTP 只做鉴权、落库、投 Kafka，立刻返回 task_id；② 按阶段拆 topic（如下载 / 转写 / 分析 / RAG 索引）由 consumer 执行；③ MySQL 记 status/stage/错误；④ 可恢复错误写 retry 与 next_retry_at，由 scheduler 到期再投递；不可恢复快速失败。 |
| **6. 取舍** | 不用进程内 goroutine 扛可靠性（进程挂任务丢、难多实例）；不用「只靠 Kafka 不 commit 死磕」表达业务重试（缺退避时间、失败原因、用户可见状态）。RocketMQ 延迟/死信在 Java 生态很合适，本项目是 Go + Kafka，业务重试用 **DB scheduler** 补齐。 |
| **7. 边界** | 这是可演示的工程化异步链路，不是多机房高可用；不说「消息绝对不丢」「响应压缩到 50ms」除非自己压测过。默认最大重试次数与退避间隔是配置（如 3 次、阶梯秒级），简历不写死数字，追问再答。 |

### 4.3 30 秒版

长视频 AI 不能堵在 HTTP 里。接口只建任务、发 Kafka，立刻返回任务 ID；ASR、摘要、RAG 索引异步跑。状态落 MySQL，可恢复错误阶梯退避再投，不可恢复快速失败，这样既有阶段追踪，也能异常恢复。

### 4.4 2 分钟版

先说痛点：转写和建索引经常是分钟级，同步会超时且状态黑盒。  
所以改成：上传/创建任务后 producer 投消息，consumer 按 topic 推进阶段，每步更新任务状态。  
失败时先分类——网络超时、429、5xx、存储临时不可用等进入延迟重试；配置缺失、密钥错误、文件不存在等直接失败，避免无效重试烧 Token。  
Kafka 负责削峰和异步流转，**用户看到的进度和恢复入口在任务表**。  
topic 拆分是因为下载、ASR、Embedding 依赖和耗时不同，混在一个 topic 会互相拖累，也不好单独扩容排查。

### 4.5 高频追问

**Q：为什么不用 goroutine？**  
A：goroutine 只能进程内异步。进程重启任务丢，也不好做堆积监控、多实例消费和统一重试策略。视频任务是分钟级，可靠性不能靠内存队列。

**Q：失败后为什么还可以 commit Kafka offset？**  
A：失败已经写入 MySQL 的重试状态（次数、下次时间、错误信息），业务恢复由 scheduler 再投递，而不是占用 consumer 死循环不 commit。消息语义上按至少一次设计，业务层用状态/幂等兜底。

**Q：怎么保证不丢？**  
A：生产端确认写入（如 RequiredAcks=All、发送失败返回错误）；任务先落库再/并与投递配合；投递失败可依赖任务状态补偿。不说「绝对不丢」，说「确认 + 状态 + 补偿降低风险」。

**Q：重复消费怎么办？**  
A：按至少一次语义做幂等：任务/job 状态推进、条件更新、关键路径锁、索引重建前清理旧数据等。细节按实现说，不夸「绝对只执行一次」。

**Q：和条 2 分段 ASR 什么关系？**  
A：条 1 是**任务级**调度与恢复；条 2 是任务内部**分片级**复用。两层一起控成本和可恢复性。

### 4.6 代码证据

- `cmd/server/main.go`：producer / consumer / retry scheduler 装配  
- `internal/mq/producer.go`  
- `internal/mq/consumer.go`  
- `internal/mq/retry.go`  
- `internal/model/task.go`、`task_job` 相关模型  

### 4.7 不要这么说

- 使用 RocketMQ / 死信队列原生能力（除非你真上了）  
- 接口从 60s 优化到 50ms（参考项目数字，你没测就别说）  
- 「高并发削峰填谷生产架构」

## 5. 简历条 2：长视频分段 ASR + 持久化 + 只重跑失败片

### 5.1 简历原句

设计长视频分段 ASR 与结果持久化机制，任务异常后仅重跑失败片段并复用已完成结果，避免整段视频重复处理。

### 5.2 七步公式（主稿）

| 步骤 | 内容 |
|------|------|
| **1. 场景** | 十几分钟甚至更长的视频需要完整转写，才能做摘要和 RAG。 |
| **2. 痛点** | ① 整段音频一次丢给 ASR：请求体过大、长音频截断/识别不完整；② 中途某一片失败，如果整任务重来，前面成功的片白做。 |
| **3. 后果** | 「状态完成但文本只有几百字」；重试一次 ≈ 再烧一整段时间与 Token；成本高、难定位是哪一段坏了。 |
| **4. 根因** | 外部 ASR 对**时长/请求体**有实际限制；转写工作天然可切分，且**部分成功可复用**。 |
| **5. 方案** | FFmpeg 提取低码率音频（如 16k 单声道）并按约 **300s** 切片 → 逐段 ASR → 写入 `video_transcription_chunks`（chunk_index、status、content、error 等）→ 重试时 **completed 复用，只跑 failed/pending** → 合并为全文再给摘要/RAG。 |
| **6. 取舍** | 不分片：实现简单，但长视频质量与失败成本不可接受。分片全并行：吞吐更好，但要控并发与 provider TPM，当前以正确性和可恢复为主，多为串行/受控。摘要优先复用已有转写，避免「总结再跑一遍 ASR」。 |
| **7. 边界** | 不吹「并行 ASR 吞吐提升 xx%」；分片粒度 300s 是工程默认，可配置，不是理论最优。空结果/配置错误等可能应快速失败而非无限重试（与条 1 错误分类一致）。 |

### 5.3 30 秒版

长视频一次 ASR 容易超限或不完整。我按时间切片转写，每片结果入库；失败只重跑失败片，已完成的复用，避免整段视频重复处理。

### 5.4 2 分钟版

真实问题里出现过：任务显示完成，但十几分钟视频只有几百字——说明不能只看 status。  
根因是长音频/请求体限制和单次调用不完整。  
方案是：压缩音频 → 固定时长切片 → 每片独立状态落库 → 合并全文。  
这样重试从「整单重做」变成「增量恢复」，和 Kafka 任务级重试配合：任务可以再进 consumer，但 consumer 内部跳过已完成 chunk。  
摘要阶段优先读已有转写，进一步减少重复 ASR。

### 5.5 高频追问

**Q：为什么是大约 300 秒？**  
A：在请求体限制、单次识别稳定性与分片数量之间折中。过短片数暴涨、调用次数多；过长又回到超限/截断。具体值可配置，面试强调**分片 + 持久化**思想，不纠缠「是否最优」。

**Q：怎么知道哪些片已完成？**  
A：表 `video_transcription_chunks`，按 task_id + chunk_index，status=completed 且 content 可复用。

**Q：某片一直失败怎么办？**  
A：片级可记 retry；超过策略或不可恢复错误（如持续空结果、鉴权失败）应上升到任务失败并写清 error，避免静默成功。

**Q：和 RAG chunk 是不是一回事？**  
A：不是。ASR 时间片是**转写生产**；RAG chunk 是**检索用文本切分**。前者解决 ASR 可靠性与成本，后者解决问答召回。

## 6. 简历条 3：分片上传 + 断点续传 + Redis + MinIO 合并

### 6.1 简历原句

采用分片上传与断点续传机制，通过 Redis 记录分片状态并使用 MinIO 服务端合并，避免上传中断后整文件重传。

### 6.2 七步公式（主稿）

| 步骤 | 内容 |
|------|------|
| **1. 场景** | 用户上传较大本地视频，网络可能抖动、中断、刷新页面。 |
| **2. 痛点** | 整文件一次 POST：中途失败必须从头再来。 |
| **3. 后果** | 弱网下几乎传不完；带宽浪费；用户放弃，后续 AI 链路根本走不到。 |
| **4. 根因** | 大文件传输失败概率高，需要**可恢复的传输进度状态**。 |
| **5. 方案** | 前端按分片上传；后端把分片写入 MinIO；**Redis Set** 记录已上传分片编号；恢复时只补缺失片；齐了校验完整性，再 **ComposeObject** 服务端合并为完整对象，并创建任务。 |
| **6. 取舍** | 为何状态放 Redis：进度是临时协调数据，读写快、好过期。为何服务端合并：客户端下载各片再拼流量大、失败点多。合并决策要避免多请求同时 merge（可用前端汇总触发 + 后端合并锁/幂等，按你实现讲）。 |
| **7. 边界** | 没压测别写「GB 级 / 成功率提升 xx%」。Redis 丢进度时可引导重新初始化上传会话或重传缺失片，要承认临时状态与持久对象的分层。 |

### 6.3 30 秒版

大文件整包上传中断成本太高。分片上传，Redis 记哪些片已到，断点只补缺片，最后 MinIO 服务端合并，避免整文件重传。

### 6.4 2 分钟版

上传是整条 AI 链路的入口。若这里不可靠，后面 Kafka、ASR 都没意义。  
弱网下整包失败最伤体验，所以拆成：分片传输 + 进度可查 + 服务端合并。  
Redis 只负责「哪些片齐了」；真正字节在 MinIO。  
合并前校验分片完整性，合并过程注意并发与幂等，避免重复 Compose。  
合并成功后再进入任务创建与后续异步处理。

### 6.5 高频追问

**Q：断点续传怎么知道缺哪片？**  
A：Redis 里已上传分片集合；前端 check 接口对比后只传缺失编号。

**Q：为什么不用只存本地磁盘拼文件？**  
A：多实例时本地盘不共享；对象存储更适合作为统一文件层，且 Compose 由存储侧完成。

**Q：Redis 挂了怎么办？**  
A：上传进度可能丢失，需要重新开始或重建会话；核心文件若已在 MinIO 可按对象存在性补偿。限流/锁同理有降级策略讨论空间——诚实说当前重点与改进方向。

**Q：和条 2「分片」有何不同？**  
A：条 3 是**文件传输分片**；条 2 是**ASR 时间分片**。一个解决上传，一个解决转写。

## 7. 简历条 4：Milvus + BM25 + RRF + 引用

### 7.1 简历原句

基于 Milvus + BM25 + RRF 实现语义与关键词混合检索，并返回引用片段，提升视频问答结果的可追溯性。

### 7.2 七步公式（主稿）

| 步骤 | 内容 |
|------|------|
| **1. 场景** | 用户针对**某一个已转写视频**提问：某概念、数字、原话、时间段讲了什么。 |
| **2. 痛点** | 模型不知道用户视频内容；全文太长塞不进 prompt；只用摘要会丢细节；纯向量对专有名词/数字/缩写不稳；只给答案无依据用户不信。 |
| **3. 后果** | 幻觉式回答；答非所问；无法回到原片段核对。 |
| **4. 根因** | 需要 **RAG：先检索再生成**；且单路召回不够覆盖「语义相似」和「精确词匹配」。 |
| **5. 方案** | 知识源用 **ASR 全文**（不是摘要）切 chunk → Embedding 写 Milvus + MySQL 存 chunk 元数据 → 问答时向量召回 + BM25 风格关键词召回 → **RRF 按排名融合** → LLM 基于 top 片段生成 → **返回 citations**。检索 filter：`user_id + task_id + embedding_model`。 |
| **6. 取舍** | 为何不用摘要当库：二次压缩丢细节。为何 RRF：两路分数尺度不同，用排名融合实现简单、适合混合。为何暂不把 ES 当必选项：chunk 规模与单视频场景下，轻量关键词召回 + 向量够用；规模上来再迁专业检索。rerank 是质量优化下一步，宜有评测再上，避免堆技术。 |
| **7. 边界** | **单视频 RAG**，不是跨视频企业知识库；BM25 是项目内/轻量实现，不是完整 Elasticsearch；没评测集就不要报准确率数字；SSE/流式按真实 provider 能力说，不支持则 fallback。 |

### 7.3 30 秒版

问答要贴视频内容、有依据。用转写全文建索引，语义向量和关键词两路召回，RRF 融合后让模型回答，并返回引用片段，提升可追溯性。

### 7.4 2 分钟版

直接问 LLM 等于让它瞎编用户视频。RAG 把「视频里相关的话」检索出来再生成。  
知识源选 ASR 全文而不是摘要，因为用户常问细节。  
纯向量对「专有名词、型号、数字」可能弱，所以加关键词路；两路分数不可比，用 RRF 融排名。  
返回 citations，用户能对照原文，也降低脱离材料编造的风险。  
隔离上业务校验 session/task 归属，向量侧再按 user_id、task_id、embedding_model 过滤，避免串视频串用户。

### 7.5 高频追问

**Q：为什么不用摘要做知识库？**  
A：摘要是二次生成，会压缩改写；细节问答容易缺证据。摘要适合展示，不适合当唯一检索源。

**Q：RRF 是什么？为什么不用简单加权分数？**  
A：Reciprocal Rank Fusion 主要看排名：\( \sum 1/(k+rank) \)。避免向量相似度与 BM25 分数量纲不一致时调权困难。

**Q：如何防止搜到别人的视频？**  
A：业务层校验归属；Milvus filter 带 user_id + task_id（及 embedding_model）。

**Q：召回质量怎么证明？**  
A：可提有 rag-eval / 样例集方向；若评测不完整，诚实说「机制已落地，系统性评测是下一步」，不要编数字。

## 13. 全链路架构（面试官：完整讲一遍流程）

> 参考小红书文档「四阶段」讲法，改成 VidLens 真实链路。**先用户动作，后中间件**。

### 13.1 总览一句话

```text
接入（上传/URL）→ 任务落库 → Kafka 异步处理（下载/转写/摘要/RAG 索引）
→ 前端轮询状态 → 基于转写全文做混合检索问答
```

### 13.2 四阶段口述稿（建议背这个）

**第一阶段：稳健接入与存储**

用户有三条入口：

1. **普通上传**：HTTP 收文件，算 MD5，写入 MinIO，创建 `video_assets`（可复用）和 `video_tasks`，返回 task_id。  
2. **分片上传**：分片进 MinIO，Redis Set 记已传片号；齐了后 merge，校验完整性，`ComposeObject` 服务端合并，再落资产与任务。  
3. **URL 下载**：HTTP **只建下载任务 + 投 Kafka**，不在请求里跑完 yt-dlp；download consumer 里下载、算 MD5、入库。

这一阶段目标：**文件真正进入系统，任务可追踪**。不是在这里跑 ASR。

**第二阶段：异步调度与削峰**

用户点「转写 / 总结 / 建索引」时：

- 同步：鉴权、校验任务归属与状态、必要时限流、**投递 Kafka**、返回。  
- 异步：consumer 执行 FFmpeg、ASR、LLM、Embedding、Milvus。

接口释放请求线程；长耗时与外部抖动进队列。前端靠 **轮询任务详情** 看 `status` / `stage` / 错误与重试信息。

**第三阶段：消费、分片 ASR、状态与容错**

Consumer 按 topic 处理，例如：

- `video-download`：远程下载与入库  
- `video-transcribe`：提音频 → 约 300s 切片 ASR → 写 `video_transcriptions` + `video_transcription_chunks`  
- `video-analyze`：优先复用已有转写 → LLM 摘要  
- `video-rag-index`：切 RAG chunk → Embedding → MySQL + Milvus  

失败时：

- 可恢复 → 写 `retry_count` / `next_retry_at` / 错误信息，scheduler 到期再投  
- 不可恢复 → 快速失败，避免烧 Token  
- 业务真相在 **MySQL 任务/job 表**，不只靠 Kafka offset  

**第四阶段：RAG 问答与结果展示**

知识源是 **ASR 全文**，不是摘要。  
索引：文本 chunk + 向量。  
问答：向量召回 + BM25 风格关键词 + RRF → 拼 prompt → LLM → **citations**。  
会话：MySQL 全量消息 + Redis 最近轮次缓存（有实现则提，没有不吹）。

### 13.3 同步 vs 异步边界（必问）

| 同步（HTTP 内） | 异步（Kafka consumer） |
|-----------------|------------------------|
| 鉴权、参数校验 | URL 真正下载 |
| 普通/分片上传收文件、合并 | FFmpeg / ASR / LLM |
| 创建 task、投递消息 | Embedding / 写 Milvus |
| 查询任务状态、聊天请求组装 | 长链路失败重试执行 |

**边界原则：**  
同步保证「任务被创建、文件可用、用户拿到 task_id」；异步保证「长任务不堵请求、可重试、可堆积」。

### 13.4 前端怎么知道完成了？

当前以 **轮询** 为主：拿到 task_id 后定时拉任务详情。

建议强调字段分工：

- `status`：整体（pending / queued / running / completed / failed / dead）  
- `stage`：阶段（downloading / transcribing / summarizing / indexing…）  
- `last_error_msg` / `retry_count` / `next_retry_at`：失败与恢复  

> 单个 `running` 解释不了卡在哪；`completed` 也不等于转写质量一定对（见 §15 案例一）。

后续可演进 SSE/WebSocket 推状态，但当前轮询 + 明确状态字段已够解释业务。

### 13.5 组件分工（防「堆技术」质疑）

| 组件 | 解决什么问题 | 不是什么 |
|------|----------------|----------|
| Kafka | 长任务异步、削峰、多阶段流转 | 业务状态真相源 |
| MySQL | 任务/结果/重试/权限归属 | 大文件存储 |
| MinIO | 视频与分片对象 | 任务状态机 |
| Redis | 分片进度、锁、限流、热数据 | 持久业务主库 |
| Milvus | 向量语义召回 | 全文专业检索引擎 |
| FFmpeg | 音频提取与切片预处理 | 识别模型本身 |

### 13.6 白板怎么画（30 秒画完）

```text
[Vue] --HTTP--> [Gin]
                 |-- MySQL (task/job/transcript/chunk/chat)
                 |-- MinIO (video objects)
                 |-- Redis (chunks set / lock / limit)
                 |-- Kafka topics --> [Consumers]
                                      |-- FFmpeg / ASR / LLM / Embed
                                      |-- Milvus
                 |-- 轮询 status
                 |-- Chat RAG (vector + BM25 + RRF + citations)
```

## 14. 数据模型与生命周期（面试官：有哪些表？为什么拆？）

### 14.1 30 秒总答

> 我没有把所有信息塞进一张 `video_tasks`。按生命周期拆：  
> **`video_assets` 管真实文件，`video_tasks` 管用户一次任务，`task_jobs` 管下载/转写/分析/索引等动作；**  
> 转写全文、ASR 分片、摘要、RAG chunk、聊天、AI 配置各自独立。  
> 目的是：文件可复用、异步可追踪、大字段不拖列表、RAG 可检索可隔离、失败可排障。

### 14.2 核心表地图

```text
users
  └── video_tasks ──► video_assets          （多任务可共用一个文件资产）
        ├── task_jobs                       （子任务：download/transcribe/analyze/rag_index）
        ├── video_transcriptions            （转写全文，大文本垂直拆分）
        ├── video_transcription_chunks      （ASR 时间片状态与结果）
        ├── ai_summaries                    （摘要）
        ├── video_rag_indexes               （索引构建状态）
        ├── video_chunks                    （RAG 文本 chunk + 元数据）
        │         └── Milvus vectors
        └── chat_sessions / messages        （会话与消息）

user_ai_profiles / ai_call_logs …        （BYOK 与调用审计，简历未写可当扩展）
```

### 14.3 为什么这样拆（用「后果」说话）

| 拆分 | 不拆的后果 | 拆了的收益 |
|------|------------|------------|
| asset vs task | 同一文件多人上传重复存、删任务误删文件难处理 | 内容复用；删除 task ≠ 一定删物理文件 |
| task vs job | 一个 status 说不清下载/转写/索引哪步失败 | 子任务独立重试与排障 |
| 转写/摘要 vs task 主表 | 列表接口加载超长文本，主表行巨大 | 列表轻、详情再查大字段 |
| transcription vs transcription_chunks | 失败只能整段重转写 | 片级复用，对应简历条 2 |
| video_chunks + rag_indexes | 只有向量无文本/无状态 | 引用展示、BM25、索引状态可解释 |
| chat 独立 | 任务表膨胀 | 会话演进不影响任务主链路 |

### 14.4 任务状态机怎么讲

`status`（整体）与 `stage`（阶段）配合：

```text
Pending → Queued → Running → Completed
                         ↘ Failed →（可重试）再 Queued/Running
                         ↘ Dead（超过最大重试，需人工/用户重触发）
```

`stage` 示例：`downloading` / `transcribing` / `summarizing` / `indexing`。

**面试金句：**  
Kafka 消息可以丢重投，**用户看到的进度和错误以任务表为准**。

### 14.5 三类「分片/块」必须分清（高频混淆）

| 名称 | 解决什么 | 存在哪 |
|------|----------|--------|
| 上传分片 | 弱网大文件传输 | MinIO 分片对象 + Redis 已传集合 |
| ASR 时间片 | 长音频转写可靠与增量重试 | `video_transcription_chunks` |
| RAG 文本 chunk | 检索粒度与引用 | `video_chunks` + Milvus |

说混会被认为「背名词没落地」。

### 14.6 删除任务时一致性（简答）

按实现说原则即可：

- 删 task 关联：转写、摘要、RAG chunk、会话等用户侧数据  
- Milvus 向量按 task/user/model 清理  
- **共享 asset**：仍有其他 task 引用则不删 MinIO 对象  

这是最终一致性 + 引用计数/检查思维，不要吹分布式事务。

## 15. 真实排障案例（证明「不是只会调 API」）

> 面试官最吃这一套。每案固定：**现象 → 证据 → 根因 → 修复 → 边界**。  
> 建议主讲 **案例 A**，备用 **案例 B**。详细证据见 `docs/troubleshooting-and-interview-notes.md`。

### 15.1 案例 A：15 分钟视频「完成」但转写只有几百字（对应简历条 2）

**现象**  
短视频正常；约 15 分钟视频要么失败（base64/体积限制），要么 status=完成但全文极短。

**证据怎么查**  
看 `video_tasks.error_msg`、`video_transcriptions` 字符数、consumer 日志；对比视频时长与合理字数。

**根因（两层）**  
1. 音频 base64/请求体过大；  
2. 即使压缩后，**单次长音频 ASR 仍可能只返回前面一段**——完成 ≠ 质量正确。

**修复**  
FFmpeg 低码率单声道 → 约 300s 切片 → 逐段 ASR → 合并；并持久化 `video_transcription_chunks`，重试复用已完成片。

**边界**  
未做片间 overlap 去重、默认非大规模并行 ASR；强调工程恢复能力，不吹识别 SOTA。

**30 秒口述**  
> 我遇到过任务显示成功但转写明显过短。说明不能只信 status。根因是长音频单次 ASR 的体积与截断问题。后来改成分片转写和片级落库，失败只重跑失败片。

### 15.2 案例 B：RAG 已构建仍提示重建 / 接口 200 却是 HTML（可对应条 4 工程化）

**现象**  
索引其实建过，前端仍让重建；或状态接口异常。

**典型根因方向**（按你排障笔记讲真的那条）  
例如 GET 状态路由缺失，Gin SPA fallback 返回 `index.html`，HTTP 仍 200，前端当 JSON 解析失败或当成无状态。

**修复方向**  
补真正的 RAG 状态接口与 `video_rag_indexes` 状态表；前端校验 Content-Type / 字段。

**金句**  
> HTTP 200 不等于业务成功，还要看返回体是不是预期的 JSON。

### 15.3 案例 C：公开部署不能继续用服务端 Key（扩展，BYOK）

**痛点**  
任意注册用户消耗维护者 Token。  

**方案**  
用户级 ASR/LLM/Embedding 配置，Key AES-GCM 入库；无配置明确失败，**不静默 fallback**。  

简历未写也可在「成本与安全」追问时讲。

### 15.4 讲排障时的态度

- 先说**自己怎么发现**（日志、DB、字数异常），再说方案  
- 承认 AI 辅助写代码，但**收敛问题靠现象与证据**  
- 主动说边界，比装生产级更加分

## 16. 扩展亮点七步卡（简历没写，被问到再展开）

> 只作钩子，**不要主动把面试带偏到你背不熟的点**。

### 16.1 Redis 分布式锁 + WatchDog

| 步骤 | 内容 |
|------|------|
| 场景 | 同一 MD5 并发分析/合并 |
| 痛点 | 重复 ASR/LLM、合并竞态 |
| 后果 | 双倍 Token、脏状态 |
| 根因 | 临界区 + 处理时长不确定 |
| 方案 | Redis 锁 + owner；Lua 校验释放；WatchDog 续期 |
| 取舍 | 非 Redisson；轻量自研，适合当前规模 |
| 边界 | 不说等价于 Redisson 全能力 |

### 16.2 MD5 内容去重 / 资产复用

| 步骤 | 内容 |
|------|------|
| 场景 | 相同视频多次上传 |
| 痛点 | 重复存对象、重复处理 |
| 方案 | `video_assets.file_md5` 唯一；task 引用 asset |
| 边界 | 去重的是文件内容，不自动共享别人的转写隐私（按 user/task 隔离结果） |

### 16.3 Redis Lua 令牌桶限流

| 步骤 | 内容 |
|------|------|
| 场景 | 分析/问答等高成本接口被连点或刷 |
| 后果 | 账单与 TPM 打满 |
| 方案 | 用户+接口维度；Lua 原子补桶扣令牌 |
| 边界 | 频率限制 ≠ 完整 token 计费配额 |

### 16.4 用户 BYOK

见 §15.3。强调**成本归属与失败明确**，不强调「企业级密钥托管」。

## 17. 一致性、幂等、重启（后端味追问）

### 17.1 一致性怎么定位

> 不是全局分布式事务，而是 **围绕任务状态的最终一致性**。  
> 用户最在意的是：状态不能撒谎——没做完别标完成；失败要写清 stage 与原因；RAG 失败不能假装 indexed。

手段：状态机、条件更新、唯一索引、锁、重试调度、索引重建前清理等。

### 17.2 幂等从哪几层想

1. **上传/资产**：MD5 唯一，复用 object  
2. **任务推进**：status/job 条件更新，避免乱序覆盖  
3. **消费**：至少一次语义 → 业务可重入（如索引先清再写）  
4. **合并**：合并锁 / 前端单次 merge + 后端幂等  

### 17.3 进程中途挂了

- 持锁崩溃：WatchDog 停，TTL 后锁释放（不永久死锁）  
- 消费中崩溃：未成功提交的可再消费；已成功落库的靠状态幂等  
- 可重试失败：scheduler 按 `next_retry_at` 再投，不让 consumer 长期 sleep 占分区  

承认：极端「外部已成功、本地未落库」仍需更细补偿——这是诚实边界。

### 17.4 可重试 vs 不可重试（各举例）

| 可重试 | 不可重试 |
|--------|----------|
| 超时、网络抖动 | 未配置 AI / Key 错误 |
| 429、5xx | 文件/对象不存在 |
| MinIO/Milvus 临时不可用 | Embedding 维度不匹配等确定性错误 |

**原则：** 会自己好的等一等；重试一百次也没用的快速失败。
