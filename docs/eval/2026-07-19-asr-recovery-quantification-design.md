# ASR 分片复用量化实验设计

## 1. 目标

本实验用于量化 VidLens 长视频分段 ASR 在可重试故障后的恢复收益。实验只验证当前已经实现的能力：完成片段持久化、失败片段重试和完成片段复用，不引入新的 ASR 算法或并发转写策略。

实验需要回答三个问题：

1. 重试时是否完全跳过已经持久化完成的片段；
2. 相比从头重跑，减少了多少实际送入 ASR Provider 的音频时长和请求次数；
3. 在真实 Provider 下，减少的处理量是否转化为稳定的恢复时间收益。

最终简历优先使用“重试阶段 ASR 输入音频时长降低率”。恢复耗时只有在重复实验稳定时才进入简历。

## 2. 非目标

- 不声称生产故障率、生产 SLA 或线上成本节省比例；
- 不比较不同 ASR Provider 或模型的识别质量；
- 不改变当前 300 秒分片配置以追求更好数字；
- 不把故障前已经发生的正常 ASR 调用包装成优化收益；
- 不使用 mock 延迟作为真实恢复耗时证据。

## 3. 现有实现基础

当前 `transcribeAudio` 会在调用 Provider 前查询 `video_transcription_chunks`。当对应 `task_id + chunk_index` 已是 `completed` 且存在内容时，直接复用持久化文本并增加 `vidlens_asr_chunk_reuse_total`，否则调用 ASR 并更新片段状态。默认音频分片长度是 300 秒。

现有单元测试已经证明“已完成第一段、重试只调用第二段”的控制流。本实验在此基础上增加真实音频时长测量、可重复故障注入、全量重跑对照和真实 Provider 验收。

## 4. 实验样本

使用三个不会提交到仓库、可长期保留并能重复运行的本地视频：

| 样本 | 目标时长 | 内容要求 |
| --- | ---: | --- |
| A | 约 30 分钟 | 中文连续讲话 |
| B | 约 60 分钟 | 英文或中英混合讲话 |
| C | 约 90 分钟 | 语速变化或带有限背景音 |

每个样本保存以下非敏感元数据：

- 视频 SHA-256；
- 提取后完整音频文件 SHA-256；
- FFprobe 得到的音频总时长；
- 实际分片数、每个分片的真实时长和逐片段 SHA-256；
- FFmpeg 可执行文件版本、可执行文件 SHA-256 和完整分片参数；
- 测试使用的 ASR Provider、模型和配置摘要；
- Git commit、数据库类型和执行环境标识。

原始视频、完整转写、API Key、请求头和 Provider 完整响应不得进入 Git artifact。

## 5. 故障矩阵

每个视频分别在约 25%、50% 和 75% 处理位置注入一次可重试故障，共九个场景。具体失败片段在读取实际分片数量后按预注册规则计算：

```text
failure_index = max(1, ceil(chunk_count * failure_fraction))
```

`failure_index` 使用一基编号展示，底层 chunk index 仍遵守现有零基数据库语义。实验 manifest 在第一次运行前写入实际失败片段，后续不得根据结果调整。

故障通过实验专用 ASR wrapper 或本地代理注入。指定请求返回一次可被当前重试分类识别的 timeout/503，其余片段调用真实 Provider。故障注入必须保留正常的 AI 调用审计、任务失败记录和 retry scheduler 路径。

## 6. 对照设计

每个场景先运行到预定故障点，得到一个可验证的失败后状态。该状态必须包含：

- 已完成片段及内容；
- 一个 failed 或尚未完成的目标片段；
- 对应 task/job/retry 元数据；
- 原视频和可重新生成的相同音频分片。

随后从逻辑相同的失败状态创建两个隔离分支：

### 6.1 断点恢复组

- 保留完成片段；
- 走当前 Kafka 重试和 `transcribeAudio` 复用逻辑；
- 只允许失败片段及其后续未完成片段进入 Provider。

### 6.2 全量重跑组

- 使用独立 task 或独立数据库副本，避免污染恢复组；
- 将已完成片段状态清空，模拟没有片段检查点的旧行为；
- 其他视频、分片、Provider、模型、重试起点和机器条件保持一致。

两组都从“重新投递 transcribe job”开始计时。全量重跑组只用于实验对照，不成为正式运行配置。

恢复组重试时会重新执行 FFmpeg 分片，因此两组进入 Provider 前必须重新计算逐片段 SHA-256，并与失败后状态冻结的片段 manifest 对比。只有片段数量、索引、真实时长、字节哈希、FFmpeg 身份和分片参数全部一致，对照才允许进入指标聚合。仅有相同 `chunk_index` 不足以证明两组处理的是同一段音频。

## 7. 指标定义

### 7.1 主指标

对每个真正进入 `Strategy.Transcribe` 的音频文件执行 FFprobe，累加真实音频秒数：

```text
retry_audio_reduction
= 1 - resume_retry_asr_seconds / full_retry_asr_seconds
```

九个场景的主报告同时展示：

- 每个场景的两组 ASR 输入秒数；
- 每个场景的降低率；
- 九个场景的宏平均降低率；
- 累计避免重复处理的音频分钟数；
- 最小、最大和中位场景收益。

不能使用“请求次数乘以 300 秒”代替真实时长，因为末尾分片通常不足 300 秒。

### 7.2 次指标

- 重试阶段 Provider 请求次数；
- 从 retry job 重新投递到 transcription 持久化完成的墙钟时间；
- Kafka transcribe job duration；
- `vidlens_asr_chunk_reuse_total` 增量；
- task/job retry 次数和最终状态；
- AI 调用成功、失败和总耗时。

### 7.3 正确性门槛

每次运行必须同时满足：

- 最终 completed chunk 数等于实际分片数；
- chunk index 连续、唯一且顺序正确；
- 恢复前、恢复组和全量组的逐片段 SHA-256、时长及分片参数完全一致；
- 已完成片段的内容哈希在恢复前后不变；
- 恢复组中已完成片段的 Provider 调用次数为零；
- 最终 transcription 由每个片段恰好拼接一次；
- task 和 transcribe job 进入预期完成状态；
- 数据库、Prometheus、AI 调用审计和实验 recorder 的计数能够对账。

任一正确性门槛失败，该次运行不得进入性能聚合。

## 8. 恢复时间复测

九个场景各运行一次即可确定调用次数和真实音频时长。时间指标额外使用约 60 分钟视频的 50% 故障点执行五组配对实验。

为减少时间段和 Provider 负载偏差，运行顺序交替使用 AB/BA：

```text
第 1 组：恢复组 -> 全量组
第 2 组：全量组 -> 恢复组
循环交替
```

报告中位数、P25、P75、最小值和最大值。五组样本不报告 P95。若恢复耗时方向不稳定，简历只保留音频时长和请求次数指标。

## 9. Artifact 与可复现性

每次运行生成一个独立目录，至少包含：

- `manifest.json`：代码、环境、视频和模型身份；
- `chunks.jsonl`：片段索引、音频时长、音频 SHA-256、状态和脱敏转写内容哈希；
- `ffmpeg.json`：FFmpeg 版本、可执行文件 SHA-256、输入音频 SHA-256 和分片参数；
- `provider-calls.jsonl`：调用片段、状态、耗时和错误分类，不保存密钥；
- `task-state.json`：运行前后 task/job 状态摘要；
- `metrics.json`：原始指标；
- `report.md`：人类可读结论和失败说明。

原始 artifact 默认保存在 Git 忽略目录。只提交脱敏汇总报告和生成报告所需的 manifest 摘要。

## 10. 实验失败处理

- Provider 临时不可用：整次运行标记 infrastructure failure，不进入效果统计；
- 注入错误未被识别为 retryable：修复实验工具后重新创建新的 run ID；
- 片段时长读取失败：停止运行，不用估算值代替；
- 数据库或 Kafka 中间状态不一致：保留 artifact，作为失败实验记录；
- 结果不支持耗时提升：如实保留报告，只写已证明的处理量收益。

## 11. 简历准入条件

只有满足以下全部条件才允许写量化结果：

1. 九个故障场景全部通过正确性门槛；
2. 每个场景都能从 artifact 复算音频秒数降低率；
3. 已完成片段在恢复组中没有重复 Provider 调用；
4. 报告明确限定为“故障后的重试阶段”；
5. 恢复耗时只有在五组配对实验方向稳定时才写入。

最终简历句式中的数字由报告生成，不在设计阶段预填：

> 设计长视频分段 ASR 与结果持久化机制；在 3 个 30～90 分钟视频、9 组超时故障注入实验中，重试复用已完成片段，使重复送入 ASR 的音频时长平均减少【实测值】，恢复耗时中位数由【实测基线】降至【实测结果】，并通过片段完整性校验确认无缺失或重复。
