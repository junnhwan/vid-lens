# 最终简历问题与回答话术

这份文档只服务于最终简历版本。每一章对应简历上的一条 bullet，按“面试官会怎么追问”来准备。

回答基本公式：

背景痛点 -> 主流程 -> 关键实现 -> 选型原因 -> tradeoff -> 兜底保障 -> 代码证据。

## 1. 项目总览追问

### Q1：你这个项目一句话怎么介绍？

答：

这是一个 AI 视频内容理解平台。用户上传本地视频或提交公开视频链接后，后端异步完成视频下载、音频提取、ASR 转写、AI 摘要和 RAG 索引构建，最后支持基于视频转写内容的智能问答。项目重点不是单纯调 AI API，而是围绕视频长任务、大文件上传、重复处理、外部服务失败和 RAG 可解释性做工程化处理。

### Q2：你这个项目最核心的技术主线是什么？

答：

我会分两条线讲。

第一条是视频处理工程化：Kafka 异步任务、MySQL 任务状态、Redis 锁去重、分片上传、限流和失败治理。

第二条是 AI 应用落地：ASR 转写全文、RAG 索引构建、Milvus 向量召回、BM25 关键词召回、RRF 融合和 citations。

### Q3：这个项目是不是就是调大模型 API？

答：

不是。调 API 只是能力入口，真正的问题是视频怎么稳定处理。比如长视频不能同步阻塞 HTTP，请求失败要能恢复；相同视频不能重复 ASR 和向量建库；大文件弱网上传要能断点续传；RAG 不能直接把摘要塞给模型，而要基于 ASR 全文检索。我的工作主要在这些工程链路上。

### Q4：你项目里最能体现后端能力的点是什么？

答：

我会选三个：

1. Kafka 异步任务和 MySQL 失败治理。
2. Redis 锁、分片上传、MinIO 合并这套大文件和并发处理。
3. RAG 混合检索和 citations。

它们分别对应可靠性、文件工程和 AI 应用落地。

## 2. Kafka 异步化

### 简历原句

引入 Kafka 将视频下载、ASR 转写、摘要生成和 RAG 索引构建等长耗时任务异步化，HTTP 接口仅负责任务落库和消息投递，避免分钟级视频处理阻塞请求链路

### Q1：背景痛点是什么？

答：

视频处理链路太长。一个视频从上传到最终可问答，中间要经历下载、FFmpeg 提取音频、ASR、LLM 摘要、Embedding 和 Milvus 写入。任何一个步骤都可能耗时几十秒甚至更久。如果放在 HTTP 请求里同步执行，接口会阻塞，前端一直等待，中间失败也不好恢复。

### Q2：异步化以后完整流程是什么？

答：

用户上传视频或提交 URL 后，接口先做鉴权、参数校验、任务创建，然后写入 `video_tasks` 和 `task_jobs`，再投递 Kafka 消息。后续由 consumer 执行下载、转写、摘要和 RAG 索引构建。每个阶段完成后更新任务状态，失败时记录错误和重试信息。前端通过任务状态接口查看进度。

### Q3：具体用了哪些 topic？

答：

项目里按任务类型拆 topic，核心包括：

- `video-download`
- `video-transcribe`
- `video-analyze`
- `video-rag-index`

拆 topic 的好处是不同任务耗时和外部依赖不同，可以独立消费、独立扩容、独立排查。例如 RAG 索引构建依赖 Embedding 和 Milvus，不应该拖住 ASR 转写结果落库。

### Q4：为什么不用 goroutine 或线程池？

答：

goroutine 适合短期异步，但不适合作为可靠任务队列。进程重启后内存里的任务会丢，也不好做堆积、重试和多实例消费。Kafka 把任务持久化到 broker，可以支持消费者恢复、消费组扩展和任务削峰。视频处理这种长耗时任务更适合 MQ。

### Q5：为什么选择 Kafka？

答：

当前项目是 Go 后端，任务更像日志式异步处理，重点是削峰、堆积能力和 consumer group 扩展。Kafka 在这些方面比较适合。RocketMQ 的延迟消息和事务消息更强，但更偏 Java 生态；RabbitMQ 路由灵活，但大规模堆积不是它最强项。我的项目选择 Kafka，是结合当前技术栈和任务特点做的。

### Q6：Kafka 和 MySQL 状态怎么配合？

答：

Kafka 解决消息传递，MySQL 解决业务状态。用户关心的是任务是否下载中、转写中、建库中、失败原因是什么，这些不能只靠 Kafka offset 表达。所以我用 Kafka 做异步流转，用 `video_tasks` 和 `task_jobs` 记录业务状态、阶段、错误和重试信息。

### Q7：重复消费怎么办？

答：

Kafka 是至少一次语义，所以业务层必须幂等。项目里通过任务状态、job 状态、唯一约束和锁来降低重复执行影响。例如分片转写按 `task_id + chunk_index` 唯一，RAG 重建前会删除旧向量并替换 chunks，上传侧用 MD5 复用 asset，合并侧用 Redis 锁保护临界区。

### Q8：压力追问：是不是 MQ 套壳？

答：

不是只把同步逻辑丢进 MQ。真正的改造是把视频处理拆成多个业务阶段，每个阶段都有独立状态、失败原因和恢复策略。Kafka 做解耦，MySQL 做业务可观测，retry scheduler 做恢复。否则用户和开发者都不知道任务卡在哪一步。

### 30 秒话术

视频下载、ASR、摘要和 RAG 索引构建都可能是分钟级任务，所以我没有在 HTTP 请求里同步执行。HTTP 接口只做任务落库和 Kafka 投递，然后由 consumer 按阶段处理。Kafka 负责异步削峰，MySQL 的 `video_tasks` 和 `task_jobs` 负责记录状态和失败原因，这样用户能看到任务进度，失败也有恢复入口。

## 3. Redis 分布式锁 + MD5 去重

### 简历原句

设计 Redis 分布式锁 + WatchDog，结合 MD5 内容指纹和视频文件复用，保护分片合并、分析任务消费等临界区，降低并发重复入库和重复处理风险

### Q1：背景痛点是什么？

答：

同一个视频如果被重复上传，后端可能重复 ASR、重复摘要、重复 Embedding 和重复写向量库。这些操作都消耗外部 AI 服务调用成本。并发场景下，两个相同视频可能同时上传，都没查到已有 asset，然后重复进入处理链路。所以需要内容指纹、文件复用和并发互斥。

### Q2：MD5 在这里起什么作用？

答：

MD5 用作视频内容指纹。相同内容的视频可以复用同一个 `video_assets` 记录，用户任务记录在 `video_tasks`。这样真实文件资产和用户任务分开，一个 asset 可以被多个 task 引用，减少重复入库和重复处理。

### Q3：为什么还需要 Redis 锁？

答：

MD5 可以识别相同内容，但并发时仍可能发生竞态。比如两个请求同时发现 asset 不存在，然后都去合并、入库和投递任务。Redis 锁保护的是同一 MD5 的关键临界区，包括分片合并和分析任务消费，减少重复合并、重复入库和重复处理。

### Q4：WatchDog 是怎么实现的？

答：

当前项目不是 Redisson，而是 Go 自实现 Redis 锁。获取锁用 `SETNX + TTL`，锁 value 带 owner。业务可能超过固定 TTL，所以启动 watchdog 按 `ttl / 3` 周期续期。续期和释放都要校验 owner，避免锁过期后误操作别人新拿到的锁。

### Q5：释放锁为什么要校验 owner？

答：

如果 A 拿锁后执行太久，锁过期了，B 又拿到同一把锁。A 执行完如果直接 `DEL key`，就会删掉 B 的锁。owner 校验能保证只有当前持锁者才能释放。

### Q6：Redis 锁和 MySQL 唯一索引有什么区别？

答：

Redis 锁是前置并发控制，尽量减少重复计算；MySQL 唯一约束是最终兜底，防止重复数据落库。对 AI 视频处理来说，重复计算成本很高，所以不能只等数据库最终报错，而要尽早在 Redis 层拦住。

### Q7：MD5 会不会碰撞？

答：

理论上会，但视频去重场景下 MD5 作为工程指纹通常够用。如果要更严谨，可以组合文件大小、SHA-256 和对象元数据一起判断。我不会说 MD5 绝对安全。

### 30 秒话术

我用 MD5 做视频内容指纹，把真实文件资产放在 `video_assets`，用户任务放在 `video_tasks`，相同视频可以复用 asset。并发场景下，为了避免两个请求同时创建资产、重复合并或重复触发分析处理，我用 Redis 分布式锁保护同一 MD5 的分片合并和分析任务消费等临界区。锁里做了 TTL、owner 校验释放和 watchdog 续期，避免长任务锁过期或误删别人锁。

## 4. 分片上传与断点续传

### 简历原句

采取分片上传与断点续传机制，通过 Redis Set 记录分片状态，使用 MinIO 存储分片并通过 ComposeObject 合并，提升弱网环境下大文件上传可靠性

### Q1：背景痛点是什么？

答：

视频文件大，单次上传时间长，弱网环境下容易中断。如果失败后必须重传整个文件，用户体验很差，服务端连接也会被长时间占用。所以要把大文件拆成分片，每个分片独立上传，失败时只补缺失分片。

### Q2：完整流程是什么？

答：

前端按分片上传文件，每个请求带 fileMD5、分片编号和分片数据。后端先把分片写入 MinIO 临时路径，然后把分片编号写入 Redis Set。前端恢复上传时可以查询已上传分片，只补传缺失部分。全部分片到齐后，后端校验完整性，用 MinIO ComposeObject 合并成最终对象。

### Q3：为什么用 Redis Set？

答：

分片状态本质是“已上传编号集合”。Set 天然去重，重复上传同一分片只会保留一份编号；查询已上传分片也很直接。分片状态是临时高频数据，放 Redis 比放 MySQL 更合适。

### Q4：为什么用 MinIO？

答：

视频文件不适合直接放后端本地磁盘。本地磁盘会绑定到单台机器，不利于扩容和重启。MinIO 提供对象存储能力，适合存视频和分片，后续也方便切换到 S3 类服务。

### Q5：ComposeObject 有什么好处？

答：

ComposeObject 在对象存储侧合并分片，后端不需要把所有分片读回内存再拼接。这样能降低应用内存压力，也更符合大文件处理方式。

### Q6：上传到 99% 断了怎么办？

答：

重新上传时，前端根据文件 MD5 查询后端已上传分片集合，只上传缺失分片，不需要重传全部文件。后端 Redis Set 保存了已完成分片编号。

### Q7：合并时并发怎么办？

答：

合并是临界区。同一个 fileMD5 可能被重复点击合并，或者多个请求同时触发合并，所以合并阶段会用 Redis 锁保护，避免重复 ComposeObject 和重复创建 asset。

### 30 秒话术

大视频直接上传容易超时，弱网失败后重传成本高。所以我把文件拆成分片，每个分片先写 MinIO，再用 Redis Set 记录分片编号。恢复上传时只补传缺失分片，全部到齐后服务端校验完整性，并用 MinIO ComposeObject 合并。合并阶段用 Redis 锁防止并发重复合并。

## 5. Redis Lua 令牌桶限流

### 简历原句

基于 Redis Lua 令牌桶限流，按用户和接口维度限制高成本 AI 请求，缓解恶意请求和重复点击带来的外部 AI 服务调用成本

### Q1：背景痛点是什么？

答：

AI 应用的请求成本不一样。ASR、LLM、Embedding 都可能调用外部服务，恶意请求或重复点击会带来真实成本，也可能触发 provider 的 429。所以需要在入口限制高成本请求频率。

### Q2：为什么选择令牌桶？

答：

令牌桶既限制平均速率，又允许一定突发。固定窗口有边界突刺问题，漏桶对突发不友好。用户偶尔连续点几次可以接受，但长期高频请求要被限制，令牌桶适合这个场景。

### Q3：为什么用 Lua？

答：

限流要读 token、计算补充、判断是否足够、扣减、写回。如果拆成多条 Redis 命令，并发下可能超发。Lua 脚本在 Redis 内部原子执行，能保证同一个 key 的状态一致。

### Q4：限流 key 怎么设计？

答：

按 route + userID 做限流，未登录或无法识别用户时按 IP。这样可以限制某个用户对某类接口的请求频率，不同接口也可以设置不同策略。

### Q5：Redis 挂了怎么办？

答：

当前策略是 fail-open，也就是 Redis 异常时放行请求。原因是限流是保护层，不能因为 Redis 短暂异常就让核心业务全部不可用。生产上可以对极高成本接口改成 fail-close 或降级。

### Q6：限流和 Kafka 削峰有什么区别？

答：

限流发生在入口，决定请求能不能进系统；Kafka 削峰发生在任务进入系统之后，让后端按消费能力处理。限流解决“要不要进来”，Kafka 解决“进来后怎么慢慢处理”。

### 30 秒话术

因为 ASR、LLM、Embedding 等能力都会产生外部 AI 服务调用成本，所以我在入口做了 Redis 令牌桶限流。限流 key 按接口和用户维度生成，Redis Hash 保存 token 和刷新时间，Lua 脚本原子完成补充和扣减。这样可以限制高频请求，同时允许合理突发。当前 Redis 异常时选择 fail-open，优先保证主流程可用。

## 6. RAG 检索增强生成

### 简历原句

基于 RAG 检索增强生成实现视频智能问答，将转写文本切分为 chunk 并生成向量写入 Milvus，结合 BM25 关键词召回与 RRF 融合排序，返回引用片段提升答案可解释性

### Q1：为什么要做 RAG？

答：

大模型本身不知道用户上传的视频内容。如果只让模型根据问题回答，它只能靠预训练知识，容易胡说。RAG 的作用是先从视频转写文本中检索相关片段，再让模型基于这些片段回答。

### Q2：为什么不用 AI 摘要做知识库？

答：

摘要是压缩后的二次生成结果，会丢细节，也可能带模型自己的表达。RAG 的知识源应该尽量贴近原始资料，所以我用 ASR 转写全文切 chunk。摘要适合展示概览，不适合作为唯一问答依据。

### Q3：RAG 索引构建流程是什么？

答：

ASR 完成后，系统拿到完整转写文本，按 chunk size 和 overlap 切分。每个 chunk 调用 Embedding 生成向量，chunk 元数据写 MySQL，向量写 Milvus。索引构建状态记录到 RAG index 表里。

### Q4：问答流程是什么？

答：

用户提问后，系统先把问题向量化，然后在 Milvus 中按 userID、taskID 和 embedding model 过滤检索相关 chunk。同时基于 MySQL chunk 表做 BM25 关键词召回，再用 RRF 融合排序。最后把检索片段、最近对话和问题组装 prompt 给 LLM，并返回答案和 citations。

### Q5：为什么要加 BM25 关键词召回？

答：

向量召回擅长语义相似，但对专有名词、数字、英文缩写和短词不一定稳定。BM25 关键词召回能补足精确匹配能力，项目里是基于 MySQL chunk 表做的轻量实现。两路召回融合后，召回更稳。

### Q6：RRF 是什么？

答：

RRF 是 Reciprocal Rank Fusion，按排名倒数融合多路检索结果。它不要求向量分数和关键词分数在同一尺度上，只看排名。一个 chunk 如果在两路召回里都靠前，最终分数会更高。

### Q7：为什么要返回引用片段？

答：

citations 能告诉用户答案依据来自哪些视频片段，降低模型脱离视频内容编造的风险。它也方便后续排查：如果答案错了，可以看是召回错了，还是模型基于正确片段生成错了。

### Q8：Milvus 挂了怎么办？

答：

Milvus 挂了会影响 RAG 索引构建和问答，但不应该影响上传、转写和摘要。项目启动时对 Milvus 初始化有超时降级思路；运行中 Milvus 异常会进入 RAG 任务失败和重试链路。

### Q9：RAG 怎么评估？

答：

可以准备一组问题和期望命中的关键词或片段，统计 Recall@K、MRR、无结果率和 source mix。项目里已经有 RAG eval 的核心逻辑，可以支持这种评估。

### 30 秒话术

我没有把 AI 摘要当 RAG 知识库，而是用 ASR 转写全文。转写文本切 chunk 后做 Embedding，向量写 Milvus，元数据写 MySQL。用户提问时先做向量召回，再基于 MySQL chunk 表做 BM25 关键词召回，最后用 RRF 融合排序，把相关片段交给 LLM，并返回 citations，让答案有依据可追溯。

## 7. 任务失败治理

### 简历原句

设计任务失败治理机制，基于错误分类和阶梯退避重试处理外部依赖异常，对可恢复错误延迟重投递，对不可恢复错误快速失败，避免重试风暴和 AI 成本浪费

### Q1：为什么要单独写失败治理？

答：

因为视频处理依赖很多外部服务：远程视频站点、MinIO、ASR、LLM、Embedding、Milvus。失败是常态，不能简单“一失败就结束”，也不能所有错误都无限重试。需要区分哪些错误可能恢复，哪些错误重试也没用。

### Q2：哪些错误可重试？

答：

网络超时、429、5xx、MinIO 或 Milvus 的临时异常通常可重试。这类问题可能是瞬时抖动，延迟后重试有成功机会。

### Q3：哪些错误不可重试？

答：

AI 配置缺失、API Key 解密失败、未授权、文件不存在、Embedding 维度不匹配、ASR 空结果等通常不可重试。因为这些问题不是等一会就能恢复，继续重试只会浪费资源。

### Q4：为什么不让 Kafka 自己一直重试？

答：

Kafka offset 只能表示消费进度，不适合承载业务重试策略。如果不 commit，让 Kafka 立即重放，遇到 429 或三方故障时会形成热循环。我的做法是先把失败状态和下一次重试信息写入 MySQL，再由 DB retry scheduler 到期重新投递。

### Q5：失败后 commit offset 会不会丢任务？

答：

关键在顺序。先把业务失败状态写入 DB，再 commit offset。这样 Kafka 这条消息可以结束，但业务恢复入口保存在 DB 中。后续 scheduler 会重新投递，不是把任务丢掉。

### Q6：什么是重试风暴？

答：

当下游服务已经不稳定时，如果上游大量任务立刻重试，会让下游更难恢复，也会占满自己系统资源。阶梯退避的作用是拉开重试间隔，给下游恢复时间，也减少无效请求。

### Q7：为什么简历不写具体 60s / 300s / 900s？

答：

具体时间是实现参数，不是简历主卖点。简历上写“阶梯退避重试”更合适。面试官追问时再说当前默认退避是 60s、300s、900s，最大重试 3 次。

### Q8：失败治理和限流有什么关系？

答：

限流是在入口控制请求频率，失败治理是在任务执行失败后控制恢复节奏。一个防止请求过多进入系统，一个防止失败任务重试过快，它们共同保护外部 AI 服务和系统资源。

### 30 秒话术

外部 AI 和中间件失败不能一概重试。我把错误分成可恢复和不可恢复：网络超时、429、5xx、MinIO / Milvus 临时异常会写入 MySQL 重试状态，后续由 retry scheduler 延迟重投递；配置缺失、密钥错误、文件不存在、Embedding 维度不匹配这类错误直接失败。这样能避免重试风暴，也避免无效消耗 AI 成本。

## 10. 扩展追问题库：Kafka 异步化

### Q1：Kafka 生产端怎么尽量保证消息不丢？

答：

项目里 producer 是同步发送，并配置 `RequiredAcks=All` 和 `MaxAttempts=3`。这表示 producer 发送时会等待 ISR 副本确认，发送失败也会重试。它不能等同于绝对不丢，但比 fire-and-forget 更可靠。业务上还会先创建任务记录，再投递消息，后续可以通过任务状态发现异常。

### Q2：Kafka 消费端为什么要手动 commit？

答：

因为业务处理不是简单读一条消息就算成功。consumer 要执行下载、转写、摘要或 RAG 索引构建，还要更新 MySQL 状态。手动 commit 可以把 offset 推进和业务处理结果绑定起来，至少能明确在什么时机承认这条消息已经处理完或已经转入 DB 重试。

### Q3：如果消息发送成功，但接口返回给前端前服务挂了怎么办？

答：

这个场景要看任务是否已经落库。当前接口核心是先落任务，再投递消息。即使返回前服务挂了，只要任务和消息已经成功，后台仍会处理；前端可能没有拿到响应，需要刷新任务列表或重新查询。生产上可以加 traceID 或幂等请求 ID，让前端重试时能查到已有任务。

### Q4：如果任务落库成功，但 Kafka 投递失败怎么办？

答：

这类问题不能让用户一直以为处理中。合理做法是捕获投递异常，把任务标记为投递失败或可重试状态，再由调度器补偿投递。项目里的失败治理思路就是把恢复入口落到 DB，而不是只依赖内存状态。

### Q5：为什么 RAG 索引构建也单独拆成 Kafka job？

答：

ASR 成功后，用户已经能看到转写文本；RAG 索引构建依赖 Embedding 和 Milvus，失败边界不同。如果把 RAG 索引构建同步塞在转写 consumer 里，Milvus 或 Embedding 异常会影响转写任务完成。拆成独立 job 后，转写成功可以先落库，RAG 失败只影响问答索引。

### Q6：不同任务 topic 是否需要顺序消费？

答：

大多数场景不需要全局顺序。视频处理只需要单个任务内部阶段顺序，比如先有视频文件，再转写，再 RAG 索引构建。这个顺序通过业务状态和后续投递控制，不需要 Kafka 全局有序。追求全局有序会牺牲吞吐。

### Q7：Kafka 积压怎么排查？

答：

先看是哪一个 topic 积压：download、transcribe、analyze 还是 rag-index。然后看 consumer 日志和外部依赖耗时。如果是 ASR 慢，瓶颈可能在 provider；如果是 RAG 慢，可能是 Embedding 或 Milvus；如果是下载慢，可能是远程站点或网络。处理方式包括增加分区和消费者、拆分慢任务、限制单任务大小，以及对外部 API 做速率控制。

### Q8：为什么不用一个统一 topic，然后靠 job_type 区分？

答：

统一 topic 实现简单，但不同任务耗时差异很大。下载、ASR、摘要、RAG 索引构建依赖不同外部系统，如果放在一个 topic，慢任务可能影响快任务排队。拆 topic 后可以独立扩容、独立监控、独立重试策略也更清晰。

### Q9：Kafka 和任务状态之间会不会不一致？

答：

会有最终一致性问题，所以不能假设 Kafka 和 DB 永远同步。设计上要让 DB 成为业务状态来源，Kafka 是状态推进的触发器。失败时记录 DB 状态，重试也从 DB 找待恢复任务，这样即使中间某次消息失败，也有恢复入口。

### Q10：面试官说“你这个项目流量不大，用 Kafka 是不是过度设计”怎么答？

答：

如果只是低并发 demo，同步处理或本地队列确实更简单。但我这里引入 Kafka 的核心原因不是当前 QPS，而是单个任务耗时长、外部依赖多、失败恢复复杂。即使用户少，一个 15 分钟视频也可能让 HTTP 请求超时。Kafka 解决的是长任务生命周期和削峰问题，不只是高并发。

## 11. 扩展追问题库：Redis 分布式锁和去重

### Q1：你的锁 key 怎么设计？

答：

锁 key 要和临界资源绑定。比如分片合并可以按 fileMD5 生成类似 `vidlens:merge:{fileMD5}` 的 key。这样同一个文件的合并互斥，不同文件之间不会互相阻塞。

### Q2：锁的 value 为什么不能随便写？

答：

value 要能标识持锁者，通常是随机 token 或 owner。释放和续期时要比较 value，确认当前请求仍是锁持有者。否则锁过期后被别人拿到，旧请求可能误删新锁。

### Q3：锁 TTL 设置过长或过短分别有什么问题？

答：

TTL 过短，长任务还没完成锁就过期，可能导致并发重复处理；TTL 过长，进程异常退出后锁释放太慢，会影响后续请求。watchdog 的作用就是在任务仍存活时续期，进程挂掉后让 TTL 自然释放。

### Q4：watchdog 续期失败怎么办？

答：

续期失败意味着锁可能已经丢失或 Redis 不可用。严谨做法是后续关键写入仍要靠 DB 幂等和唯一约束兜底，不能把锁当唯一一致性保障。面试里要承认：分布式锁是减少重复执行，不是强一致的唯一来源。

### Q5：为什么不用本地互斥锁？

答：

本地锁只在单进程有效。后端如果多实例部署，相同视频请求可能落到不同实例，本地锁无法互斥。Redis 是所有实例共享的外部组件，适合做跨进程互斥。

### Q6：如果 Redis 挂了，去重是不是完全失效？

答：

Redis 挂了会影响前置互斥，但不应该让最终数据完全失控。兜底包括 MD5 asset 复用、数据库唯一约束、任务状态幂等和对象存储命名策略。生产上可以用 Redis 高可用，或者在关键路径上加强 DB 约束。

### Q7：同一个视频不同用户上传，是否共用同一个 asset？

答：

设计上真实文件资产和用户任务是分开的。相同 MD5 的视频可以复用同一个 `video_assets`，但每个用户仍然有自己的 `video_tasks`。这样既节省存储和处理成本，又能保留用户维度的任务记录。

### Q8：删除视频时怎么避免误删共享文件？

答：

删除用户任务时不能直接删除对象存储里的真实文件，要先检查 asset 是否还有其他 task 引用。只有没有引用时，才能考虑清理真实对象。RAG chunks、转写、摘要等任务关联数据也要按 task 清理。

### Q9：锁能不能保证绝对不重复调用 AI？

答：

不能说绝对。锁可以大幅降低并发重复处理，但网络、进程崩溃、Redis 故障都可能让边界复杂。真正的可靠性要靠锁、MD5、DB 幂等、任务状态和失败恢复组合。

### Q10：面试官说“你这个 WatchDog 不就是 Redisson 吗”怎么答？

答：

思路类似，但实现不是 Redisson。Redisson 是 Java 生态成熟库，我这个 Go 项目是自实现最小能力：`SETNX + TTL + owner + Lua 释放 + watchdog 续期`。简历里如果写 WatchDog，我会明确说是续期机制，不是用了 Redisson。

## 12. 扩展追问题库：分片上传与断点续传

### Q1：分片大小怎么定？

答：

分片大小是上传效率和失败重传成本的折中。分片太小，请求数太多；分片太大，失败重传成本高。当前简历不写固定大小，面试中可以说会根据文件大小、网络环境和服务端限制调整。

### Q2：前端怎么知道哪些分片已经传过？

答：

前端可以根据 fileMD5 调用检查接口，后端从 Redis Set 读取已上传分片编号返回。前端拿总分片列表和已上传集合做差集，只上传缺失分片。

### Q3：重复上传同一个分片会怎样？

答：

分片对象可以覆盖或幂等写入，Redis Set 记录编号天然去重。重复 `SADD` 同一个编号不会产生重复状态，所以对整体合并影响较小。

### Q4：Redis 里记录分片状态，为什么不怕 Redis 数据丢？

答：

Redis 状态是上传过程中的临时状态，丢失后可能需要重新校验 MinIO 已有分片，或者让用户重传部分分片。最终资产状态仍在 MySQL。生产上可以给 Redis key 设置 TTL，并提供根据对象存储反查分片的恢复能力。

### Q5：合并前怎么校验分片完整性？

答：

至少要校验已上传分片数量是否等于 totalChunks，以及编号是否连续完整。更严格可以校验每个分片大小或分片 hash。当前简历只写 Redis Set 记录和合并前校验，不夸大成完整强校验系统。

### Q6：MinIO ComposeObject 失败怎么办？

答：

合并失败不能创建最终 asset。应该保留分片状态，返回错误或记录失败原因，允许后续重新触发合并。因为分片还在 MinIO，用户不一定需要重新上传全部内容。

### Q7：临时分片怎么清理？

答：

合并成功后可以删除 `chunks/{md5}/...` 临时对象；长时间未完成的上传可以靠定时任务或对象存储生命周期清理。当前可以作为后续生产化优化，不要说已经有完整生命周期管理，除非代码里实现了。

### Q8：为什么不让前端直传 MinIO？

答：

前端直传可以减少后端带宽压力，但需要预签名 URL、权限控制和回调确认。当前项目后端接收分片再写 MinIO，业务控制更简单，适合当前阶段。后续如果文件量更大，可以升级成预签名直传。

### Q9：分片上传和 MD5 秒传是什么关系？

答：

MD5 可以先判断服务端是否已有相同 asset，如果已有可以直接复用，类似秒传；如果没有，则进入分片上传。分片上传解决传输可靠性，MD5 去重解决内容复用。

### Q10：面试官问“弱网怎么测”怎么答？

答：

可以用浏览器 devtools、代理工具或网络模拟工具制造丢包、限速和断网。验证重点是：中断后 Redis Set 是否保留已上传分片，恢复后是否只补传缺失分片，合并后文件是否可用。没有真实压测数据时，不要写具体成功率。

## 13. 扩展追问题库：Redis Lua 令牌桶限流

### Q1：令牌桶核心字段有哪些？

答：

核心是当前 token 数和上次刷新时间。每次请求根据当前时间和上次刷新时间计算应该补充多少 token，然后判断是否足够扣减。

### Q2：为什么不用简单计数器？

答：

简单计数器通常是固定窗口，容易出现窗口边界突刺。例如上一秒末尾打满额度，下一秒开头又打满额度，短时间内实际流量会翻倍。令牌桶按速率补充，更平滑。

### Q3：令牌桶和漏桶怎么区别？

答：

漏桶强调匀速流出，突发请求需要排队；令牌桶允许桶里积累 token，所以能处理合理突发。AI 请求场景下，用户偶尔连续点击可以允许，但持续高频要限制，令牌桶更合适。

### Q4：为什么限流维度要带 route？

答：

不同接口成本不同。查询任务状态成本低，ASR、摘要、RAG 问答成本高。如果只按用户总限流，会影响普通查询体验；按 route + user 可以对高成本接口单独限制。

### Q5：为什么未登录时按 IP 限流？

答：

未登录时没有 userID，只能退化到 IP 维度。IP 维度不完美，比如 NAT 下多个用户共享一个 IP，但它比完全不限制更好。

### Q6：Redis Lua 会不会阻塞 Redis？

答：

Lua 在 Redis 单线程中执行，所以脚本必须短小。令牌桶脚本只是简单计算和读写 hash，执行时间很短，适合用 Lua。不要在 Lua 里做复杂循环或慢操作。

### Q7：限流返回给前端什么？

答：

一般返回 429 或统一错误响应，提示请求过于频繁。前端可以禁用按钮、显示冷却提示，避免用户继续重复点击。

### Q8：fail-open 会不会被攻击打穿？

答：

会降低保护能力，这是可用性和保护性的权衡。当前项目选择 Redis 异常时优先保证主业务可用。生产上可以对高成本接口改为 fail-close，或增加本地降级限流。

### Q9：限流能不能替代鉴权？

答：

不能。限流控制频率，鉴权控制身份和权限。用户是否能访问某个视频、AI profile、RAG session，必须由鉴权和用户隔离保证。

### Q10：限流能不能完全控制 AI 成本？

答：

不能。限流只是入口保护。成本控制还需要 MD5 去重、任务失败最大重试次数、RAG 检索不到不调用 LLM、BYOK 和调用审计等组合手段。

## 14. 扩展追问题库：RAG

### Q1：chunk size 和 overlap 怎么影响效果？

答：

chunk 太大，检索结果噪声多，prompt 成本高；chunk 太小，语义可能不完整。overlap 可以缓解句子或段落被切断的问题，但 overlap 太大会增加 chunk 数和 embedding 成本。当前默认 800 和 120 是工程折中。

### Q2：为什么 embedding model 要和索引绑定？

答：

不同 embedding model 的向量空间不同，不能混用。检索时必须用和建库相同的 embedding model，否则向量相似度没有意义。所以 Milvus filter 和 MySQL chunk 都带 embedding_model。

### Q3：embedding 维度为什么重要？

答：

Milvus collection 的向量维度是固定的。如果用户配置的 embedding dim 和 collection 不一致，写入或搜索会失败。这类错误不是重试能解决的，所以应快速失败并提示配置问题。

### Q4：为什么检索要带 user_id？

答：

RAG chunk 本质是视频转写内容，属于用户数据。如果不按 user_id 过滤，可能把其他用户视频片段召回给当前用户，既是隐私问题，也是回答污染。

### Q5：为什么检索要带 task_id？

答：

当前问答是针对单个视频的。task_id 可以限定检索范围，避免召回同一用户其他视频的内容。这样答案更准确，也更符合用户预期。

### Q6：RAG 检索不到怎么办？

答：

不能让模型凭空编。应该提示当前视频没有检索到相关内容，或者建议用户换个问题。项目里如果没有 retrieved context，会拒绝继续生成，避免无依据回答。

### Q7：检索到了但答案还是错，怎么排查？

答：

先看 citations 是否包含正确片段。如果 citations 错，是召回问题，要调整 chunk、embedding、关键词召回或 RRF；如果 citations 对但答案错，是 prompt 或 LLM 生成问题，要加强 prompt 约束或考虑 rerank / 更强模型。

### Q8：为什么不用 Elasticsearch？

答：

当前是单视频 RAG，每个视频 chunk 数量有限，用基于 MySQL chunk 表实现的 BM25 关键词召回配合 Milvus 已能验证混合召回链路。Elasticsearch 更适合大规模全文检索，但会增加部署复杂度。后续跨视频知识库规模上来后可以考虑。

### Q9：为什么没有 rerank？

答：

rerank 能提升排序质量，但会增加一次模型调用，带来延迟和成本。当前先用向量召回 + 关键词召回 + RRF 做低成本融合。等评估发现 topK 里有正确片段但排序靠后时，再加 rerank 更合理。

### Q10：RAG 和摘要是什么关系？

答：

摘要适合快速浏览视频内容，RAG 适合回答具体问题。摘要是展示结果，不适合作为唯一知识源；RAG 用 ASR 全文保留细节。

### Q11：RAG 和多轮上下文怎么结合？

答：

问题检索仍以当前问题为主，同时可以带最近几轮对话帮助模型理解指代。项目里 MySQL 保存完整聊天记录，Redis 缓存最近消息，用于控制 prompt 大小。

### Q12：怎么证明 RAG 有效果？

答：

用评估集。准备问题和期望命中的关键词或片段，统计 Recall@K、MRR、无结果率和来源分布。项目里有 RAG eval 核心逻辑，可以围绕这些指标讲。

## 15. 扩展追问题库：任务失败治理

### Q1：错误分类靠什么做？

答：

当前是基于错误信息和错误类型做工程分类，比如 timeout、429、5xx、MinIO、Milvus 这类临时异常归为可重试；配置缺失、密钥错误、文件不存在、embedding 维度不匹配归为不可重试。生产上可以进一步标准化错误码。

### Q2：为什么不可重试错误要快速失败？

答：

因为重试不会改变结果。例如 API Key 解密失败、AI 配置缺失、文件不存在，这些需要用户或配置修复。继续重试只会占用队列和消耗资源。

### Q3：为什么可重试错误要延迟，而不是马上重试？

答：

马上重试可能遇到同样的下游故障，尤其是 429 和 5xx。延迟重试能给下游恢复时间，也避免大量任务同时重试形成重试风暴。

### Q4：最大重试次数怎么定？

答：

这是成功率和资源消耗之间的折中。次数太少可能错过短暂恢复，次数太多会拖长任务和浪费成本。当前默认最大 3 次，属于保守配置，简历正文不写死，面试追问时再讲。

### Q5：超过最大重试次数怎么办？

答：

任务进入终态失败或 dead 状态，记录最后错误原因。这样系统不会无限重试，用户也能看到失败原因，后续可以手动重新触发或人工排查。

### Q6：如果 LLM 已经生成结果，但写 MySQL 失败怎么办？

答：

这是比较麻烦的场景，因为外部调用成本已经发生。理想上要尽量先保存关键结果，或把结果和任务状态放到可恢复记录中。当前项目主要通过任务失败记录和重试机制保证恢复入口，生产上可以增加本地容灾日志或 outbox 表。

### Q7：如果 retry scheduler 自己挂了怎么办？

答：

挂了会影响待重试任务恢复速度，但不会丢任务，因为待重试状态在 MySQL。scheduler 恢复后可以继续扫描到期任务。生产上要给 scheduler 加健康检查和告警。

### Q8：为什么不用 Kafka 延迟队列？

答：

Kafka 原生不提供像 RocketMQ 那样直接的业务延迟消息。可以用延迟 topic 或时间轮模拟，但复杂度较高。当前项目用 MySQL 保存 `next_retry_at`，scheduler 到期重投递，业务状态更可见，也方便排查。

### Q9：任务失败治理和最终一致性怎么关联？

答：

视频任务涉及 Kafka、MySQL、MinIO、Milvus 和外部 AI，不做全局强事务。系统通过状态落库、幂等处理和重试补偿让任务最终到达成功、失败或 dead 的明确状态，这就是最终一致性思路。

### Q10：面试官问“任务怎么不丢”怎么答？

答：

不要说绝对不丢。应该说：Kafka 生产端有 ack 和重试，业务状态落 MySQL；consumer 失败先写 DB 再 commit offset；待重试任务由 scheduler 扫描恢复。这样把丢失风险分层降低，并且有可观测的恢复入口。

## 16. 横向交叉追问

### Q1：Kafka、Redis 限流、失败重试三者都和“保护系统”有关，它们区别是什么？

答：

限流是入口保护，决定请求能不能进来；Kafka 是异步削峰，决定进来的任务怎么排队处理；失败重试是恢复治理，决定已经失败的任务何时再试。三者作用阶段不同。

### Q2：Redis 在项目里用了哪些数据结构？

答：

主要有三类：Set 用于分片上传状态，Hash 用于令牌桶 token 和 last time，String 用于分布式锁 key-value。还可以提 Redis 最近会话记忆，但它不是最终简历主线。

### Q3：MySQL 在项目里承担什么角色？

答：

MySQL 是业务事实来源。任务状态、job 状态、转写、摘要、RAG chunk 元数据、聊天记录和 AI 配置都落 MySQL。Kafka 和 Redis 都偏过程状态或中间件，最终给用户看的业务状态要能从 MySQL 查到。

### Q4：MinIO 和 Milvus 都依赖对象存储，它们在项目角色有什么区别？

答：

MinIO 在业务层存用户视频和上传分片；Milvus 是向量数据库，存 RAG 检索向量，它内部也可能依赖对象存储保存索引数据。业务上不要混淆：视频文件在 MinIO，语义检索向量在 Milvus。

### Q5：如果让你现在优化一版，优先做什么？

答：

我会先做三件：任务进度推送，减少前端轮询；RAG 增加 rerank 并用评估集验证；失败治理补更标准的错误码和告警。它们分别提升体验、问答质量和可运维性。

### Q6：这个项目有什么生产化不足？

答：

当前还不是生产级系统。比如没有完整 Prometheus 告警、没有 KMS 密钥管理、RAG 没有 rerank、URL 下载还需要更强网络隔离、分片临时对象清理可以更完善。面试里我会明确这些边界，不把 demo 说成生产系统。

## 17. 高压连环拷打模板

### 模板一：面试官质疑项目真实性

面试官：你说 Kafka 异步，那从入口到 consumer 代码怎么走？

你：入口在 media handler，service 创建 task，然后 producer 投递对应 topic。启动入口 `cmd/server/main.go` 初始化 producer、consumer 和 retry scheduler。consumer 里按 job type 处理下载、转写、分析和 RAG 索引构建。

面试官：你怎么证明不是写死状态？

你：任务状态和 job 状态在 MySQL 模型里，consumer 成功或失败都会更新。失败时还会写 retry_count、next_retry_at 和 last_error。

面试官：代码里最能证明你做过的是哪几个文件？

你：`internal/mq/consumer.go`、`internal/mq/retry.go`、`internal/service/media.go`、`internal/service/rag_index.go`、`internal/service/chat.go`。

### 模板二：面试官质疑技术选型过度

面试官：你用户量不大，Kafka、Milvus 是不是堆技术？

你：如果只看 QPS，确实可以更简单。但这个项目的核心不是高并发，而是单个任务长、外部依赖多、失败恢复复杂。Kafka 解决长任务异步和状态推进，Milvus 解决视频转写的语义检索。它们分别对应实际问题，不是为了堆栈。

面试官：那为什么不用更简单的 MySQL LIKE 做问答？

你：关键词能解决精确匹配，但语义问法会弱。当前我也没有完全放弃关键词，而是 Milvus 向量召回加 BM25 关键词召回，再用 RRF 融合。

### 模板三：面试官质疑 AI 成本控制

面试官：用户疯狂上传和问答，成本怎么控？

你：入口用 Redis 令牌桶限流；相同视频用 MD5 和 Redis 锁去重；转写结果复用给摘要和 RAG；失败重试有分类和最大次数；RAG 检索不到上下文时不调用或不让模型编造。公开部署场景还可以结合 BYOK 和调用审计。

面试官：限流挂了怎么办？

你：当前 Redis 异常时 fail-open，优先保证主流程可用。生产上对高成本接口可以改 fail-close 或加本地降级限流。
